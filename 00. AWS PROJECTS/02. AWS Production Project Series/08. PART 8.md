## Part 8: Security Groups — Production-Grade Firewall Design

Security Groups are **stateful** virtual firewalls that control traffic at the instance level. They're the first and last line of defense for your resources. Get them wrong, and you've got open ports to the internet. Get them right, and attackers bounce off invisible walls.

**Key characteristics:**
- **Stateful:** If you allow inbound traffic, the return traffic is automatically allowed (no need for outbound rules for responses)
- **Allow-only:** There are no "deny" rules — only explicit allows. Everything else is implicitly denied
- **Attached to ENIs:** Security Groups attach to Elastic Network Interfaces (one per instance)
- **Region-specific:** You reference them by ID, not name, across AZs

---

### Step 1: The Security Group Architecture

We're building three distinct security groups, each with a single responsibility:

| Security Group | Attached To | Purpose |
|----------------|-------------|---------|
| **ALB-SG** | Application Load Balancer | Accepts HTTP/HTTPS from the internet |
| **Bastion-SG** | Bastion Host | Accepts SSH from trusted IPs only |
| **App-SG** | EC2 instances in Auto Scaling | Accepts HTTP from ALB, SSH from Bastion only |

**The Traffic Flow:**

```
Internet → ALB-SG (80/443) → App-SG (80) → Application
                ↑
SSH Admin → Bastion-SG (22) → App-SG (22 from Bastion only)
```

**Critical principle:** App servers NEVER accept traffic directly from the internet. Everything goes through the ALB.

---

### Step 2: Create the ALB Security Group

This security group faces the public internet. It should only accept HTTP and HTTPS traffic.

```bash
aws ec2 create-security-group \
    --group-name veriqta-alb-sg \
    --description "Security group for Application Load Balancer - accepts HTTP/HTTPS from internet" \
    --vpc-id $VPC_ID \
    --tag-specifications 'ResourceType=security-group,Tags=[{Key=Name,Value=veriqta-alb-sg},{Key=Tier,Value=LoadBalancer},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

**Capture the Security Group ID:**
```json
{
    "GroupId": "sg-0alb1a2b3c4d5e6f7"
}
```

```bash
ALB_SG_ID="sg-0alb1a2b3c4d5e6f7"
```

**Add inbound rules:**

```bash
# HTTP from anywhere (we'll add HTTPS later with ACM)
aws ec2 authorize-security-group-ingress \
    --group-id $ALB_SG_ID \
    --protocol tcp \
    --port 80 \
    --cidr 0.0.0.0/0 \
    --tag-specifications 'ResourceType=security-group-rule,Tags=[{Key=Name,Value=veriqta-alb-http},{Key=Project,Value=Veriqta-HA-WebApp}]'

# HTTPS from anywhere (for future SSL termination)
aws ec2 authorize-security-group-ingress \
    --group-id $ALB_SG_ID \
    --protocol tcp \
    --port 443 \
    --cidr 0.0.0.0/0 \
    --tag-specifications 'ResourceType=security-group-rule,Tags=[{Key=Name,Value=veriqta-alb-https},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

**Verify the ALB security group:**
```bash
aws ec2 describe-security-groups --group-ids $ALB_SG_ID \
    --query 'SecurityGroups[*].[GroupName,IpPermissions[*].[IpProtocol,FromPort,ToPort,IpRanges[*].CidrBlock]]'
```

**Expected output:**
```json
[
    "veriqta-alb-sg",
    [
        ["tcp", 80, 80, ["0.0.0.0/0"]],
        ["tcp", 443, 443, ["0.0.0.0/0"]]
    ]
]
```

**Outbound rules:** By default, Security Groups allow ALL outbound traffic. For the ALB, this is fine — it needs to forward traffic to backend instances.

---

### Step 3: Create the Bastion Security Group

The bastion host is your secure jump box. It should ONLY accept SSH from your specific IP address — never from `0.0.0.0/0`.

**Get your public IP:**
```bash
MY_IP=$(curl -s https://checkip.amazonaws.com)
echo "Your public IP: $MY_IP"
```

```bash
aws ec2 create-security-group \
    --group-name veriqta-bastion-sg \
    --description "Security group for Bastion Host - SSH from admin IP only" \
    --vpc-id $VPC_ID \
    --tag-specifications 'ResourceType=security-group,Tags=[{Key=Name,Value=veriqta-bastion-sg},{Key=Tier,Value=Bastion},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

```bash
BASTION_SG_ID="sg-0bas1a2b3c4d5e6f7"
```

**Add SSH rule from YOUR IP only:**
```bash
aws ec2 authorize-security-group-ingress \
    --group-id $BASTION_SG_ID \
    --protocol tcp \
    --port 22 \
    --cidr "${MY_IP}/32" \
    --tag-specifications 'ResourceType=security-group-rule,Tags=[{Key=Name,Value=veriqta-bastion-ssh},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

> **CRITICAL:** Using `/32` means ONLY your exact IP. If your ISP changes your IP (common with residential internet), you'll be locked out. For learning, this is fine. In production, use a VPN or bastion with MFA.

**Verify:**
```bash
aws ec2 describe-security-groups --group-ids $BASTION_SG_ID \
    --query 'SecurityGroups[*].[GroupName,IpPermissions[*].[IpProtocol,FromPort,ToPort,IpRanges[*].CidrBlock]]'
```

### Step 4: Create the Application Security Group

This is the most important one. Your application servers live in private subnets and should ONLY accept traffic from the ALB and Bastion.

```bash
aws ec2 create-security-group \
    --group-name veriqta-app-sg \
    --description "Security group for application servers - HTTP from ALB, SSH from Bastion only" \
    --vpc-id $VPC_ID \
    --tag-specifications 'ResourceType=security-group,Tags=[{Key=Name,Value=veriqta-app-sg},{Key=Tier,Value=Application},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

```bash
APP_SG_ID="sg-0app1a2b3c4d5e6f7"
```

**Add HTTP rule from ALB Security Group (not CIDR!):**

This is a **Security Group reference** — much more secure than IP ranges. It means: "Any instance with the ALB security group can talk to me on port 80."

```bash
aws ec2 authorize-security-group-ingress \
    --group-id $APP_SG_ID \
    --protocol tcp \
    --port 80 \
    --source-group $ALB_SG_ID \
    --tag-specifications 'ResourceType=security-group-rule,Tags=[{Key=Name,Value=veriqta-app-http-from-alb},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

**Add SSH rule from Bastion Security Group only:**

```bash
aws ec2 authorize-security-group-ingress \
    --group-id $APP_SG_ID \
    --protocol tcp \
    --port 22 \
    --source-group $BASTION_SG_ID \
    --tag-specifications 'ResourceType=security-group-rule,Tags=[{Key=Name,Value=veriqta-app-ssh-from-bastion},{Key:Project,Value:Veriqta-HA-WebApp}]'
```

**Verify the Application security group:**
```bash
aws ec2 describe-security-groups --group-ids $APP_SG_ID \
    --query 'SecurityGroups[*].[GroupName,IpPermissions[*].[IpProtocol,FromPort,ToPort,UserIdGroupPairs[*].GroupId,IpRanges[*].CidrBlock]]'
```

**Expected output:**
```json
[
    "veriqta-app-sg",
    [
        ["tcp", 80, 80, ["sg-0alb1a2b3c4d5e6f7"], []],
        ["tcp", 22, 22, ["sg-0bas1a2b3c4d5e6f7"], []]
    ]
]
```

Notice:
- **No CIDR blocks** for application rules — only Security Group references
- **No `0.0.0.0/0`** anywhere on the App SG
- **Port 80 only from ALB**, Port 22 only from Bastion

### Step 5: Restrict Outbound Traffic on Application Security Group

By default, Security Groups allow ALL outbound traffic. For production hardening, let's restrict this.

**First, remove the default "allow all" outbound rule:**

```bash
aws ec2 revoke-security-group-egress \
    --group-id $APP_SG_ID \
    --protocol all \
    --port all \
    --cidr 0.0.0.0/0
```

**Add specific outbound rules:**

```bash
# HTTP/HTTPS to anywhere (for software updates, API calls)
aws ec2 authorize-security-group-egress \
    --group-id $APP_SG_ID \
    --protocol tcp \
    --port 80 \
    --cidr 0.0.0.0/0

aws ec2 authorize-security-group-egress \
    --group-id $APP_SG_ID \
    --protocol tcp \
    --port 443 \
    --cidr 0.0.0.0/0

# DNS (UDP and TCP)
aws ec2 authorize-security-group-egress \
    --group-id $APP_SG_ID \
    --protocol udp \
    --port 53 \
    --cidr 0.0.0.0/0

aws ec2 authorize-security-group-egress \
    --group-id $APP_SG_ID \
    --protocol tcp \
    --port 53 \
    --cidr 0.0.0.0/0

# NTP (time sync)
aws ec2 authorize-security-group-egress \
    --group-id $APP_SG_ID \
    --protocol udp \
    --port 123 \
    --cidr 0.0.0.0/0
```

**Verify outbound rules:**
```bash
aws ec2 describe-security-groups --group-ids $APP_SG_ID \
    --query 'SecurityGroups[*].[GroupName,IpPermissionsEgress[*].[IpProtocol,FromPort,ToPort,IpRanges[*].CidrBlock]]'
```

### Step 7: Security Group Best Practices (Interview Gold)

**1. Use Security Group References, Not CIDRs**
- ❌ `10.0.0.0/16` (brittle, doesn't adapt to changes)
- ✅ `sg-xxxxxxxx` (dynamic, adapts to instance changes)

**2. Principle of Least Privilege**
- Open only the ports you need
- Use specific source ranges, not `0.0.0.0/0` when possible
- Separate security groups by tier/function

**3. No Direct Internet Access for App Servers**
- App servers should NEVER have public IPs
- All inbound traffic flows through the ALB
- SSH only through Bastion

**4. Tag Everything**
- Name, Tier, Project, Environment
- Makes cost allocation and troubleshooting easier

**5. Regular Audits**
```bash
# Find security groups with 0.0.0.0/0 on SSH (security risk)
aws ec2 describe-security-groups \
    --filters "Name=ip-permission.from-port,Values=22" \
    --query 'SecurityGroups[*].[GroupId,GroupName,IpPermissions[*].IpRanges[*].CidrBlock]'

# Find security groups with overly permissive outbound
aws ec2 describe-security-groups \
    --query 'SecurityGroups[?length(IpPermissionsEgress[?IpProtocol==`all` && IpRanges[?CidrBlock==`0.0.0.0/0`]]) > `0`].[GroupId,GroupName]'
```
