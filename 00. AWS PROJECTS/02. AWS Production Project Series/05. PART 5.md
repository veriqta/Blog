## Part 5: VPC Design & CIDR Planning

This is the foundation of everything. Every resource you build lives inside a VPC. Get this wrong, and you'll have routing issues, IP conflicts, and security gaps. Get it right, and you have a scalable, secure network that mirrors real production environments.

---

### Step 1: Understanding CIDR Notation

CIDR (Classless Inter-Domain Routing) defines your IP address range. Before typing any command, you need to understand what these numbers mean.

**The VPC CIDR: `10.0.0.0/16`**

| CIDR | Total IPs | Usable IPs | Description |
|------|-----------|------------|-------------|
| `/16` | 65,536 | 65,531 | Large VPC, plenty of room to grow |
| `/20` | 4,096 | 4,091 | Medium VPC |
| `/24` | 256 | 251 | Small subnet (not enough for production) |

**How it works:**
- `/16` means the first 16 bits are fixed: `10.0.x.x`
- The remaining 16 bits are yours: `10.0.0.0` to `10.0.255.255`
- AWS reserves 5 IPs per subnet (network, broadcast, router, DNS, future use)

**Why `10.0.0.0/16`?**
- Private IP range (RFC 1918) â€” won't conflict with the public internet
- Large enough for multi-tier, multi-AZ architecture
- Easy to subnet logically

### Step 2: Subnet Design Plan

We're building across **2 Availability Zones (AZs)** for high availability. Here's our IP allocation:

| AZ | Subnet Type | CIDR | IP Range | Purpose |
|----|------------|------|----------|---------|
| **us-east-1a** | Public | `10.0.1.0/24` | 10.0.1.0 - 10.0.1.255 | ALB, NAT GW, Bastion |
| **us-east-1a** | Private | `10.0.2.0/24` | 10.0.2.0 - 10.0.2.255 | App servers (EC2) |
| **us-east-1b** | Public | `10.0.3.0/24` | 10.0.3.0 - 10.0.3.255 | ALB, NAT GW |
| **us-east-1b** | Private | `10.0.4.0/24` | 10.0.4.0 - 10.0.4.255 | App servers (EC2) |

**Key design decisions:**

1. **Public subnets get odd numbers (1, 3):** Easy mental model â€” "odd = public, even = private"
2. **Each AZ has one public + one private:** Balanced, symmetric, predictable
3. **/24 subnets (256 IPs each):** 251 usable. More than enough for ALB + NAT GW + a few bastions in public, and plenty of room for auto-scaling in private
4. **Room to grow:** We have 250+ unused /24 subnets in our /16 VPC for future expansion

### Step 3: Create the VPC

```bash
# Create the VPC with DNS support (critical for internal resolution)
aws ec2 create-vpc \
    --cidr-block 10.0.0.0/16 \
    --tag-specifications 'ResourceType=vpc,Tags=[{Key=Name,Value=veriqta-ha-webapp-vpc},{Key=Project,Value=Veriqta-HA-WebApp},{Key=Environment,Value=Production}]' \
    --enable-dns-support \
    --enable-dns-hostnames
```

**What each flag does:**
- `--cidr-block 10.0.0.0/16`: Our IP range
- `--enable-dns-support`: Allows instances to resolve hostnames to IPs within the VPC
- `--enable-dns-hostnames`: Assigns public DNS hostnames to instances with public IPs
- `--tag-specifications`: Tags for cost tracking and resource management

**Capture the VPC ID from the output:**

```json
{
    "Vpc": {
        "VpcId": "vpc-0a1b2c3d4e5f6789a",
        "CidrBlock": "10.0.0.0/16",
        ...
    }
}
```

Save that `VpcId` â€” you'll use it in every subsequent command.

**Set it as a variable for convenience:**

```bash
VPC_ID="vpc-0a1b2c3d4e5f6789a"  # Replace with your actual VPC ID
```

**Verify:**
```bash
aws ec2 describe-vpcs --vpc-ids $VPC_ID
```

### Step 4: Create the Internet Gateway (IGW)

The IGW is the doorway from your VPC to the public internet. Only public subnets will route to it.

```bash
# Create the IGW
aws ec2 create-internet-gateway \
    --tag-specifications 'ResourceType=internet-gateway,Tags=[{Key=Name,Value=veriqta-ha-webapp-igw},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

**Capture the IGW ID:**
```json
{
    "InternetGateway": {
        "InternetGatewayId": "igw-0a1b2c3d4e5f6789a",
        ...
    }
}
```

```bash
IGW_ID="igw-0a1b2c3d4e5f6789a"  # Replace with your actual IGW ID
```

**Attach the IGW to your VPC:**
```bash
aws ec2 attach-internet-gateway \
    --internet-gateway-id $IGW_ID \
    --vpc-id $VPC_ID
```

**Verify attachment:**
```bash
aws ec2 describe-internet-gateways --internet-gateway-ids $IGW_ID
```

### Step 5: Create Subnets

We need 4 subnets: 2 public, 2 private, across 2 AZs.

**First, get your AZ names:**
```bash
aws ec2 describe-availability-zones --region us-east-1
```

Look for AZs ending in `a` and `b` (e.g., `us-east-1a`, `us-east-1b`). These are physically separate data centers.

**Create Public Subnet in AZ-a:**
```bash
aws ec2 create-subnet \
    --vpc-id $VPC_ID \
    --cidr-block 10.0.1.0/24 \
    --availability-zone us-east-1a \
    --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=veriqta-public-1a},{Key=Type,Value=Public},{Key=AZ,Value=us-east-1a},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

**Capture the Subnet ID:**
```json
{
    "Subnet": {
        "SubnetId": "subnet-0pub1a2b3c4d5e6f7",
        ...
    }
}
```

```bash
PUBLIC_SUBNET_1A="subnet-0pub1a2b3c4d5e6f7"
```

**Create Private Subnet in AZ-a:**
```bash
aws ec2 create-subnet \
    --vpc-id $VPC_ID \
    --cidr-block 10.0.2.0/24 \
    --availability-zone us-east-1a \
    --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=veriqta-private-1a},{Key=Type,Value=Private},{Key=AZ,Value=us-east-1a},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

```bash
PRIVATE_SUBNET_1A="subnet-0pri1a2b3c4d5e6f7"
```

**Create Public Subnet in AZ-b:**
```bash
aws ec2 create-subnet \
    --vpc-id $VPC_ID \
    --cidr-block 10.0.3.0/24 \
    --availability-zone us-east-1b \
    --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=veriqta-public-1b},{Key=Type,Value=Public},{Key=AZ,Value=us-east-1b},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

```bash
PUBLIC_SUBNET_1B="subnet-0pub1b2b3c4d5e6f7"
```

**Create Private Subnet in AZ-b:**
```bash
aws ec2 create-subnet \
    --vpc-id $VPC_ID \
    --cidr-block 10.0.4.0/24 \
    --availability-zone us-east-1b \
    --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=veriqta-private-1b},{Key=Type,Value=Private},{Key=AZ,Value=us-east-1b},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

```bash
PRIVATE_SUBNET_1B="subnet-0pri1b2b3c4d5e6f7"
```

**Verify all subnets:**
```bash
aws ec2 describe-subnets --filters "Name=vpc-id,Values=$VPC_ID"
```

You should see 4 subnets with the correct CIDRs and AZs.

### Step 6: Enable Auto-Assign Public IP on Public Subnets

Public subnets need instances to automatically get public IPs. Private subnets don't.

```bash
# Public subnet 1a
aws ec2 modify-subnet-attribute \
    --subnet-id $PUBLIC_SUBNET_1A \
    --map-public-ip-on-launch

# Public subnet 1b
aws ec2 modify-subnet-attribute \
    --subnet-id $PUBLIC_SUBNET_1B \
    --map-public-ip-on-launch
```

**Verify:**
```bash
aws ec2 describe-subnets --subnet-ids $PUBLIC_SUBNET_1A $PUBLIC_SUBNET_1B \
    --query 'Subnets[*].[SubnetId,MapPublicIpOnLaunch,CidrBlock,AvailabilityZone]'
```

Both should show `"MapPublicIpOnLaunch": true`.
