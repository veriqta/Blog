## Part 3: Least-Privilege IAM Policy for This Project

AdministratorAccess is fine for learning, but in production, you'd never hand that out. A compromised account with full admin access can delete everything, rack up massive bills, or expose sensitive data. Let's build a custom policy that gives us **only** the permissions we need for this specific project.

This is a **critical interview topic**. Expect questions like: *"How do you implement least-privilege in AWS?"* or *"What's the difference between IAM policies, roles, and resource-based policies?"*

---

### Step 1: Understand the Permission Model

AWS uses JSON-based policies with this structure:

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",           // or "Deny"
            "Action": [                   // what you can do
                "ec2:DescribeInstances"
            ],
            "Resource": "*",              // what you can do it to
            "Condition": {}               // optional restrictions
        }
    ]
}
```

**Key Principles:**
- **Explicit Deny** always wins over Allow
- **Least Privilege:** Only grant what's needed
- **Resource-level permissions:** Restrict to specific ARNs when possible
- **Conditions:** Add time-based, IP-based, or MFA restrictions


**Good.** That's exactly what we want. Least privilege in action.

### Step 5: Understanding IAM Roles vs. IAM Users

| | IAM User | IAM Role |
|---|---|---|
| **What** | Permanent identity for a person | Temporary identity for a service or person |
| **Credentials** | Access keys (long-lived) | No credentials — assumed dynamically |
| **Use case** | Your CLI, console login | EC2 instances, Lambda, cross-account access |
| **Security** | Risky if keys leak | No keys to leak — uses STS tokens |
| **Rotation** | Manual | Automatic (1 hour default) |

**We'll create IAM Roles in Part 4** for EC2 instances to access CloudWatch and S3 without embedding access keys in our servers. This is a **must-know interview topic**.
