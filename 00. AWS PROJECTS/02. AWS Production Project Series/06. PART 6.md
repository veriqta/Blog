## Part 6: Route Tables — The Traffic Directors of Your VPC

Route tables are the most misunderstood part of AWS networking. They're not firewalls — they're **maps**. Every packet leaving a subnet checks the route table and asks: *"Where do I go?"*

Get this wrong, and your private instances leak to the internet. Get it right, and you have surgical control over every packet's path.

---

### Step 1: Understand How Route Tables Work

**The Routing Decision Process:**

```
1. Instance sends packet
        ↓
2. Subnet checks its associated Route Table
        ↓
3. Route Table looks at Destination IP
        ↓
4. Finds the most specific matching route
        ↓
5. Sends packet to the Target (IGW, NAT GW, local, etc.)
```

**Key Rules:**
- Every subnet MUST have exactly one route table association
- The **main route table** is the default for all subnets
- Routes are evaluated by **longest prefix match** (most specific first)
- `0.0.0.0/0` = "everything else" (default route, least specific)

---

### Step 2: The Route Table Design

| Route Table | Associated Subnets | Purpose |
|-------------|-------------------|---------|
| **Public RT** | Public subnets (1a, 1b) | Route `0.0.0.0/0` → IGW (internet access) |
| **Private RT** | Private subnets (1a, 1b) | Route `0.0.0.0/0` → NAT GW (outbound only) |

**Why separate route tables?**
- Public subnets need direct internet access (ALB, Bastion)
- Private subnets need **outbound-only** internet (for updates, patches)
- You never want private app servers directly accessible from the internet

### Step 3: Create the Public Route Table

```bash
aws ec2 create-route-table \
    --vpc-id $VPC_ID \
    --tag-specifications 'ResourceType=route-table,Tags=[{Key=Name,Value=veriqta-public-rt},{Key=Type,Value=Public},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

**Capture the Route Table ID:**
```json
{
    "RouteTable": {
        "RouteTableId": "rtb-0pub1a2b3c4d5e6f7",
        ...
    }
}
```

```bash
PUBLIC_RT_ID="rtb-0pub1a2b3c4d5e6f7"
```

**Add the route to the Internet Gateway:**
```bash
aws ec2 create-route \
    --route-table-id $PUBLIC_RT_ID \
    --destination-cidr-block 0.0.0.0/0 \
    --gateway-id $IGW_ID
```

**What this does:**
- Any traffic destined for `0.0.0.0/0` (any IP not in the VPC) goes to the IGW
- This makes the subnet "public" — instances can reach the internet directly

**Verify the route:**
```bash
aws ec2 describe-route-tables --route-table-ids $PUBLIC_RT_ID
```

Expected routes:
```json
{
    "Routes": [
        {
            "DestinationCidrBlock": "10.0.0.0/16",
            "GatewayId": "local",
            "State": "active"
        },
        {
            "DestinationCidrBlock": "0.0.0.0/0",
            "GatewayId": "igw-0a1b2c3d4e5f6789a",
            "State": "active"
        }
    ]
}
```

**The `local` route is automatic.** It handles all traffic within the VPC (10.0.0.0/16). You never create or delete this — AWS manages it.

### Step 4: Associate Public Route Table with Public Subnets

```bash
# Associate with Public Subnet 1a
aws ec2 associate-route-table \
    --route-table-id $PUBLIC_RT_ID \
    --subnet-id $PUBLIC_SUBNET_1A

# Capture the Association ID (for future disassociation if needed)
# Output: {"AssociationId": "rtbassoc-0a1b2c3d4e5f6789a"}

# Associate with Public Subnet 1b
aws ec2 associate-route-table \
    --route-table-id $PUBLIC_RT_ID \
    --subnet-id $PUBLIC_SUBNET_1B
```

**Verify associations:**
```bash
aws ec2 describe-route-tables --route-table-ids $PUBLIC_RT_ID \
    --query 'RouteTables[*].Associations[*].[SubnetId,RouteTableAssociationId]'
```

Both public subnets should be listed.

aws ec2 create-route-table \

    --tag-specifications 'ResourceType=route-table,Tags=[{Key=Name,Value=veriqta-private-rt},{Key=Type,Value=Private},{Key=Project,Value=Veriqta-HA-WebApp}]'

**Capture the Route Table ID:**
```json
{
    "RouteTable": {

```

### Step 7: Verify the Main Route Table (Understanding the Default)

Every VPC has a **main route table** created automatically. Check yours:

```bash
aws ec2 describe-route-tables \
    --filters "Name=vpc-id,Values=$VPC_ID" \
    --query 'RouteTables[?Associations[?Main==`true`]].RouteTableId'
```

**Why does this matter?**
- Any subnet NOT explicitly associated with a route table uses the main route table
- In production, you should **remove the main route table's IGW route** so unassociated subnets don't accidentally become public
- Or, leave the main route table with only the `local` route as a "safe default"

**Check your main route table:**
```bash
MAIN_RT_ID=$(aws ec2 describe-route-tables \
    --filters "Name=vpc-id,Values=$VPC_ID" \
    --query 'RouteTables[?Associations[?Main==`true`]].RouteTableId' \
    --output text)

aws ec2 describe-route-tables --route-table-ids $MAIN_RT_ID
```

If it has a `0.0.0.0/0` route to an IGW, **remove it**:
```bash
aws ec2 delete-route \
    --route-table-id $MAIN_RT_ID \
    --destination-cidr-block 0.0.0.0/0
```

This is a **security best practice**. The main route table should be a "safe" table with only local routing. Explicitly associate subnets with public or private route tables.

### Step 8: Visualize the Current Routing State

| Subnet | Route Table | 10.0.0.0/16 | 0.0.0.0/0 | Internet Access? |
|--------|-------------|-------------|-----------|------------------|
| Public 1a | Public RT | local | → IGW | Yes (inbound + outbound) |
| Public 1b | Public RT | local | → IGW | Yes (inbound + outbound) |
| Private 1a | Private RT | local | No route | No (isolated) |
| Private 1b | Private RT | local | No route | No (isolated) |

**Right now, your private subnets are completely isolated.** They can't reach the internet at all — not even for software updates. That's intentional. In Part 7, we'll add NAT Gateways to give them controlled outbound access.

### Step 10: Common Route Table Mistakes (Learn from Others' Pain)

| Mistake | Symptom | Fix |
|---------|---------|-----|
| Private subnet routes `0.0.0.0/0` to IGW | Private instances get public IPs, security breach | Change target to NAT Gateway |
| Subnet not associated with any RT | Uses main RT, unpredictable behavior | Explicitly associate with correct RT |
| Wrong AZ in subnet | Resources fail to launch | Double-check AZ names with `describe-availability-zones` |
| CIDR overlap with on-prem | VPN/Peering doesn't work | Plan CIDRs before creating VPC |
| Route table has no `local` route | Impossible — AWS manages this | If missing, your VPC is corrupted |
