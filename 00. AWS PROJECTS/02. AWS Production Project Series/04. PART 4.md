## Part 4: SSH Key Pair & IAM Role for EC2 Instances

Two critical security foundations before we touch any infrastructure:

1. **SSH Key Pair** — How you securely access EC2 instances (no passwords, ever)
2. **IAM Role for EC2** — How instances get permissions without hardcoded credentials

Both are non-negotiable in production. Let's build them properly.

---

### Step 1: Create an SSH Key Pair

SSH key pairs use asymmetric cryptography. You keep the private key. AWS keeps the public key. When you SSH, the server verifies your private key matches the public key it has stored.

**Option A: Create via AWS CLI (Recommended — we'll use this key throughout the project)**

```bash
# Create the key pair — AWS generates the key and gives you the private key ONCE
aws ec2 create-key-pair \
    --key-name veriqta-ha-webapp-key \
    --key-type rsa \
    --key-format pem \
    --query 'KeyMaterial' \
    --output text > veriqta-ha-webapp-key.pem
```

**What just happened:**
- `--key-name`: The name AWS uses to reference this key
- `--key-type rsa`: RSA algorithm (widely supported)
- `--key-format pem`: PEM format works with OpenSSH on all platforms
- The private key was saved to `veriqta-ha-webapp-key.pem`
- The public key was automatically stored in AWS

**Secure the private key immediately:**

```bash
# Linux/macOS: Restrict permissions so only you can read it
chmod 400 veriqta-ha-webapp-key.pem

# Verify
ls -l veriqta-ha-webapp-key.pem
# Should show: -r-------- 1 user group ... veriqta-ha-webapp-key.pem
```

**Windows:**
```powershell
# PowerShell: Set proper ACL
icacls veriqta-ha-webapp-key.pem /inheritance:r
icacls veriqta-ha-webapp-key.pem /grant:r "$($env:USERNAME):(R)"
```

> **CRITICAL:** If you lose this `.pem` file, you CANNOT recover it from AWS. You'd have to create a new key pair and replace it on all instances. Store it in a password manager or secure vault.

**Verify the key pair exists in AWS:**

```bash
aws ec2 describe-key-pairs --key-names veriqta-ha-webapp-key
```

Expected output:
```json
{
    "KeyPairs": [
        {
            "KeyPairId": "key-0123456789abcdef0",
            "KeyFingerprint": "1f:51:ae:28:bf:89:e9:d8:1f:25:5d:37:2d:7d:b8:ca",
            "KeyName": "veriqta-ha-webapp-key",
            "KeyType": "rsa",
            "Tags": [],
            "CreateTime": "2026-07-23T21:30:00.000Z"
        }
    ]
}
```

**Option B: Create via Console**
- EC2 → Key Pairs → Create key pair
- Name: `veriqta-ha-webapp-key`
- Key pair type: RSA
- Private key file format: `.pem`
- Download and secure the `.pem` file

### Step 2: Create an IAM Role for EC2 Instances

**The Problem:** If your web application needs to write logs to CloudWatch or access S3, you might be tempted to:
- Hardcode access keys in your application code ❌
- Store access keys in environment variables on the server ❌
- Bake access keys into your AMI ❌

**All of these are security risks.** If the instance is compromised, the attacker gets your long-lived credentials.

**The Solution: IAM Roles + Instance Profiles**

EC2 instances can assume an IAM Role automatically. AWS handles the credential rotation behind the scenes using the Instance Metadata Service (IMDSv2). Your application just works — no keys to manage.

---

### Step 3: Create the IAM Role with Trust Policy

First, create a trust policy file. This defines WHO can assume the role:

```bash
cat > ec2-trust-policy.json << 'EOF'
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Principal": {
                "Service": "ec2.amazonaws.com"
            },
            "Action": "sts:AssumeRole"
        }
    ]
}
EOF
```

**What this means:** Only EC2 instances (`ec2.amazonaws.com`) can assume this role.

**Create the role:**

```bash
aws iam create-role \
    --role-name Veriqta-EC2-WebApp-Role \
    --assume-role-policy-document file://ec2-trust-policy.json \
    --description "IAM role for HA web application EC2 instances"
```

Expected output includes:
```json
{
    "Role": {
        "RoleName": "Veriqta-EC2-WebApp-Role",
        "Arn": "arn:aws:iam::123456789012:role/Veriqta-EC2-WebApp-Role",
        ...
    }
}
```

**Save that ARN** — you'll need it when creating the Launch Template.

---

### Step 4: Create the Permission Policy for the Role

What should our web servers be allowed to do?

| Permission | Why |
|------------|-----|
| CloudWatch Logs | Write application logs |
| CloudWatch Metrics | Publish custom metrics |
| S3 (specific bucket) | Store/retrieve files if needed |
| Systems Manager | Use Session Manager for SSH-less access |

Create the permission policy:

```bash
cat > ec2-permission-policy.json << 'EOF'
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "CloudWatchLogs",
            "Effect": "Allow",
            "Action": [
                "logs:CreateLogGroup",
                "logs:CreateLogStream",
                "logs:PutLogEvents",
                "logs:DescribeLogGroups",
                "logs:DescribeLogStreams"
            ],
            "Resource": "*"
        },
        {
            "Sid": "CloudWatchMetrics",
            "Effect": "Allow",
            "Action": [
                "cloudwatch:PutMetricData",
                "cloudwatch:GetMetricStatistics",
                "cloudwatch:ListMetrics"
            ],
            "Resource": "*"
        },
        {
            "Sid": "S3AccessForALBLogs",
            "Effect": "Allow",
            "Action": [
                "s3:PutObject",
                "s3:GetObject",
                "s3:ListBucket"
            ],
            "Resource": [
                "arn:aws:s3:::veriqta-alb-logs-*",
                "arn:aws:s3:::veriqta-alb-logs-*/*"
            ]
        },
        {
            "Sid": "SSMForSessionManager",
            "Effect": "Allow",
            "Action": [
                "ssm:UpdateInstanceInformation",
                "ssmmessages:CreateControlChannel",
                "ssmmessages:CreateDataChannel",
                "ssmmessages:OpenControlChannel",
                "ssmmessages:OpenDataChannel",
                "ec2messages:AcknowledgeMessage",
                "ec2messages:DeleteMessage",
                "ec2messages:FailMessage",
                "ec2messages:GetEndpoint",
                "ec2messages:GetMessages",
                "ec2messages:SendReply"
            ],
            "Resource": "*"
        }
    ]
}
EOF
```

**Attach the policy to the role:**

```bash
aws iam put-role-policy \
    --role-name Veriqta-EC2-WebApp-Role \
    --policy-name Veriqta-EC2-WebApp-Permissions \
    --policy-document file://ec2-permission-policy.json
```

### Step 5: Create an Instance Profile

An **Instance Profile** is the container that lets EC2 instances assume an IAM Role. You attach the role to the instance profile, then attach the instance profile to the EC2 instance.

```bash
# Create the instance profile
aws iam create-instance-profile \
    --instance-profile-name Veriqta-EC2-WebApp-Profile

# Add the role to the instance profile
aws iam add-role-to-instance-profile \
    --instance-profile-name Veriqta-EC2-WebApp-Profile \
    --role-name Veriqta-EC2-WebApp-Role
```

**Verify everything is connected:**

```bash
aws iam get-instance-profile --instance-profile-name Veriqta-EC2-WebApp-Profile
```

Expected output shows the role attached:
```json
{
    "InstanceProfile": {
        "InstanceProfileName": "Veriqta-EC2-WebApp-Profile",
        "Roles": [
            {
                "RoleName": "Veriqta-EC2-WebApp-Role",
                "Arn": "arn:aws:iam::123456789012:role/Veriqta-EC2-WebApp-Role"
            }
        ],
        ...
    }
}
```

### Step 6: Understanding IMDSv2 (Instance Metadata Service v2)

AWS has two versions of the metadata service:

| | IMDSv1 | IMDSv2 |
|---|---|---|
| **Authentication** | None required | Session-based token required |
| **Security** | Vulnerable to SSRF attacks | Protected against SSRF |
| **Default** | Old default | New default (2023+) |

**Always use IMDSv2.** We'll enforce this when creating our Launch Template.

How it works:
1. EC2 instance requests a token from `169.254.169.254`
2. Token is valid for a short time
3. All metadata requests must include this token

Your applications using AWS SDKs automatically handle this. But if you're manually curling metadata, you need the token:

```bash
# Get token (valid for 6 hours)
TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" \
    -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

# Use token to get metadata
curl -H "X-aws-ec2-metadata-token: $TOKEN" \
    http://169.254.169.254/latest/meta-data/iam/info
```
