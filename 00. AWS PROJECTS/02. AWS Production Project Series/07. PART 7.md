## Part 7: NAT Gateways — Secure Outbound Internet for Private Subnets

Your private subnets are currently islands. They can't reach the internet at all — which means no software updates, no package installations, no AWS API calls that need public endpoints. But you also don't want them directly exposed to inbound internet traffic.

**NAT Gateway is the answer.** It sits in your public subnet, has a public IP, and translates outbound traffic from your private instances so they can reach the internet. Responses come back through the NAT Gateway, but no one on the internet can initiate connections to your private instances.

This is a **$32–45/month per NAT Gateway** service. In production, you need one per AZ for redundancy. For learning, you could use a single NAT Gateway to save money, but I'll show you the production pattern.

---

### Step 1: Allocate Elastic IPs for NAT Gateways

NAT Gateways need static public IPs. Elastic IPs (EIPs) are AWS's static IPv4 addresses.

**Why Elastic IPs instead of dynamic public IPs?**
- Dynamic IPs change if the NAT Gateway is recreated
- Elastic IPs persist across replacements
- You can remap them instantly if needed
- Required for production stability

**Allocate EIP for AZ-a:**
```bash
aws ec2 allocate-address \
    --domain vpc \
    --tag-specifications 'ResourceType=elastic-ip,Tags=[{Key=Name,Value=veriqta-nat-eip-1a},{Key=AZ,Value=us-east-1a},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

**Capture the Allocation ID:**
```json

    "AllocationId": "eipalloc-0a1b2c3d4e5f6789a",
    "PublicIp": "203.0.113.10"
}
```

### Step 3: Update Private Route Tables with NAT Gateway Routes

Now we give private subnets controlled outbound internet access.

**Add route to Private Route Table for AZ-a traffic:**
Wait — actually, we have ONE private route table shared across both private subnets. In production, you typically use **separate route tables per AZ** so each AZ routes to its own NAT Gateway. This prevents cross-AZ traffic charges and improves resilience.

**Let's create a second private route table for AZ-b:**

```bash
aws ec2 create-route-table \
    --vpc-id $VPC_ID \
    --tag-specifications 'ResourceType=route-table,Tags=[{Key=Name,Value=veriqta-private-rt-1b},{Key=AZ,Value=us-east-1b},{Key=Type,Value=Private},{Key=Project,Value=Veriqta-HA-WebApp}]'
```

```bash
PRIVATE_RT_1B_ID="rtb-0pri1b2b3c4d5e6f8"  # Replace with your actual ID
```

**Disassociate Private Subnet 1b from the original private RT and associate with the new one:**

```bash
# Get the association ID for private subnet 1b
ASSOC_ID_1B=$(aws ec2 describe-route-tables \
    --route-table-ids $PRIVATE_RT_ID \
    --query 'RouteTables[0].Associations[?SubnetId==`'$PRIVATE_SUBNET_1B'`].RouteTableAssociationId' \

    --route-table-id $PRIVATE_RT_1B_ID \
    --subnet-id $PRIVATE_SUBNET_1B
```

### Step 7: NAT Gateway Deep Dive — How It Actually Works

**The Packet Journey (Private Instance → Internet):**

```
1. Private instance (10.0.2.10) sends packet to 8.8.8.8
        ↓
2. Checks route table: 0.0.0.0/0 → NAT Gateway
        ↓
3. Packet sent to NAT Gateway (in public subnet, has Elastic IP 203.0.113.10)
        ↓
4. NAT Gateway replaces source IP with its Elastic IP
        ↓
5. Packet goes to IGW, then internet
        ↓
6. Response comes back to NAT Gateway's Elastic IP
        ↓
7. NAT Gateway remembers it translated for 10.0.2.10
        ↓
8. Replaces destination IP back to 10.0.2.10
        ↓
9. Packet delivered to private instance
```

**Key characteristics:**
- **Source IP is hidden:** Internet sees NAT Gateway's IP, not your private IP
- **Connection tracking:** NAT Gateway maintains state tables for active connections
- **No inbound initiation:** Internet cannot start connections to private instances
- **No port forwarding:** Unlike a traditional NAT router, you can't forward specific ports
- **Automatic scaling:** AWS handles bandwidth scaling (up to 45 Gbps per NAT Gateway)

**NAT Gateway vs. NAT Instance:**

| Feature | NAT Gateway | NAT Instance (legacy) |
|---------|-------------|----------------------|
| Availability | Highly available by design | You manage HA yourself |
| Bandwidth | Up to 45 Gbps | Depends on instance size |
| Maintenance | None (managed by AWS) | You patch the OS |
| Performance | No bottleneck | Single point of failure |
| Cost | ~$32.40/month + data processing | EC2 cost only |
| Use in 2026 | **Standard** | Legacy, avoid for new designs |

### Step 8: Cost Considerations

| Component | Cost (us-east-1) |
|-----------|-----------------|
| NAT Gateway per hour | $0.045 (~$32.40/month) |
| Data processing per GB | $0.045 |
| Elastic IP (attached) | Free |
| Elastic IP (unattached) | $0.005/hour (~$3.60/month) |

**For this project:** 2 NAT Gateways = ~$65/month + data charges

**Money-saving tips for learning:**
- Use a single NAT Gateway (not HA, but fine for learning)
- Delete NAT Gateways when not practicing
- Use NAT Instances only if cost is absolutely critical (not recommended)
- Consider AWS VPC Endpoints for S3 and DynamoDB to reduce NAT Gateway data charges
