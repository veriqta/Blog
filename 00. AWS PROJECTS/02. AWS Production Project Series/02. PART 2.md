## Part 2: Installing AWS CLI v2 & Configuring Credentials

Every production engineer works from the command line. The AWS Console is great for learning, but real work happens in scripts, pipelines, and terminals. Let's get your CLI set up properly.

---

### Step 1: Install AWS CLI v2

AWS CLI v2 is the current standard. Do NOT use v1 — it's in maintenance mode and lacks modern features like SSO integration and auto-prompting.

**For macOS:**

```bash
# Download the official installer
curl "https://awscli.amazonaws.com/AWSCLIV2.pkg" -o "AWSCLIV2.pkg"

# Install it
sudo installer -pkg AWSCLIV2.pkg -target /

# Verify installation
aws --version
# Expected output: aws-cli/2.x.x Python/3.x.x ...
```

**For Linux (x86_64):**

```bash
# Download the bundled installer
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"

# Unzip (install unzip if you don't have it: sudo apt install unzip)
unzip awscliv2.zip

# Install (use --update if upgrading)
sudo ./aws/install --bin-dir /usr/local/bin --install-dir /usr/local/aws-cli --update

# Verify
aws --version
```

**For Windows:**

1. Download the MSI installer from: `https://awscli.amazonaws.com/AWSCLIV2.msi`
2. Run the installer and follow the wizard
3. Open PowerShell and verify: `aws --version`

**For Linux (ARM/Graviton):**

```bash
curl "https://awscli.amazonaws.com/awscli-exe-linux-aarch64.zip" -o "awscliv2.zip"
unzip awscliv2.zip
sudo ./aws/install --bin-dir /usr/local/bin --install-dir /usr/local/aws-cli --update
```

> **Production Tip:** Always pin your CLI version in team environments. Different versions can behave differently. On shared machines, use a version manager or Docker container for CLI consistency.

### Step 2: Create an IAM User (NOT Root User)

**Never use your root account for daily work.** Root has unrestricted access to everything — billing, account deletion, password changes. If those credentials leak, your entire AWS account is compromised.

**Console Steps:**

1. Sign in to AWS Console as **root user**
2. Navigate to **IAM** → **Users** → **Create user**
3. Username: `veriqta-prod-engineer`
4. Check **"Provide user access to the AWS Management Console"** if you want console access too
5. For now, select **"Attach policies directly"**
6. Attach **"AdministratorAccess"** (we'll tighten this in Part 3 — for learning, this is acceptable in a sandbox)
7. Click **Create user**
8. Save the console password if you created one

---

### Step 3: Generate Access Keys for Programmatic Access

1. Go to **IAM** → **Users** → `veriqta-prod-engineer`
2. Click **Security credentials** tab
3. Scroll to **Access keys** → **Create access key**
4. Select **"Command Line Interface (CLI)"**
5. Check the box acknowledging the recommendation to use IAM Identity Center (we'll cover that in later projects)
6. Click **Create access key**
7. **IMPORTANT:** Copy the **Access Key ID** and **Secret Access Key** immediately, or download the `.csv` file. You cannot view the Secret Access Key again.

> **Production Reality:** In real environments, you'd use IAM Identity Center (SSO) or IAM Roles with OIDC federation instead of long-lived access keys. But for learning CLI basics, access keys are the starting point. We'll migrate to roles in Project 5 (CI/CD).

### Step 4: Configure AWS CLI

Run this command in your terminal:

```bash
aws configure
```

You'll be prompted for:

```
AWS Access Key ID [None]: AKIAIOSFODNN7EXAMPLE
AWS Secret Access Key [None]: wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
Default region name [None]: us-east-1
Default output format [None]: json
```

**Why `us-east-1`?**
- Most services launch here first
- Largest selection of instance types
- Slightly lower pricing on some services
- If you're in Europe, use `eu-west-1` (Ireland) or `eu-central-1` (Frankfurt)

**Verify your configuration:**

```bash
aws sts get-caller-identity
```

Expected output:
```json
{
    "UserId": "AIDAXXXXXXXXXXXXXXXX",
    "Account": "123456789012",
    "Arn": "arn:aws:iam::123456789012:user/veriqta-prod-engineer"
}
```

This confirms:
- Your credentials work
- You're authenticated as the correct user
- You know your AWS Account ID (save this — you'll need it)

### Step 5: Configure Multiple Profiles (Production Habit)

Real engineers work across multiple AWS accounts (dev, staging, prod). Set up profiles now:

```bash
# Configure a profile for this project
aws configure --profile veriqta-prod
```

This creates a separate profile. To use it:

```bash
aws sts get-caller-identity --profile veriqta-prod
```

Your credentials are stored in:
- **Linux/macOS:** `~/.aws/credentials` and `~/.aws/config`
- **Windows:** `%UserProfile%\.aws\credentials` and `%UserProfile%\.aws\config`

**View all profiles:**
```bash
aws configure list-profiles
```

**Set a profile as environment variable (temporary):**
```bash
export AWS_PROFILE=veriqta-prod
```

### Troubleshooting Common Issues

| Problem | Cause | Fix |
|---------|-------|-----|
| `aws: command not found` | PATH not updated | Close and reopen terminal, or add `/usr/local/bin` to PATH |
| `Unable to locate credentials` | Not configured or wrong profile | Run `aws configure` or specify `--profile` |
| `AccessDenied` | Insufficient permissions | Check IAM policies attached to user |
| `InvalidClientTokenId` | Wrong access key | Re-copy from IAM console |
| `SignatureDoesNotMatch` | Clock skew | Sync your system time (`sudo ntpdate pool.ntp.org`) |
