# 1. Introduction to Cloud Computing

## 1.1 What Is Cloud Computing?

Cloud computing is the delivery of computing resources over the internet when they are needed.

These resources can include:

* Virtual servers
* File storage
* Databases
* Networks
* Security tools
* Monitoring systems
* Application development tools
* Content delivery systems

Instead of buying and maintaining physical servers in your home or office, you can request computing resources from a cloud provider.

AWS defines cloud computing as the on-demand delivery of IT resources over the internet with pay-as-you-go pricing.

The phrase **on demand** means that you can create a resource when you need it.

For example, you can:

1. Sign in to AWS.
2. Launch a virtual server.
3. Use the server for a project.
4. Terminate the server when the project is complete.

You do not have to order a physical computer, wait for it to arrive, install it in a data centre, or replace damaged hardware yourself.

---

## 1.2 Traditional Infrastructure

Before cloud computing became widely available, organizations commonly purchased their own physical infrastructure.

A company that wanted to host a website might need to:

* Purchase physical servers.
* Rent or build a server room.
* Install networking equipment.
* Configure internet connectivity.
* Provide electricity and backup power.
* Install cooling systems.
* Replace failed hardware.
* Hire people to manage the equipment.
* Estimate future computing requirements.
* Purchase additional servers before traffic increased.

This approach can require a large initial investment.

It can also be difficult to predict exactly how much infrastructure an application will need.

If a company purchases too little capacity, the application may become slow or unavailable.

If the company purchases too much capacity, expensive equipment may remain unused.

---

## 1.3 Cloud Infrastructure

With cloud computing, the cloud provider owns and operates the underlying physical infrastructure.

The customer creates and manages the resources needed for a workload.

For example, when you launch an Amazon EC2 instance, AWS manages the physical data centre, physical server, networking infrastructure, electricity, cooling, and hardware replacement.

You manage the virtual server and the software you install on it.

This does not mean AWS manages everything for you. Your responsibilities depend on the service you use.

For an EC2 instance, you may still need to:

* Select the operating system.
* Install software.
* Update installed packages.
* Configure network access.
* Protect login credentials.
* Monitor the server.
* Remove the instance when it is no longer required.

Cloud computing changes how infrastructure is obtained, but it does not remove the need for security, monitoring, or responsible administration.

---

## 1.4 Pay-As-You-Go Pricing

Many AWS services use a pay-as-you-go model.

This means your charges are connected to the resources you create and consume.

Depending on the service, AWS may charge for:

* Time a resource runs
* Amount of data stored
* Number of requests
* Data transferred
* Provisioned capacity
* Public IPv4 addresses
* Additional features
* Backups and snapshots

Pay-as-you-go does not mean every AWS service is free.

It means you can use resources without purchasing the physical infrastructure yourself. You must still understand the pricing model for each service.

A small resource can also create charges if it remains active for a long time.

For this reason, every project in this volume includes:

* Cost warnings
* Validation steps
* Cleanup instructions
* Resource deletion steps

---

## 1.5 Benefits of Cloud Computing

### Speed

Cloud resources can often be created in minutes.

You can launch a server, create a storage bucket, or configure a network without waiting for physical equipment to arrive.

### Flexibility

You can choose resources according to your current requirements.

You can increase or reduce capacity as those requirements change.

### Global Reach

Cloud providers operate infrastructure in different parts of the world.

You can choose where supported resources are deployed.

### Reduced Initial Hardware Costs

You do not need to purchase a physical server for every project.

### Experimentation

You can create temporary environments for learning, testing, and development.

### Managed Services

Some AWS services manage more of the underlying infrastructure for you.

For example, Amazon S3 lets you store objects without creating or maintaining a traditional file server.

---

## 1.6 Important Cloud Computing Limitations

Cloud computing has many benefits, but it still requires careful planning.

Possible challenges include:

* Unexpected charges
* Incorrect security settings
* Dependence on internet connectivity
* Service quotas
* Regional service differences
* Vendor-specific knowledge
* Data transfer costs
* Misconfigured permissions
* Resources being created in the wrong Region
* Resources being forgotten after testing

A beginner should never assume that a resource is automatically secure or free.

You must learn how to:

* Configure access.
* Monitor usage.
* understand pricing.
* Delete resources.
* Protect credentials.
* Verify your work.

# 2. What Is Amazon Web Services?

Amazon Web Services, commonly called **AWS**, is a cloud computing platform provided by Amazon.

AWS offers services for areas such as:

* Computing
* Storage
* Databases
* Networking
* Security
* Monitoring
* Analytics
* Application integration
* Software development
* Artificial intelligence
* Content delivery

Organizations and individual learners can use AWS to build and operate applications without owning the underlying physical data centres.

AWS began offering infrastructure services as web services in 2006.

---

## 2.1 What Is an AWS Service?

An AWS service is a cloud capability that performs a particular function.

Examples include:

* Amazon S3 for object storage
* Amazon EC2 for virtual servers
* Amazon CloudFront for content delivery
* AWS Identity and Access Management for access control
* Amazon CloudWatch for monitoring
* Amazon RDS for managed relational databases
* Amazon VPC for virtual networking

You combine different services to build an architecture.

A complete application rarely uses only one service.

For example, a website might use:

* Amazon S3 to store website files.
* CloudFront to deliver those files.
* IAM to control administrative access.
* CloudWatch to record metrics and support monitoring.
* Route 53 to manage a custom domain name.

In Volume 1, you will begin with a small number of services so that you can understand each one clearly.

---

## 2.2 How You Interact with AWS

AWS resources can be managed in several ways.

### AWS Management Console

The AWS Management Console is a web-based interface for accessing and managing AWS resources.

You can use the console to:

* Search for AWS services.
* Create resources.
* Change configurations.
* Review resource status.
* View billing information.
* Monitor services.
* Delete resources.

Most of the steps in this beginner volume use the AWS Management Console.

### AWS Command Line Interface

The AWS Command Line Interface, or AWS CLI, lets you manage AWS services by entering commands in a terminal.

Example:

```bash
aws s3 ls
```

This command can list S3 buckets that the currently configured identity has permission to view.

You will install the AWS CLI later as an optional learning tool. The projects in this volume can still be completed mainly through the console.

### AWS Software Development Kits

AWS provides software development kits, usually called SDKs, for programming languages.

Developers can use SDKs to connect applications to AWS services.

Examples include SDKs for:

* Python
* JavaScript
* Java
* .NET
* Go
* Rust
* Ruby
* PHP

You will not need to use an AWS SDK for the first three projects.

### Infrastructure as Code

AWS resources can also be defined through configuration files and deployment tools.

Examples include:

* AWS CloudFormation
* AWS Cloud Development Kit
* Terraform

Infrastructure as Code becomes more important as environments grow. It is not required for the projects in this first volume.

# 3. Common AWS Services

AWS provides many services. You do not need to learn all of them at once.

Begin by learning the services that solve the requirements of your current project.

---

## 3.1 Amazon S3

**Amazon Simple Storage Service**, normally called **Amazon S3**, is an object storage service.

Amazon S3 is designed to store and retrieve objects at scale.

An object can be:

* An HTML file
* A CSS file
* An image
* A document
* A video
* A log file
* A backup
* An application file

Objects are stored inside containers called **buckets**.

Example:

```text
Bucket
└── Objects
    ├── index.html
    ├── style.css
    └── error.html
```

In Project 1, you will upload website files to an S3 bucket.

In Project 2, you will upload your cloud resume files to another S3 bucket.

S3 is object storage. It is not the same as logging in to a normal Linux server and browsing a traditional local disk.

---

## 3.2 Amazon CloudFront

Amazon CloudFront is a content delivery service.

It can distribute static and dynamic content through a worldwide network of points of presence, including edge locations. When a user requests content, CloudFront routes the request through infrastructure intended to provide low-latency delivery.

CloudFront can deliver files such as:

* HTML
* CSS
* JavaScript
* Images
* Videos
* Application responses

In this volume, CloudFront will sit in front of your S3 websites.

Without CloudFront:

```text
User → S3 website
```

With CloudFront:

```text
User → CloudFront → S3 website
```

CloudFront can provide:

* HTTPS access
* Content caching
* Global content delivery
* A CloudFront domain name
* Reduced repeated requests to the origin
* Additional security and delivery options

The system that stores or generates the original content is called the **origin**.

In your first projects, the S3 website will act as the CloudFront origin.

---

## 3.3 Amazon EC2

**Amazon Elastic Compute Cloud**, known as **Amazon EC2**, provides virtual computing capacity.

An EC2 virtual server is called an **instance**.

An instance can run an operating system such as:

* Amazon Linux
* Ubuntu
* Red Hat Enterprise Linux
* SUSE Linux
* Microsoft Windows Server

You can use an EC2 instance to:

* Host a website
* Run an application
* Perform testing
* Install development tools
* Run a background service
* Host internal software
* Process data

In Project 3, you will launch an Amazon Linux EC2 instance and install Nginx.

The architecture will be:

```text
User’s browser
      ↓
Security group
      ↓
EC2 instance
      ↓
Nginx
      ↓
HTML page
```

Unlike S3 static website hosting, EC2 gives you access to an operating system.

This also gives you more responsibility.

You must manage areas such as:

* Operating system updates
* Installed packages
* Running services
* Login access
* Network rules
* Server configuration
* Resource termination

---

## 3.4 AWS Identity and Access Management

**AWS Identity and Access Management**, commonly called **IAM**, helps control access to AWS services and resources.

IAM can be used to manage:

* Users
* Groups
* Roles
* Policies
* Permissions
* Access keys

A permission answers questions such as:

* Who may create an S3 bucket?
* Who may launch an EC2 instance?
* Who may delete a CloudFront distribution?
* Which actions may an application perform?
* Which resources may a user access?

IAM is a global service, so it is not managed in the same Region-specific way as many resources such as EC2 instances. The AWS console also notes that some services, including IAM, do not require the user to select a Region.

You will learn more about IAM in Part 1C.

## 3.5 Amazon VPC

**Amazon Virtual Private Cloud**, called **Amazon VPC**, provides a logically isolated virtual network for AWS resources.

A VPC can contain:

* Subnets
* Route tables
* Internet gateways
* Network connections
* Security controls
* Private and public resources

Your EC2 instance must run inside a VPC.

For the beginner EC2 project, you will use the default VPC rather than building a complete custom network.

A later volume will teach you how to create:

* A custom VPC
* Public subnets
* Private subnets
* Route tables
* Internet access
* Controlled communication between resources

---

## 3.6 Security Groups

A security group acts as a virtual firewall for supported AWS resources, including EC2 instances.

Security group rules control allowed inbound and outbound network traffic.

For the EC2 web server, you will use rules similar to:

```text
Inbound traffic

SSH   TCP 22   Your current public IP address
HTTP  TCP 80   Internet users
```

The SSH rule allows administrative access to the server.

The HTTP rule allows a web browser to request the website.

Security groups are stateful. When permitted traffic enters and the instance responds, the response traffic is automatically allowed.

You will configure the rules in Project 3.

It can work with:

For example, CloudWatch can help you observe:

## 3.8 AWS Billing and Cost Management

These tools include areas such as:

# 5. AWS Regions

## 5.1 What Is a Region?

An AWS Region is a separate geographical area where AWS operates infrastructure.

Examples of Region display names include:

* US East, N. Virginia
* US West, Oregon
* Europe, London
* Europe, Frankfurt
* Asia Pacific, Tokyo
* Asia Pacific, Sydney
* Africa, Cape Town

Each Region also has a code.

Examples:

```text
US East, N. Virginia    us-east-1
US West, Oregon         us-west-2
Europe, London          eu-west-2
Europe, Frankfurt       eu-central-1
Africa, Cape Town       af-south-1
```

A Region name and a Region code refer to the same Region.

The display name is easier for humans to read.

The Region code is commonly used in:

* Commands
* Configuration files
* API requests
* Resource identifiers
* Technical documentation

---

## 5.2 Regions Are Separate

AWS Regions are designed as separate geographical areas.

When you create many AWS resources, you choose the Region where they will exist.

For example, if you launch an EC2 instance in `eu-west-2`, you will not see it when your console is displaying `us-east-1`.

The instance has not disappeared. You are looking at a different Region.

This is one of the most common sources of confusion for AWS beginners.

Whenever a resource appears to be missing, check the selected Region first.

---

## 5.3 Regional and Global Services

Many AWS services are Regional.

Examples include:

* Amazon EC2
* Amazon VPC
* Amazon RDS
* Many CloudWatch resources
* Most S3 bucket data operations, although S3 presents some account-level views globally

Some services are considered global or have important global components.

Examples include:

* AWS Identity and Access Management
* Amazon CloudFront
* Amazon Route 53

The exact behavior depends on the service.

Do not assume that every AWS service uses Regions in exactly the same way.

---

## 5.4 Resources in Different Regions

Consider this example:

```text
Region: Europe, London
└── EC2 instance A

Region: US East, N. Virginia
└── EC2 instance B
```

Instance A and Instance B are different resources.

Stopping Instance A does not stop Instance B.

Deleting a resource in one Region does not automatically delete a similar resource in another Region.

During cleanup, check every Region you used.

---

# 6. Availability Zones

## 6.1 What Is an Availability Zone?

An Availability Zone, normally shortened to **AZ**, is an isolated location inside an AWS Region.

AWS describes Availability Zones as isolated locations within a Region. An Availability Zone consists of one or more discrete data centres with redundant power, networking, and connectivity.

A simplified view looks like this:

```text
AWS Region
├── Availability Zone A
├── Availability Zone B
└── Availability Zone C
```

AWS documentation states that each Region has at least three Availability Zones.

---

## 6.2 Region and Availability Zone Example

Consider the Region:

```text
Region: eu-west-2
```

It contains multiple Availability Zones, which may appear using identifiers such as:

```text
eu-west-2a
eu-west-2b
eu-west-2c
```

The final letter identifies an Availability Zone mapping in an AWS account.

For beginner learning, the main point is:

* A Region is a geographical area.
* An Availability Zone is an isolated infrastructure location inside that Region.

---

## 6.3 Why Availability Zones Exist

Availability Zones help customers design applications that are more resilient.

If an application runs in only one Availability Zone, a problem affecting that zone may make the application unavailable.

A more resilient application can place resources across multiple Availability Zones.

Example:

```text
                    Load balancer
                    /           \
                   /             \
        Availability Zone A   Availability Zone B
              Server 1             Server 2
```

If Server 1 or its Availability Zone becomes unavailable, traffic may be directed to Server 2.

This is called a **Multi-AZ architecture**.

You will not build a Multi-AZ EC2 application in Volume 1, but you should understand why multiple Availability Zones matter.

---

## 6.4 Availability Zones Are Connected

Availability Zones in the same Region are designed to be isolated from failures in other zones while still providing low-latency connectivity between zones.

This balance allows architects to:

* Separate resources.
* Replicate data.
* Distribute applications.
* Reduce the effect of a single-zone failure.

However, using multiple Availability Zones does not automatically make an application highly available.

The application, network, data, and traffic routing must also be designed correctly.

---

# 7. Edge Locations

## 7.1 What Is an Edge Location?

An Edge Location is part of AWS infrastructure positioned closer to users for services that benefit from fast delivery and low latency.

Amazon CloudFront uses edge infrastructure to distribute content.

When a visitor requests your website through CloudFront, the request is routed through CloudFront’s network rather than requiring every request to travel directly to the origin.

A simplified request path is:

```text
Website visitor
      ↓
Nearby CloudFront edge infrastructure
      ↓
CloudFront cache check
      ↓
S3 origin, when the file is not already cached
```

---

## 7.2 What Does Caching Mean?

Caching means temporarily storing a copy of frequently requested content closer to the user.

Suppose your `index.html` file is stored in an S3 bucket.

The first visitor requests the file:

```text
Visitor
   ↓
CloudFront
   ↓
S3 bucket
   ↓
index.html returned
```

CloudFront may cache the file.

A later visitor requests the same file:

```text
Visitor
   ↓
CloudFront cached copy
   ↓
index.html returned
```

This can reduce the time needed to deliver the content and reduce repeated requests to the origin.

---

## 7.3 Origin and Edge Location Are Different

The **origin** stores or generates the original content.

An **Edge Location** helps deliver content closer to users.

For Project 1:

```text
Origin: S3 website

Delivery layer: CloudFront

Visitor entry point: CloudFront domain
```

CloudFront does not replace your S3 bucket.

The S3 bucket remains the source of the original website files.

---

## 7.4 Region, Availability Zone, and Edge Location Compared

| Infrastructure concept | Meaning                           | Example use                          |
| ---------------------- | --------------------------------- | ------------------------------------ |
| Region                 | Separate geographical AWS area    | Choose where an EC2 instance runs    |
| Availability Zone      | Isolated location inside a Region | Distribute servers for resilience    |
| Edge Location          | Delivery location closer to users | Cache and deliver CloudFront content |

A simple mental model is:

```text
Region
└── Where many AWS resources are created

Availability Zone
└── An isolated location inside a Region

Edge Location
└── A location used to deliver content closer to users
```

# 8. How to Select an AWS Region

Selecting a Region is an architectural decision.

Do not choose a Region only because you recognize its name.

Consider the following factors.

---

## 8.1 Proximity to Users

A Region closer to your main users may provide lower network latency.

Latency is the time required for data to travel between locations.

If your application mainly serves users in one geographical area, a nearby Region may improve responsiveness.

CloudFront can improve global content delivery, but the location of your origin and other Regional services still matters.

---

## 8.2 Service Availability

Not every AWS service or feature is available in every Region at the same time.

Before selecting a Region for a production project, confirm that the required services and features are supported there.

For Volume 1, you need access to services such as:

* Amazon S3
* Amazon CloudFront
* Amazon EC2
* Amazon VPC
* IAM
* Amazon CloudWatch

---

## 8.3 Cost

AWS pricing can differ between Regions.

An EC2 instance type may cost more in one Region than another.

Data transfer pricing and other service costs may also vary.

For a real project, check current AWS pricing before selecting a Region.

---

## 8.4 Data Residency and Legal Requirements

Some organizations must store or process data in a particular country or geographical area.

Requirements can come from:

* Laws
* Contracts
* Industry standards
* Customer agreements
* Company policies

A personal beginner project may not have complex regulatory requirements, but you should understand that Region selection affects where Regional resources and data are located.

---

## 8.5 Sustainability Requirements

Some organizations consider environmental impact when selecting a Region.

This may form part of an organization’s sustainability strategy.

For a beginner project, this is normally a secondary consideration, but it becomes more important in larger environments.

---

## 8.6 Resource Availability and Quotas

Some resources, instance types, or features may not be available in every Region or Availability Zone.

AWS accounts also have service quotas.

If a particular resource cannot be created, investigate:

* The selected Region
* The selected Availability Zone
* Service availability
* Account quotas
* Account verification status

---

## 8.7 Use One Main Region During Training

For this beginner volume, choose one main Region and use it consistently where possible.

This makes it easier to:

* Find resources.
* Understand the architecture.
* Complete cleanup.
* Compare configurations.
* Avoid creating duplicate resources in different Regions.

CloudFront and IAM have global characteristics, but your EC2 and many supporting resources will use your selected Region.

Write down your chosen Region:

```text
My training Region:

Region name: ______________________

Region code: ______________________
```

---

## Recommended Beginner Region Selection Checklist

Before choosing a Region, ask:

* Is the Region close to me or my intended users?
* Are the required services available?
* Is the selected EC2 option available?
* Have I checked current pricing?
* Does the project have location or compliance requirements?
* Can I use this Region consistently for the exercises?

## 10.2 What Is a Static Website?

Common static website files include:

```text

style.css
script.js

error.html
```

The server delivers the stored files without generating each page through server-side application code.

Static websites are suitable for:

A static website does not normally perform server-side operations such as:

## 10.3 Project 1 Architecture

```text
┌──────────────────────┐
│   Website visitor    │
└──────────┬───────────┘
           │ HTTPS request
           ▼
┌──────────────────────┐
│   Amazon CloudFront  │
│   Content delivery   │
└──────────┬───────────┘
           │ Retrieves content
           │ when required

┌──────────────────────┐

└──────────┬───────────┘
           │
           ▼

│ index.html           │
│ style.css            │
│ error.html           │
└──────────────────────┘
```

---

## 10.4 How a Request Moves Through the Architecture

When a visitor opens the CloudFront domain:

1. The browser sends an HTTPS request.
2. CloudFront receives the request.
3. CloudFront checks whether it has a valid cached copy.
4. If the file is cached, CloudFront returns the cached copy.
5. If the file is not cached, CloudFront requests it from the S3 origin.
6. S3 returns the object.
7. CloudFront delivers the object to the browser.
8. CloudFront may cache the object for later requests.

## 12.4 What Is SSH?

SSH means **Secure Shell**.

It is a protocol used to connect securely to a remote computer.

For the project, an SSH command will look similar to:

```bash
ssh -i aws-beginner-key.pem ec2-user@PUBLIC-IP
```

The parts mean:

```text
ssh
└── Start the SSH client

-i aws-beginner-key.pem
└── Use the specified private key file

ec2-user
└── Connect using the Amazon Linux username

PUBLIC-IP
└── Address of the EC2 instance
```

You will learn how to use this command safely in Project 3.

## 12.5 What Is Nginx?

Nginx is software that can act as a web server.

A web server listens for browser requests and returns website content.

```text
        ↓
Nginx reads index.html
        ↓
Nginx sends the page
        ↓
Browser displays the page
```

You will manage Nginx using Linux commands such as:

# 17. Common Beginner Misunderstandings

## “AWS is one server.”

AWS is not one server.

It is a cloud platform containing many services and infrastructure locations.

---

## “The Region and Availability Zone are the same.”

They are not the same.

A Region is a geographical area.

An Availability Zone is an isolated location inside a Region.

---

## “CloudFront stores my original website.”

CloudFront can cache and deliver copies of content.

The origin stores or generates the original content.

In your first projects, the original files are stored in Amazon S3.

---

## “S3 is a Linux server.”

S3 is an object storage service.

You do not connect to it through SSH as though it were an EC2 instance.

---

## “EC2 automatically manages my Linux server.”

AWS manages the underlying physical infrastructure.

You remain responsible for many operating system and application tasks on an EC2 instance.

---

## “If I stop using a resource, AWS stops charging me.”

Charges do not stop merely because you close the browser or sign out.

You must stop, terminate, disable, or delete resources according to the service.

Some associated resources can continue generating charges after the main resource is stopped.

---

## “A budget prevents all spending.”

A budget primarily tracks spending or usage and provides configured notifications.

Do not treat it as a replacement for monitoring and cleanup.

---

## “My resource disappeared.”

The console may be displaying a different Region.

Check the Region selector before recreating the resource.

---

## “Anything in the Free Tier is always free.”

Free Tier eligibility and limits apply.

A service can generate charges when:

* Usage exceeds an allowance.
* The selected configuration is not covered.
* Eligibility has ended.
* Additional billable features are used.
* Related resources create separate charges.

---

## “Public means secure.”

A publicly available resource can be reached from the internet.

Public access must be intentional and limited to content that is safe to expose.

Never place credentials, private keys, personal documents, or confidential data inside a public website bucket.
