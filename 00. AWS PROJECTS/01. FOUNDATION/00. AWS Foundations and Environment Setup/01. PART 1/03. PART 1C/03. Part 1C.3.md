# Part 1C.3: Securing the AWS Root User

## Goal

In this lesson, you will secure the AWS account root user.

The root user is created automatically when an AWS account is registered. It has complete access to the account and can perform sensitive actions that ordinary identities may not be allowed to perform.

AWS strongly recommends that you do not use the root user unless a task specifically requires root credentials.

By the end of this chapter, you will:

* Understand what the AWS root user is.
* Understand the AWS Shared Responsibility Model.
* Recognize the risks associated with root access.
* Protect the root account email address.
* Create or confirm a strong root password.
* Enable multi-factor authentication.
* Understand the available MFA device types.
* Verify that root MFA is active.
* Understand root access keys.
* Confirm that no root access keys exist.
* Know when root credentials may be required.
* Know why root should never be used for everyday AWS work.
* Complete a root-account security checklist.
* Troubleshoot common MFA and sign-in problems.

Do not create AWS project resources until this lesson is complete.

---

# 1. The AWS Root User Explained

When you created your AWS account, you provided:

* An email address
* A password
* Contact information
* A telephone number
* A payment method

The combination of the account email address and root password forms the primary root-user sign-in credentials.

The root user is not an IAM user.

It is the original identity associated directly with the AWS account.

A simplified account structure looks like this:

```text
AWS Account
│
├── Root user
│   ├── Account email address
│   ├── Root password
│   ├── MFA device
│   └── Complete account authority
│
├── IAM users
├── IAM groups
├── IAM roles
├── IAM policies
└── AWS resources
```

The root user exists at the account level.

An IAM user exists inside the AWS account and receives permissions through IAM policies.

---

## 1.1 Root User vs IAM User

The root user and an IAM user are different identities.

| Root user                                        | IAM user                                    |
| ------------------------------------------------ | ------------------------------------------- |
| Created automatically with the AWS account       | Created inside IAM                          |
| Signs in using the account email address         | Signs in using an IAM username              |
| Has complete access by default                   | Has only the permissions granted to it      |
| Cannot be restricted using ordinary IAM policies | Can be restricted using IAM policies        |
| Reserved for limited account-level tasks         | Can be used for specific operational tasks  |
| Must be protected with MFA                       | Should also be protected with MFA           |
| Should not be used for daily work                | May be used where IAM users are appropriate |

The root user should not be treated as your normal AWS administrator.

It should be treated as an emergency and account-ownership identity.

---

## 1.2 Why the Root User Is So Powerful

The root user begins with complete access to the account.

It can potentially:

* Change account-level information.
* Change the root password.
* Manage root security credentials.
* Change the account email address.
* Access billing settings.
* Activate IAM access to billing.
* Restore administrative access.
* Close the AWS account.
* Create or delete AWS resources.
* Change IAM permissions.
* Create access keys.
* Modify security settings.

Some sensitive account-management and recovery tasks specifically require root credentials. AWS provides a documented list of tasks that may require root-user access.

Because the root user is so powerful, an attacker who obtains its credentials may be able to affect the entire account.

---

## 1.3 The Root User Cannot Be Deleted

You do not create and delete the root user like an IAM user.

It remains associated with the AWS account.

Your security goal is therefore to:

1. Protect the root email address.
2. Protect the root password.
3. Enable MFA.
4. Avoid root access keys.
5. Keep recovery information current.
6. Stop using the root user for routine activity.
7. Sign in only when a root-required task must be completed.

# 2. Understanding the AWS Shared Responsibility Model

AWS security is a shared responsibility between AWS and the customer.

AWS describes this as:

```text
AWS:
Security OF the cloud

Customer:
Security IN the cloud
```

AWS operates and protects the infrastructure that runs AWS services, including facilities, physical hardware, networking infrastructure, and underlying virtualization components. Customers remain responsible for the security configuration and management of the AWS services and resources they use.

---

## 2.1 AWS Responsibilities

AWS is generally responsible for protecting the infrastructure that supports AWS services.

Examples include:

* Physical data centres
* Physical servers
* Physical storage devices
* Core networking infrastructure
* Hardware maintenance
* Environmental controls
* The virtualization layer
* Infrastructure availability
* Physical access controls

As an AWS customer, you do not:

* Guard the AWS data-centre entrance.
* Replace damaged physical disks.
* Repair AWS networking equipment.
* Maintain the physical EC2 host.
* Operate the data-cententre cooling system.

AWS manages these responsibilities.

---

## 2.2 Customer Responsibilities

You are responsible for how you configure and use your AWS account.

Examples include:

* Protecting the root user.
* Enabling MFA.
* Controlling IAM permissions.
* Creating secure passwords.
* Protecting access keys.
* Configuring security groups.
* Protecting data.
* Selecting appropriate Regions.
* Reviewing billing.
* Deleting unused resources.
* Applying operating-system updates where required.
* Configuring logging and monitoring.
* Preventing public exposure.
* Responding to compromised credentials.

AWS may secure the physical infrastructure, but it cannot decide which person should have access to your AWS account.

That is your responsibility.

---

## 2.3 Root Security Under Shared Responsibility

Consider this example:

```text
AWS responsibility:
Protect the infrastructure running IAM and the AWS sign-in systems.

Customer responsibility:
Protect the root email, password, MFA device, recovery methods,
and account permissions.
```

If you choose a weak root password or share the root credentials, that is not a failure of AWS physical infrastructure.

It is a customer-side security failure.

---

## 2.4 Service Choice Changes Your Responsibility

Your responsibilities can change depending on the AWS service you choose.

For example:

### Amazon EC2

AWS manages:

* Physical hardware
* Data-centre security
* The virtualization layer

You manage:

* Guest operating system
* Software updates
* Applications
* Security groups
* User access
* Data
* Credentials

### Amazon S3

AWS manages:

* Physical storage infrastructure
* Service availability
* Underlying hardware

You manage:

* Bucket permissions
* Public access settings
* Bucket policies
* Object permissions
* Encryption choices
* Data classification

No matter which service you use, identity protection remains one of your responsibilities.

---

## 2.5 Beginner Security Principle

Remember:

```text
AWS gives you security controls.

You are responsible for configuring and using them correctly.
```

Creating an AWS account does not automatically mean every security setting is complete.

You must deliberately protect the account.

# 3. Root User Security Risks

The root user is dangerous when used casually because it combines enormous authority with long-term credentials.

AWS identifies root-user credentials as highly privileged and recommends closely protecting them and using MFA.

---

## 3.1 Password Theft

An attacker may obtain a password through:

* Phishing
* Password reuse
* Malware
* Browser compromise
* Shared devices
* Insecure notes
* Public screenshots
* Email compromise
* Data breaches on another website
* Social engineering

If the AWS root password is reused on another website, a breach of that website may put the AWS account at risk.

---

## 3.2 Email Account Compromise

The root user signs in using the AWS account email address.

The same email address may also be involved in:

* Password recovery
* Security notifications
* Billing alerts
* Account verification
* Support communication

If an attacker controls both the email account and the AWS password, account recovery becomes much more difficult.

Protecting the root email address is therefore part of protecting AWS.

---

## 3.3 Missing MFA

Without MFA, possession of the root email and password may be enough to attempt account access.

MFA adds another authentication requirement.

An attacker who steals only the password should still need the configured MFA factor.

MFA does not remove the need for a strong password, but it provides an important additional security layer.

---

## 3.4 Root Access Keys

Access keys are credentials used for programmatic AWS access.

They can be used with:

* AWS CLI
* AWS SDKs
* API requests
* Automation tools
* Scripts
* Applications

A root access key inherits the root user’s extremely powerful access.

If exposed, it can create severe account risk.

AWS strongly recommends that customers do not create root access-key pairs.

---

## 3.5 Public Screenshots

A screenshot can accidentally reveal:

* Account email
* Account ID
* Resource names
* Billing information
* Public IP addresses
* MFA device names
* Access key identifiers
* Personal contact information

Always review and crop screenshots before sharing them.

Never include:

* Passwords
* Secret access keys
* MFA codes
* QR codes used during MFA enrollment
* Payment details
* Private keys

---

## 3.6 Root Session Left Open

A root session left open on an unattended computer may allow someone else to access the account.

Risks increase when using:

* Shared computers
* Public computers
* School or library computers
* Remote desktop sessions
* Screen-sharing sessions
* Browsers with untrusted extensions

Always sign out when the root task is complete.

---

## 3.7 Accidental Changes

Not every root incident is caused by an attacker.

A beginner signed in as root can accidentally:

* Delete a resource.
* Change billing settings.
* Make an S3 bucket public.
* Modify IAM permissions.
* Create expensive resources.
* Disable a security feature.
* Create unnecessary credentials.
* Close or damage the account.

Using a separate operational identity reduces the chance that every routine action is performed with unrestricted authority.

---

## 3.8 Phishing

A fake website may imitate the AWS sign-in page.

Before entering root credentials:

* Use a trusted bookmark.
* Check that the domain belongs to AWS.
* Avoid unexpected sign-in links from emails.
* Do not enter credentials after clicking suspicious advertisements.
* Do not share MFA codes with anyone.
* Do not approve an unexpected authentication request.

AWS Support should not need your password, secret access key, or MFA code.

---

# 4. Protecting the Root Email Address

The root email address is part of the account’s security and recovery system.

Use an email account that:

* You control.
* You monitor regularly.
* Is not shared casually.
* Will remain available long term.
* Has its own strong password.
* Has MFA enabled.
* Has accurate recovery information.

---

## 4.1 Secure the Email Account First

Before proceeding, secure the email account associated with AWS.

Confirm:

* The email password is unique.
* Email MFA is enabled.
* Recovery telephone information is current.
* Recovery email information is current.
* Unknown sessions have been signed out.
* Suspicious forwarding rules do not exist.
* Security alerts are enabled.

An AWS account cannot be considered secure while its root email account is weak.

---

## 4.2 Do Not Use a Temporary Address

Avoid:

* Disposable email addresses
* School addresses that may expire
* Employer-controlled addresses you may lose
* Shared household addresses
* Addresses that another person administers

Losing control of the root email can make account recovery difficult.

---

## 4.3 Monitor AWS Messages

Pay attention to AWS emails involving:

* Sign-in attempts
* Password changes
* MFA changes
* Account information changes
* Billing activity
* Support cases
* New services or resources
* Security incidents

Do not automatically trust every email claiming to be from AWS.

Open the AWS console through a trusted route and verify the information directly.

---

# 5. Creating a Strong Root Password

The root password should be:

* Long
* Unique
* Difficult to guess
* Stored securely
* Never reused
* Never shared

AWS recommends keeping the root password private and using a different password from the passwords used on other websites.

---

## 5.1 Password Length

Length is one of the most important characteristics of a strong password.

A long password is generally more resistant to guessing than a short password.

Use a password manager to generate a long, random password.

Do not create a password that is merely:

```text
Password123!
```

This includes several character types but remains easy to guess.

---

## 5.2 Avoid Personal Information

Do not build the password from:

* Your name
* Child’s name
* Business name
* Birthday
* Phone number
* Address
* Favourite team
* Social-media username
* Common phrase
* AWS account name

Attackers may obtain personal information from public profiles.

---

## 5.3 Never Reuse the Password

The AWS root password must not be the same as:

* Email password
* GitHub password
* Banking password
* Social-media password
* Computer login password
* IAM-user password
* Any other cloud account password

Password reuse allows one compromised account to endanger another.

---

## 5.4 Use a Password Manager

A password manager can:

* Generate long random passwords.
* Store passwords securely.
* Reduce password reuse.
* Help identify duplicate passwords.
* Make strong passwords practical.

Protect the password manager itself with:

* A strong master password
* MFA
* Secure recovery information
* Device protection

---

## 5.5 Do Not Store the Password in Project Files

Never place the root password in:

```text
README.md
notes.txt
passwords.txt
index.html
config.json
.env
Git repository
Cloud-init script
Terminal command history
Source code
```

Do not send it through:

* Email
* Chat
* Social media
* Screenshots
* Shared documents

---

## 5.6 Changing the Root Password

Change the root password when:

* You suspect exposure.
* It was reused elsewhere.
* Another person knows it.
* It appeared in a screenshot.
* Your password manager reports compromise.
* An unauthorized sign-in may have occurred.
* AWS directs you to reset it.

Do not change it repeatedly without a reason if that causes insecure storage or forgotten passwords.

---

## 5.7 Root Password Verification

Confirm:

* [ ] The root password is unique.
* [ ] It is long and difficult to guess.
* [ ] It is stored in a password manager.
* [ ] It is not stored in a project folder.
* [ ] It is not used by the IAM administrator.
* [ ] It has not been shared with anyone.
* [ ] The root email account also has a unique password.

---

# 6. Understanding Multi-Factor Authentication

Multi-factor authentication requires more than one form of authentication.

Without MFA:

```text
Email address + password
```

With MFA:

```text
Email address + password + MFA factor
```

The MFA factor may be generated or provided by:

* An authenticator application
* A passkey
* A hardware security key
* A hardware time-based token

AWS supports MFA for the root user and strongly recommends enabling it.

---

## 6.1 Why MFA Matters

Suppose an attacker learns your root password.

Without MFA:

```text
Stolen password
      ↓
Possible account sign-in
```

With MFA:

```text
Stolen password
      ↓
Additional MFA required
      ↓
Sign-in attempt may be blocked
```

MFA makes password theft alone less useful.

It does not protect against every possible attack, but it significantly improves account security.

---

## 6.2 MFA Device Options

AWS supports several root-user MFA methods.

The options shown in the console may include:

* Passkey or security key
* Authenticator application
* Hardware TOTP token

AWS documentation provides root-user setup procedures for virtual authenticator applications, passkeys or security keys, and hardware TOTP devices.

---

## 6.3 Authenticator Application

An authenticator application generates temporary codes.

During setup, AWS displays a QR code or secret configuration key.

You scan or enter the information into the authenticator application.

The application then produces codes that change regularly.

Advantages include:

* Easy beginner setup
* No mobile network required after configuration
* Widely supported
* Often available at no additional cost

Risks include:

* Losing the phone
* Resetting the device without backup
* Accidentally deleting the authenticator entry
* Exposing the enrollment QR code
* Failing to transfer the MFA configuration to a new phone

---

## 6.4 Passkey or Security Key

A passkey can use a supported device or password manager.

A hardware security key is a physical device used during authentication.

Advantages can include:

* Strong phishing resistance
* No manually typed one-time code
* Convenient authentication
* Physical control of the security key

Risks include:

* Losing the device
* Hardware damage
* Unsupported browser or device configuration
* Failure to register a backup method

---

## 6.5 Hardware TOTP Token

A hardware TOTP token is a dedicated physical device that generates temporary codes.

Advantages include:

* Separate from the mobile phone
* Dedicated security purpose
* No application installation

Risks include:

* Loss
* Damage
* Battery failure
* Replacement cost
* Availability limitations

---

## 6.6 Multiple MFA Devices

AWS recommends registering multiple MFA devices for highly privileged identities, including the root user. This can increase security and reduce the risk of being locked out when one device is unavailable.

Where supported, a sensible arrangement is:

```text
Primary MFA:
Authenticator, passkey, or security key

Backup MFA:
A separately protected second device
```

Do not store both devices together if one event could cause both to be lost.

---

# 7. Preparing to Enable Root MFA

Before beginning:

* Confirm you are signed in as the root user.
* Confirm you have access to the root email.
* Confirm your device date and time are correct.
* Prepare the chosen MFA device.
* Ensure the internet connection is stable.
* Close unnecessary screen-sharing tools.
* Do not take screenshots of the QR code.
* Do not publish the setup key.
* Do not leave the page unattended.

---

## 7.1 Confirm the Current Identity

Open the account menu.

Look for indications that you are signed in as the root user.

The root user normally signs in using the account email address rather than:

* IAM username
* Account alias and username
* IAM role
* Federated identity

Do not attach root MFA while signed in as an IAM user.

The MFA device is attached to the identity currently being configured.

---

## 7.2 Protect the Enrollment QR Code

The QR code displayed during authenticator setup contains information used to configure the MFA generator.

Treat it as sensitive.

Do not:

* Screenshot it.
* Email it.
* Place it in documentation.
* Upload it to cloud storage without proper security.
* Share it in a class group.
* Include it in a tutorial video.
* Leave it visible during a livestream.

Someone who captures the enrollment information may be able to generate matching MFA codes.

---

# 8. Enabling MFA for the Root User

The exact labels may change as AWS updates the console, but the main process remains similar.

AWS’s current root-user procedure begins by signing in with root credentials, opening the account menu, and selecting **Security credentials**.

---

## Step 1: Sign In as the Root User

1. Open the AWS sign-in page through a trusted browser bookmark.
2. Choose **Root user**.
3. Enter the AWS account email address.
4. Select **Next**.
5. Enter the root password.
6. Complete any verification AWS requests.
7. Wait for Console Home to load.

---

## Step 2: Open Security Credentials

1. Locate the account menu in the upper-right area.
2. Select the account name or account identifier.
3. Choose **Security credentials**.
4. Review the security page carefully.

You may see a warning that the root user has unrestricted account access.

---

## Step 3: Locate Multi-Factor Authentication

Find the section named:

```text
Multi-factor authentication (MFA)
```

The page may indicate that MFA is:

```text
Not assigned
```

or:

```text
Disabled
```

AWS documents that the MFA status can be checked in the Multi-factor Authentication section of the Security credentials page.

---

## Step 4: Choose to Assign an MFA Device

Select the option that may be labelled:

```text
Assign MFA device
```

or similar wording.

AWS may ask you to enter an MFA device name.

Choose a descriptive name that identifies the device without exposing sensitive personal details.

Examples:

```text
root-primary-authenticator
root-security-key
root-backup-mfa
```

Avoid unclear names such as:

```text
phone
device1
test
new
```

---

## Step 5: Choose the MFA Device Type

Select the device type you prepared.

Possible choices include:

* Passkey or security key
* Authenticator app
* Hardware TOTP token

The following detailed procedure uses an authenticator application because it is commonly available to beginners.

---

# 9. Configuring an Authenticator Application

## Step 1: Open the Authenticator

Open your selected authenticator application on the secured device.

Protect the device with:

* PIN
* Password
* Biometrics
* Automatic screen locking
* Device encryption where supported

---

## Step 2: Add a New Account

In the authenticator application, choose the option to add an account.

This may be represented by:

* A plus icon
* Add account
* Scan QR code
* Enter setup key

Choose the QR-code scanning option when available.

---

## Step 3: Scan the AWS QR Code

Point the device camera at the QR code displayed in the AWS console.

After scanning, the authenticator application should create a new entry for AWS.

Check that the entry is clearly identifiable.

Do not share or save an image of the QR code.

---

## Step 4: Enter Consecutive MFA Codes

AWS may request two consecutive codes from the authenticator application.

For example:

```text
MFA code 1: ______

Wait for the code to change.

MFA code 2: ______
```

Do not enter the same code twice unless the AWS screen specifically instructs you to do so.

Enter the first current code.

Wait for the authenticator to generate the next code.

Enter the next code.

---

## Step 5: Complete the Assignment

Select the button that completes the MFA assignment.

The wording may include:

```text
Add MFA
Assign MFA
Complete
```

Wait for a success message.

Do not close the page until AWS confirms that the device was assigned successfully.

---

## Step 6: Verify the Displayed Status

Return to the MFA section of the Security credentials page.

Confirm that it shows:

* An assigned MFA device
* The device name
* The MFA type
* An enabled or active status

Do not assume setup succeeded merely because you scanned the QR code.

The AWS console must confirm the assignment.

---

# 10. Configuring a Passkey or Security Key

The exact prompts depend on the browser, operating system, password manager, or hardware key.

A general workflow is:

1. Sign in as root.
2. Open **Security credentials**.
3. Find the MFA section.
4. Choose **Assign MFA device**.
5. Select **Passkey or security key**.
6. Enter a descriptive device name.
7. Follow the browser or operating-system prompt.
8. Touch, unlock, or approve the security key or passkey.
9. Complete the registration.
10. Confirm that AWS shows the device as assigned.

AWS provides a dedicated procedure for enabling a passkey or security key for the root user.

Do not register a passkey on a shared device.

---

# 11. Testing Root MFA

Never wait for an emergency to discover that MFA does not work.

Test it after setup.

---

## Step 1: Confirm You Can Recover Your Password Manager

Before signing out, confirm that you can access the stored root password independently.

Do not rely on a password that exists only in an active browser session.

---

## Step 2: Confirm the MFA Device Is Available

Check that:

* The authenticator entry exists.
* Codes are being generated.
* The passkey is available.
* The security key is available.
* The hardware token is working.

---

## Step 3: Sign Out

1. Open the account menu.
2. Select **Sign out**.
3. Wait for the AWS sign-out page.
4. Close the browser tab.

Do not simply close the console without selecting Sign out.

---

## Step 4: Sign In Again as Root

1. Open the AWS sign-in page.
2. Choose **Root user**.
3. Enter the root email address.
4. Enter the root password.
5. Provide the MFA factor.
6. Confirm that Console Home opens.

A successful sign-in proves that:

* The password works.
* The MFA device works.
* The correct device was enrolled.
* You understand the sign-in process.

---

## Step 5: Sign Out Again

After the test:

1. Open the account menu.
2. Select **Sign out**.
3. Do not continue browsing as root.

You have confirmed the root identity.

The next everyday identity will be created in Part 1C.4.

# 12. Root Access Keys Explained

Access keys are used for programmatic authentication.

An access key normally consists of:

```text
Access key ID
+
Secret access key
```

They can be used by tools such as:

* AWS CLI
* AWS SDKs
* Scripts
* Applications
* Automation platforms
* API clients

Access keys are not the same as console passwords.

---

## 12.1 Console Credentials vs Programmatic Credentials

### Console sign-in

Typically uses:

```text
Username or email
+
Password
+
MFA
```

### Long-term programmatic access

Traditionally uses:

```text
Access key ID
+
Secret access key
```

For human access, AWS increasingly recommends federation and temporary credentials rather than long-term IAM-user access keys.

The important rule for this chapter is:

```text
Never create root access keys for these projects.
```

---

## 12.2 Why Root Access Keys Are Dangerous

Root access keys can potentially provide programmatic access with root-level authority.

They may be exposed through:

* Git repositories
* Source code
* Configuration files
* Terminal history
* CI/CD logs
* Shared folders
* Screenshots
* Unencrypted backups
* Malware
* Public code examples

Because the root identity has unrestricted authority, the impact can be severe.

AWS strongly recommends not creating root access keys.

---

## 12.3 Installing the AWS CLI Does Not Require Root Keys

You installed the AWS CLI in Part 1B.

Installation does not require AWS credentials.

You can verify the installation with:

```bash
aws --version
```

That command displays the installed AWS CLI version.

It does not require:

* Root password
* Root access key
* IAM access key
* An active AWS session

Do not create root access keys simply because the CLI is installed.

---

# 13. Checking for Root Access Keys

While signed in as root:

1. Open the account menu.
2. Select **Security credentials**.
3. Locate the **Access keys** section.
4. Check whether any access keys are listed.

For a secure beginner account, the expected result is:

```text
Root access keys: None
```

or equivalent wording.

---

## 13.1 When No Root Access Keys Exist

Do not select:

```text
Create access key
```

Leave the section unchanged.

Record:

```text
Root access-key status: No root access keys
```

Do not publish a screenshot of the security page.

---

## 13.2 When a Root Access Key Exists

Do not copy it into project files.

Do not begin using it.

First determine:

* Who created it?
* Why was it created?
* Is any legitimate system using it?
* Has it ever been exposed?
* Can it be replaced with a safer identity or temporary credentials?

For a new personal training account, an unexpected root access key is a serious warning.

A safe remediation process generally involves:

1. Stop relying on the root key.
2. Replace any legitimate usage with an appropriate identity.
3. Review recent account activity.
4. Inactivate the key when safe.
5. Delete the key after confirming it is no longer needed.
6. Investigate any unexplained usage.
7. Review billing and resources.

Do not delete an unknown key in an established business environment without understanding its dependencies. For this new beginner account, there should normally be no root access key at all.

---

## 13.3 Never Test a Root Access Key

Do not create a root access key “just to test the CLI.”

A credential is not harmless because it was created temporarily.

Temporary experiments can become permanent security risks when:

* The key is forgotten.
* The key is copied into a file.
* The file is committed to Git.
* The key remains active.
* The secret cannot be recovered later.
* Another person gains access to the machine.

---

# 14. Why Root Must Not Be Used for Daily Work

AWS strongly recommends avoiding root-user access except for tasks that require it.

Daily work includes:

* Creating S3 buckets
* Uploading website files
* Launching EC2 instances
* Configuring security groups
* Creating CloudFront distributions
* Viewing CloudWatch metrics
* Managing ordinary resources
* Running normal project commands
* Learning the console
* Testing application deployments

These activities should not be performed routinely as root.

---

## 14.1 Root Has Too Much Authority

The principle of least privilege means an identity should receive only the permissions necessary to perform its role.

The root user violates this principle for routine work because it always has complete account authority.

You cannot reduce the root user to an ordinary project role through a standard IAM policy.

---

## 14.2 Mistakes Have a Larger Impact

Consider two identities:

```text
Project identity:
Can manage only project resources.

Root user:
Can control the entire account.
```

If both identities make the same accidental action, the root user may have access to far more sensitive settings.

---

## 14.3 Root Activity Is Harder to Separate

Using named identities creates clearer accountability.

For example:

```text
ann-admin
deployment-role
billing-viewer
application-role
```

These identities indicate who or what performed an action.

Using root for everything reduces the distinction between:

* Account ownership
* Administration
* Project activity
* Automation
* Emergency recovery

---

## 14.4 Root Credentials Are Long-Term Credentials

The root email and password remain associated with the account.

They are not temporary project credentials.

Long-term credentials must be protected carefully because they remain valuable to attackers.

---

## 14.5 Daily Use Increases Exposure

Every root sign-in creates another opportunity for:

* Phishing
* Screen exposure
* Session theft
* Password-manager mistakes
* Browser compromise
* Accidental configuration
* Credential sharing

Using root less often reduces exposure.

---

# 15. Tasks That May Require Root Access

The root user should be used only when a task specifically requires it.

Depending on account type and AWS configuration, examples may include:

* Changing certain root-user credentials.
* Changing the account email address.
* Recovering from loss of administrative permissions.
* Activating IAM access to Billing and Cost Management.
* Closing a standalone AWS account.
* Performing certain account or billing tasks.
* Recovering from some resource-policy lockouts.
* Completing specific service actions reserved for root.

AWS maintains the authoritative list of tasks that require root credentials.

Do not assume that an important task automatically requires root.

First check whether an administrative identity can perform it.

---

## 15.1 Root-Use Decision

Before signing in as root, ask:

```text
1. Does AWS specifically require root credentials?
2. Can an administrative identity perform this task?
3. Is this an account-level or recovery operation?
4. Can I complete the task quickly and sign out?
5. Is the device and network trusted?
```

When the answer to the first question is no, do not use root.

# 16. Root-Account Operating Procedure

Use this procedure whenever a legitimate root task is required.

## Before Signing In

* Identify the exact task.
* Confirm that root is required.
* Use a trusted computer.
* Use a trusted network.
* Close screen-sharing software.
* Open AWS through a trusted bookmark.
* Have the MFA device available.

## During the Session

* Perform only the required task.
* Do not browse unrelated services.
* Do not create root access keys.
* Do not leave the computer unattended.
* Do not publish screenshots.
* Verify every change.

## After the Task

* Confirm the intended result.
* Record the administrative change when appropriate.
* Review unexpected notifications.
* Sign out.
* Close the tab.
* Return to the normal operational identity.

# 18. Troubleshooting MFA Problems

## Problem 1: AWS Rejects the Authenticator Codes

Possible causes include:

* Device time is incorrect.
* The wrong authenticator entry is being used.
* The code expired while being entered.
* The same code was entered twice during setup.
* The QR code was scanned more than once.
* MFA setup was not completed.
* The device is linked to another AWS identity.

Try:

1. Confirm the correct authenticator entry.
2. Wait for a new code.
3. Enter the code promptly.
4. Enable automatic date and time on the device.
5. Confirm the device time zone.
6. Return to the MFA section and check whether the device is actually assigned.
7. Restart the setup if AWS did not confirm completion.

---

## Problem 2: Two Consecutive Codes Are Rejected

During enrollment, AWS may request consecutive codes.

Follow this sequence:

1. Wait for a fresh code.
2. Enter it as the first code.
3. Wait until the application displays a different code.
4. Enter the new code as the second code.
5. Submit before the second code expires.

Do not enter:

```text
Code 1: 123456
Code 2: 123456
```

The codes normally need to come from consecutive time periods.

---

## Problem 3: The Authenticator Entry Disappeared

Possible causes include:

* The application was deleted.
* The phone was reset.
* Application data was cleared.
* The entry was removed.
* The device was replaced without migration.

Use another registered MFA method when available.

AWS provides recovery procedures for a lost or unavailable root MFA device, including alternative authentication factors where available.

Do not repeatedly guess MFA codes.

---

## Problem 4: The Phone Was Lost or Damaged

First:

* Do not panic.
* Do not create another AWS account.
* Do not share the password with anyone offering help.
* Do not provide MFA codes to unknown people.

Try:

1. Use another registered MFA device.
2. Use the available alternative authentication process.
3. Confirm access to the root email.
4. Confirm access to the account telephone number.
5. Follow AWS account-recovery instructions.
6. Contact AWS account support through official channels when required.

After recovering access:

* Remove the lost device.
* Register a replacement MFA method.
* Review recent activity.
* Review billing.
* Change credentials if compromise is suspected.

---

## Problem 5: The Security Key Is Not Detected

Check:

* The key is connected correctly.
* The browser supports the key.
* The correct browser profile is being used.
* The operating system recognizes the device.
* USB or NFC access is available.
* The key is not damaged.
* You are using the security key registered with this account.

Try another supported browser or device only when it is trusted.

---

## Problem 6: The Passkey Prompt Does Not Appear

Possible causes include:

* Browser incompatibility
* Operating-system limitations
* Password-manager configuration
* Private browsing restrictions
* The passkey is stored on another device
* Browser pop-ups or prompts are blocked

Check the browser and device passkey settings.

Do not register the passkey on a public or shared device.

---

## Problem 7: MFA Appears Enabled but Sign-In Does Not Request It

Check:

* You signed out completely.
* You are signing in as the root user.
* You did not sign in as an IAM user.
* The MFA device is attached to the root identity.
* The previous session is not still active.
* The console shows the MFA device as assigned.

Use a fresh browser session to test carefully.

---

## Problem 8: The MFA Section Shows No Device

Possible explanations include:

* Assignment was not completed.
* The page did not refresh.
* You configured MFA for another identity.
* You signed in as an IAM user.
* The operation failed.

Refresh the Security credentials page and verify the current identity.

---

## Problem 9: The Device Time Is Incorrect

Authenticator codes depend on accurate time.

Enable:

```text
Automatic date and time
Automatic time zone
```

Then reopen the authenticator application and try a new code.

---

## Problem 10: You Cannot Access the Root Email

The root email is essential for sign-in and recovery.

Try:

* Email-provider account recovery.
* Recovery telephone verification.
* Recovery email verification.
* Checking saved password-manager credentials.
* Contacting the email provider through official support.

After regaining access:

* Change the email password.
* Enable email MFA.
* Review active email sessions.
* Review forwarding rules.
* Review AWS security notifications.

---

## Problem 11: You Forgot the Root Password

Use the root-user password-recovery process from the official AWS sign-in page.

You will need access to the account email and may need to complete additional verification.

Do not create a second account to replace an account containing existing resources or charges.

---

## Problem 12: You Receive an Unexpected MFA Request

Do not approve it.

An unexpected authentication request may indicate that someone else knows the password or is attempting to sign in.

Immediately:

1. Open AWS through a trusted route.
2. Change the root password.
3. Review account activity.
4. Review IAM identities.
5. Check billing.
6. Check for unfamiliar resources.
7. Secure the root email account.
8. Contact AWS Support when compromise is suspected.

---

## Problem 13: You Accidentally Shared the MFA QR Code

Treat the MFA configuration as compromised.

Do not simply delete the screenshot and assume the problem is solved.

Take steps to:

1. Remove or replace the exposed MFA device.
2. Register a new MFA configuration.
3. Remove the shared image from all locations.
4. Review account activity.
5. Change the root password if other information was exposed.
6. Check for unfamiliar sign-ins or resources.

---

## Problem 14: Root Access Keys Appear Unexpectedly

Do not ignore them.

Investigate:

* Creation date
* Status
* Last-used information
* Region or service of last use
* Who may have created them

For a new training account, the expected number of root access keys is zero.

---

# 19. Common Root-Security Mistakes

## Mistake 1: Enabling MFA but Never Testing It

Always complete one controlled sign-out and sign-in test.

---

## Mistake 2: Saving the QR Code as a Backup

The QR code is not an ordinary screenshot.

It can contain the secret required to generate authentication codes.

Do not save or share it casually.

---

## Mistake 3: Using the Same Password for Root and IAM

Each identity must have separate credentials.

---

## Mistake 4: Using Root Because It Avoids Permission Errors

A permission error is not a reason to switch immediately to root.

Investigate:

* Which identity is active
* Which permission is required
* Whether the action is appropriate
* Whether an IAM policy should be adjusted

---

## Mistake 5: Creating Root Keys for the CLI

Do not do this.

CLI access does not require root credentials.

---

## Mistake 6: Assuming MFA Replaces Password Security

MFA and a strong password work together.

A weak or reused password is still dangerous.

---

## Mistake 7: Protecting AWS but Ignoring the Email Account

The root email is part of account security and recovery.

Secure it with the same seriousness.

---

## Mistake 8: Keeping Root Signed In

After a root task:

```text
Verify
Document
Sign out
```

Do not continue normal project work in the root session.

---

## Mistake 9: Sharing Root Credentials With a Team

Do not share one root identity among multiple people.

Professional environments use individually attributable identities and controlled access methods.

---

## Mistake 10: Ignoring Recovery Preparation

Security includes both:

* Preventing unauthorized access
* Preserving authorized recovery

Registering appropriate backup authentication methods can help reduce lockout risk.

---
