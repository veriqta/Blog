# 4A: Understanding AWS Identity and Access Management

## Goal

The goal of this lesson is to help you understand AWS Identity and Access Management before creating an administrative identity.

AWS Identity and Access Management, commonly called **IAM**, is the AWS service used to control identities and permissions within an AWS account. It helps determine who can sign in, which AWS services they can access, which actions they can perform, and which resources they can use.

In this lesson, you will learn:

* What AWS IAM is.
* Why AWS accounts need identity and access management.
* The difference between authentication and authorization.
* What IAM users are.
* What IAM user groups are.
* What IAM policies are.
* How policies grant or deny permissions.
* The difference between AWS managed policies and customer managed policies.
* What the `AdministratorAccess` policy provides.
* Why administrator access must be protected.
* What least privilege means.
* The most important IAM security practices.
* How to verify your understanding before creating an administrator.

You will **not** create the administrator in this lesson.

The administrator will be created in:

```text
Part 1C.4B – Creating the IAM Administrator
```

This chapter focuses on understanding the security model first.

# 1. What Is AWS IAM?

IAM stands for:

```text
Identity and Access Management
```

AWS IAM is a web service that helps you securely control access to AWS services and resources.

It answers two essential questions:

```text
1. Who are you?

2. What are you allowed to do?
```

The first question concerns **authentication**.

The second question concerns **authorization**.

AWS describes IAM as the infrastructure used to control both authentication and authorization within AWS accounts.

---

## 1.1 Identity Management

Identity management involves creating and managing identities that can interact with AWS.

Examples include:

* Human administrators
* Developers
* Auditors
* Finance personnel
* Applications
* EC2 instances
* Lambda functions
* Automated deployment systems
* External users
* Federated company employees

An AWS identity can represent:

```text
A person
A workload
An application
An AWS service
An external organization
```

IAM helps AWS understand which identity is making a request.

---

## 1.2 Access Management

Access management determines what an authenticated identity can do.

Examples include:

* View an S3 bucket
* Upload an S3 object
* Launch an EC2 instance
* Stop an EC2 instance
* View CloudWatch metrics
* Create an IAM role
* Delete a security group
* View billing information
* Change a password
* Create a CloudFront distribution

AWS access management is commonly called **authorization**. You manage authorization by using policies that define which actions an identity or resource is allowed or denied from performing.

---

## 1.3 IAM Is a Global Service

IAM is not managed separately in every AWS Region in the same way as EC2.

For example, an IAM user created in the account is not created only for:

```text
us-east-1
```

or:

```text
eu-west-2
```

The identity belongs to the AWS account.

However, IAM policies can restrict an identity so that it can use resources only in particular Regions.

Example:

```text
IAM identity
      ↓
Permission allows EC2 actions
      ↓
Condition restricts activity to eu-west-2
```

IAM itself is global, but the permissions it controls can affect both global and Regional resources.

---

## 1.4 IAM Does Not Create AWS Resources

IAM controls access to AWS resources.

IAM does not itself act as:

* A virtual server
* A storage bucket
* A database
* A content delivery network
* A monitoring dashboard
* A website host

Instead, IAM determines whether an identity can interact with those services.

Example:

```text
IAM user
    ↓
IAM policy
    ↓
Permission to create an S3 bucket
    ↓
Amazon S3 performs the bucket creation
```

IAM makes the access decision.

Amazon S3 performs the storage operation.

---

# 2. Why IAM Exists

Imagine an organization with 100 employees.

Without identity and access management, everyone might share one AWS account password.

That would create serious problems:

* No individual accountability
* No separation of duties
* No controlled permissions
* No easy employee removal
* No reliable activity attribution
* Greater credential exposure
* Greater risk of accidental changes
* Greater risk of malicious actions

IAM allows the organization to create controlled identities and grant different permissions according to responsibilities.

---

## 2.1 Example Organization

Consider the following teams:

```text
Company AWS Account
│
├── Cloud Administrators
├── Developers
├── Security Team
├── Finance Team
├── Auditors
└── Applications
```

These teams should not all have identical permissions.

### Cloud Administrators

May need to:

* Manage infrastructure
* Configure IAM
* Create networking resources
* Troubleshoot services

### Developers

May need to:

* Deploy applications
* View logs
* Use selected development resources
* Update code-related services

### Finance Team

May need to:

* View bills
* Review budgets
* Analyze spending

They normally do not need permission to:

* Delete EC2 instances
* Change security groups
* Create IAM administrators

### Auditors

May need to:

* View configurations
* Review logs
* Inspect policies

They may not need permission to modify resources.

### Applications

May need permission to:

* Read a specific S3 bucket
* Write logs
* Access one database
* Publish to one message queue

IAM allows each identity to receive permissions appropriate to its job.

# 3. IAM and the Root User

Every standard AWS account begins with the AWS account root user.

The root user has complete account authority and should not be used for everyday work.

IAM allows you to establish additional identities instead of continuing to use root.

A simplified structure is:

```text
AWS Account
│
├── Root user
│   └── Reserved for limited account-level tasks
│
└── IAM
    ├── Users
    ├── User groups
    ├── Roles
    └── Policies
```

The root user is the account-ownership identity.

IAM identities are created for controlled access.

---

## 3.1 Root Is Not an IAM User

This distinction is important.

The root user:

* Is created with the AWS account.
* Uses the account email address.
* Has complete access.
* Is not placed in an IAM group.
* Does not receive its authority from `AdministratorAccess`.
* Cannot be restricted like an ordinary IAM user.

An IAM user:

* Is created inside the account.
* Has an IAM username.
* Receives permissions through IAM.
* Can belong to groups.
* Can have permissions added or removed.
* Can be deleted when no longer needed.

---

## 3.2 Why IAM Comes After Root Security

The root user must be secured first because it can control IAM.

An attacker with root access may be able to:

* Create new IAM users.
* Create access keys.
* Grant administrator permissions.
* Delete legitimate administrators.
* Change account settings.
* Remove security protections.

That is why the correct setup order is:

```text
1. Create the AWS account

2. Secure the root user

3. Create a separate administrator

4. Sign out of root

5. Use controlled identities for normal work
```

---

# 4. Authentication vs Authorization

Authentication and authorization are related, but they are not the same.

Understanding the difference is essential for working with AWS security.

---

# 4.1 Authentication

Authentication answers:

```text
Who are you?
```

It is the process of proving an identity.

Examples of authentication information include:

* Email address
* Username
* Password
* MFA code
* Passkey
* Security key
* Temporary security credentials
* Access key credentials
* Federated company sign-in

When you sign in to the AWS console, AWS attempts to match the credentials to a trusted principal.

---

## 4.2 Authentication Example

Suppose an IAM user has:

```text
Account ID: 123456789012
Username: ann-admin
Password: ************
MFA code: 482901
```

The user submits the information to AWS.

AWS verifies whether:

* The account exists.
* The IAM user exists.
* The password is correct.
* The MFA code is valid.
* The identity is permitted to sign in.

When the identity is successfully verified, authentication succeeds.

---

## 4.3 Failed Authentication

Authentication may fail because:

* The username is incorrect.
* The password is incorrect.
* The wrong AWS account is being used.
* The MFA code is invalid.
* The MFA device is unavailable.
* The IAM user was deleted.
* Console access is disabled.
* The sign-in URL is incorrect.
* The credentials have expired.
* The account has a sign-in restriction.

A failed authentication attempt means AWS could not successfully verify the identity.

AWS does not proceed to normal resource authorization when the user cannot authenticate.

---

# 4.4 Authorization

Authorization answers:

```text
What are you allowed to do?
```

After authentication, AWS evaluates the applicable permissions.

Examples include:

```text
Can this identity view EC2 instances?

Can this identity create S3 buckets?

Can this identity delete IAM users?

Can this identity view the bill?

Can this identity update a CloudFront distribution?
```

Authorization is controlled primarily through policies.

---

## 4.5 Authorization Example

Suppose the user successfully signs in.

The user then attempts to create an S3 bucket.

AWS evaluates the request:

```text
Authenticated identity:
ann-developer

Requested action:
s3:CreateBucket

Requested resource:
New S3 bucket

Applicable policies:
Developer permissions
Organization restrictions
Permissions boundaries
Resource policies
Other applicable controls
```

AWS decides whether to allow or deny the request.

---

## 4.6 Authentication Success Does Not Guarantee Authorization

A user can sign in successfully but still receive:

```text
AccessDenied
```

This means:

```text
Authentication succeeded.

Authorization failed.
```

The identity is known, but it does not have permission to perform the requested action.

---

## 4.7 Real-World Analogy

Think of a secured office building.

### Authentication

You present an employee identification card.

Security confirms that you are an employee.

### Authorization

Your identification card determines which rooms you may enter.

You may be allowed into:

* The main office
* The meeting room
* Your department

You may be denied access to:

* The server room
* The finance archive
* The executive office

Being recognized as an employee does not mean you can enter every room.

AWS works in a similar way.

---

## 4.8 Authentication and Authorization Flow

```text
User attempts to sign in
          ↓
AWS authenticates the identity
          ↓
Identity makes an AWS request
          ↓
AWS collects the request context
          ↓
AWS evaluates applicable policies
          ↓
Request is allowed or denied
```

AWS evaluates applicable policies and request information before deciding whether an action is permitted.

---

# 5. IAM Identities

The main identity concepts you will encounter are:

* IAM users
* IAM user groups
* IAM roles

This chapter focuses mainly on users and groups.

Roles will be introduced briefly because they are an important part of modern AWS access management.

---

# 6. IAM Users

An IAM user is an identity created inside an AWS account.

It can represent:

* A person
* A specialized administrative identity
* A legacy application requiring long-term credentials

An IAM user can have:

* A username
* Console access
* A console password
* MFA devices
* Permissions
* Group membership
* Access keys, where genuinely required
* Tags
* A permissions boundary

---

## 6.1 IAM User Example

```text
IAM user

Username:
ann-admin

Console access:
Enabled

MFA:
Required

Group:
Administrators

Permissions:
Inherited from Administrators group
```

---

## 6.2 IAM Users Do Not Have Permissions Automatically

Creating an IAM user does not automatically grant permission to use AWS resources.

By default, an IAM user starts without useful resource permissions unless permissions are granted through policies.

This follows the principle of default deny:

```text
No permission has been granted
            ↓
Request is denied
```

IAM users and roles have no permissions by default; permissions must be deliberately granted.

---

## 6.3 Console Access

An IAM user can be configured for AWS Management Console access.

Console access generally involves:

```text
AWS account or account alias
+
IAM username
+
Password
+
MFA
```

Console access is appropriate when a person needs to work through the browser-based AWS Management Console.

---

## 6.4 Programmatic Access

An IAM user can also have long-term access keys for programmatic access.

These may be used by:

* AWS CLI
* AWS SDKs
* API tools
* Legacy automation

However, AWS security guidance increasingly favors temporary credentials, roles, and federation for human and workload access instead of long-term access keys whenever possible.

Do not create access keys simply because an IAM user exists.

---

## 6.5 One Identity Per Person

Do not create one shared user such as:

```text
company-admin
```

and give the password to several people.

Instead:

```text
ann-admin
james-admin
sarah-admin
```

Individual identities improve:

* Accountability
* Password management
* MFA management
* Access removal
* Activity investigation
* Permission reviews

When one person leaves, their access can be removed without changing everyone else’s credentials.

---

## 6.6 IAM User Lifecycle

An IAM user should have a managed lifecycle.

```text
Business need identified
        ↓
User created
        ↓
Required permissions assigned
        ↓
MFA enabled
        ↓
Usage monitored
        ↓
Permissions reviewed
        ↓
Unused access removed
        ↓
User deleted when no longer required
```

An IAM user should not remain active forever without review.

---

## 6.7 When IAM Users May Be Used

IAM users may be appropriate in limited situations, including:

* A simple standalone training account
* A specific workload that cannot use roles
* A compatibility requirement
* A documented emergency-access design
* A legacy system that requires long-term credentials

For larger organizations and multi-account environments, AWS commonly recommends centralized workforce access through IAM Identity Center and temporary credentials rather than creating many individual IAM users.

---

## 6.8 IAM Users in This Handbook

This beginner handbook creates one IAM administrative user to establish a clear separation between:

```text
Root ownership identity
```

and:

```text
Daily administrative learning identity
```

This teaches core IAM concepts in a standalone training account.

In a professional organization, the identity architecture may instead use:

* IAM Identity Center
* Company federation
* External identity providers
* Temporary role sessions
* Multi-account permission sets

---

# 7. IAM User Groups

An IAM user group is a collection of IAM users.

Groups make it easier to assign the same permissions to multiple users. AWS explains that a user placed in a group automatically receives the permissions attached to that group.

---

## 7.1 Group Example

Suppose a company has three administrators:

```text
ann-admin
james-admin
sarah-admin
```

Instead of attaching the same administrator policy separately to every user, the company can create:

```text
Administrators
```

Then add all three users to the group:

```text
Administrators group
│
├── ann-admin
├── james-admin
└── sarah-admin
```

The policy is attached to the group:

```text
AdministratorAccess
        ↓
Administrators group
        ↓
All group members receive the permissions
```

---

## 7.2 Why Groups Are Useful

Groups provide:

* Centralized permission assignment
* Easier user onboarding
* Easier user removal
* Consistent permissions
* Reduced policy duplication
* Clear role-based organization

If a new administrator joins:

```text
Create user
    ↓
Add user to Administrators group
    ↓
User receives group permissions
```

If an administrator changes departments:

```text
Remove user from Administrators group
    ↓
Administrator group permissions are removed
```

---

## 7.3 Group Examples

An organization might create:

```text
Administrators
Developers
SecurityAuditors
BillingViewers
DatabaseOperators
NetworkEngineers
ReadOnlyUsers
```

Each group receives policies appropriate to its responsibilities.

---

## 7.4 Groups Contain Users, Not Groups

IAM user groups can contain IAM users.

They cannot be nested inside other IAM user groups.

You cannot create:

```text
Engineering group
└── Developers group
```

Instead, individual users can belong to multiple groups.

Example:

```text
ann
├── Developers
└── BillingViewers
```

The user receives the combined allowed permissions from both groups, subject to any explicit denies or other applicable restrictions.

---

## 7.5 Groups Are Not Identities That Sign In

A group does not have:

* A password
* An MFA device
* An access key
* A console session

A group is a permission-management container.

Users sign in.

Groups do not.

---

## 7.6 Why This Handbook Uses an Administrators Group

In Part 1C.4B, you will create:

```text
Administrators
```

The AWS managed policy named:

```text
AdministratorAccess
```

will be attached to that group.

The administrative IAM user will then be added to the group.

This design is better than attaching the administrator policy directly to the user because it teaches reusable group-based permission management.

---

# 8. IAM Policies

An IAM policy is a document that defines permissions.

A policy can state:

* Which actions are allowed
* Which actions are denied
* Which resources are affected
* Which conditions must be met

AWS policies are commonly written as JSON documents.

Policies can be attached to IAM identities or, for supported services, directly to resources.

---

## 8.1 Policy Example

The following simplified policy allows someone to list all S3 buckets:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:ListAllMyBuckets",
      "Resource": "*"
    }
  ]
}
```

This example contains several important elements:

```text
Version
Statement
Effect
Action
Resource
```

---

## 8.2 Version

The `Version` element specifies the version of the IAM policy language.

A commonly used value is:

```json
"Version": "2012-10-17"
```

This is the policy-language version.

It is not the date the policy was created.

---

## 8.3 Statement

The `Statement` element contains one or more permission statements.

Example:

```json
"Statement": [
  {
    "Effect": "Allow",
    "Action": "s3:ListAllMyBuckets",
    "Resource": "*"
  }
]
```

A policy can contain several statements.

---

## 8.4 Effect

The `Effect` element can commonly be:

```text
Allow
```

or:

```text
Deny
```

Example:

```json
"Effect": "Allow"
```

means the statement attempts to grant the listed permission.

Example:

```json
"Effect": "Deny"
```

means the statement explicitly denies the listed action.

---

## 8.5 Action

The `Action` element identifies the AWS operation.

Examples include:

```text
s3:ListAllMyBuckets
s3:GetObject
ec2:DescribeInstances
ec2:StartInstances
ec2:StopInstances
iam:CreateUser
cloudfront:CreateDistribution
```

An action normally uses this format:

```text
service-prefix:Operation
```

---

## 8.6 Resource

The `Resource` element identifies which AWS resource the statement applies to.

A specific S3 bucket object path may look like:

```text
arn:aws:s3:::example-bucket/*
```

The asterisk:

```text
*
```

is a wildcard.

Depending on its location, it can represent all matching actions or resources.

Wildcards must be used carefully because they can make permissions much broader than intended.

---

## 8.7 Condition

The optional `Condition` element adds requirements that must be satisfied.

Conditions can evaluate information such as:

* Source IP address
* Requested Region
* Resource tags
* Principal tags
* Whether MFA is present
* Time
* Encryption settings
* Transport security

AWS evaluates policy conditions against information in the request context.

---

## 8.8 Example With a Condition

A simplified policy might allow an action only when MFA is present:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "ec2:StopInstances",
      "Resource": "*",
      "Condition": {
        "Bool": {
          "aws:MultiFactorAuthPresent": "true"
        }
      }
    }
  ]
}
```

This demonstrates that permission decisions can consider more than the username.

---

# 9. How AWS Evaluates Permissions

AWS does not simply look for one policy and stop.

It evaluates the policies and controls applicable to the request.

A simplified model is:

```text
Request begins
      ↓
Default result is deny
      ↓
Check for explicit deny
      ↓
If explicit deny exists:
DENY
      ↓
If no explicit deny:
Check for applicable allow
      ↓
If applicable allow exists:
ALLOW
      ↓
If no applicable allow:
DENY
```

AWS policy evaluation follows the principle that an explicit deny overrides an allow. If no applicable allow exists, the request remains denied by default.

---

## 9.1 Default Deny

An identity does not receive access merely because no policy says “deny.”

There must generally be an applicable allow.

Example:

```text
No S3 permissions assigned
          ↓
User tries to list buckets
          ↓
No applicable Allow
          ↓
Request denied
```

---

## 9.2 Explicit Allow

A policy may grant an action:

```json
{
  "Effect": "Allow",
  "Action": "ec2:DescribeInstances",
  "Resource": "*"
}
```

This can allow the identity to view EC2 instance information, provided no other applicable control explicitly denies it.

---

## 9.3 Explicit Deny

An explicit deny takes precedence.

Example:

```text
Policy A:
Allow EC2 actions

Policy B:
Deny ec2:TerminateInstances
```

Result:

```text
View instances: Allowed

Start instances: Possibly allowed

Terminate instances: Denied
```

The deny overrides the broader allow.

---

# 10. Identity-Based Policies

Identity-based policies are attached to:

* IAM users
* IAM user groups
* IAM roles

They define what the identity is allowed or denied from doing.

Example:

```text
ReadOnlyAccess policy
        ↓
Auditors group
        ↓
Auditor users receive read permissions
```

Identity-based policies can be:

* AWS managed policies
* Customer managed policies
* Inline policies

---

# 11. Resource-Based Policies

Some AWS services support policies attached directly to resources.

Examples include:

* S3 bucket policies
* KMS key policies
* SQS queue policies
* SNS topic policies
* IAM role trust policies

Example:

```text
S3 bucket
└── Bucket policy
    └── Controls who can access the bucket
```

Resource-based and identity-based policies can both affect an authorization decision.

The detailed use of resource policies will be covered during the relevant projects.

---

# 12. Managed Policies and Inline Policies

IAM identity-based policies can be organized into three main types:

```text
1. AWS managed policies

2. Customer managed policies

3. Inline policies
```

AWS distinguishes them according to who creates and manages the policy and whether the policy can be reused across identities.

---

# 13. AWS Managed Policies

An AWS managed policy is created and maintained by AWS.

Examples include:

```text
AdministratorAccess
ReadOnlyAccess
AmazonS3ReadOnlyAccess
AmazonEC2ReadOnlyAccess
IAMReadOnlyAccess
```

AWS managed policies are standalone policies that can be attached to multiple:

* IAM users
* IAM user groups
* IAM roles

AWS may update these policies as services and API operations change. When AWS updates a managed policy, the update affects the identities to which that policy is attached.

---

## 13.1 Advantages of AWS Managed Policies

They are:

* Quick to use
* Maintained by AWS
* Available across AWS accounts
* Helpful for common use cases
* Useful for initial setup
* Easier for beginners than writing complex policies

---

## 13.2 Limitations of AWS Managed Policies

They may be:

* Broader than your exact requirements
* Updated by AWS
* Unsuitable for precise least-privilege designs
* Unable to enforce your organization’s unique resource naming
* Unable to reflect every business-specific condition

You cannot directly edit an AWS managed policy.

---

## 13.3 AWS Managed Policy Example

```text
AmazonEC2ReadOnlyAccess
```

This type of policy may allow a user to view EC2-related information without granting normal permission to launch or terminate instances.

Always inspect the current policy document rather than relying only on the policy name.

---

# 14. Customer Managed Policies

A customer managed policy is a standalone policy created and managed within your AWS account.

You control:

* The policy name
* The policy description
* The allowed actions
* The denied actions
* The resources
* The conditions
* Policy updates
* Policy versions

Customer managed policies can be attached to multiple users, groups, and roles.

---

## 14.1 Customer Managed Policy Example

Suppose a developer should upload files only to one bucket:

```text
veriqta-static-website
```

A customer managed policy could allow:

```text
s3:ListBucket
s3:GetObject
s3:PutObject
```

only for:

```text
arn:aws:s3:::veriqta-static-website
arn:aws:s3:::veriqta-static-website/*
```

The developer would not automatically receive access to every S3 bucket.

---

## 14.2 Advantages of Customer Managed Policies

They provide:

* Greater control
* Reusability
* Policy versioning
* Organization-specific permissions
* Better support for least privilege
* Easier centralized updates
* Custom resource restrictions
* Custom conditions

---

## 14.3 Limitations of Customer Managed Policies

They require you to:

* Understand IAM JSON
* Maintain the policy
* Review new service actions
* Test updates
* Avoid syntax errors
* Prevent accidental over-permission
* Document the business purpose

AWS can validate policy syntax and IAM Access Analyzer can provide additional policy checks, but the customer remains responsible for the intended permission design.

---

# 15. Inline Policies

An inline policy is embedded directly in one IAM user, group, or role.

It has a one-to-one relationship with that identity.

If the identity is deleted, the inline policy is also deleted.

AWS generally recommends managed policies when permissions may need to be reused across identities.

---

## 15.1 Inline Policy Example

```text
IAM user: temporary-project-user
└── Inline policy: unique-project-permission
```

The policy exists only inside that user.

It cannot be attached independently to another user.

---

## 15.2 When Inline Policies May Be Used

An inline policy may be considered when:

* A strict one-to-one relationship is intentionally required.
* The permission must remain inseparable from one identity.
* A specialized design has been reviewed carefully.

For most reusable permission designs, managed policies are easier to govern.

---

# 16. AWS Managed vs Customer Managed Policies

| Feature                            | AWS managed policy | Customer managed policy |
| ---------------------------------- | ------------------ | ----------------------- |
| Created by                         | AWS                | Your AWS account        |
| Updated by                         | AWS                | You                     |
| Editable by you                    | No                 | Yes                     |
| Reusable                           | Yes                | Yes                     |
| Has its own ARN                    | Yes                | Yes                     |
| Designed for common use cases      | Yes                | Not necessarily         |
| Can be customized precisely        | No                 | Yes                     |
| Supports least-privilege tailoring | Limited            | Stronger control        |
| Maintenance responsibility         | AWS                | Customer                |

---

## 16.1 When to Use AWS Managed Policies

They may be useful when:

* Learning AWS
* Establishing initial access
* Testing a common job function
* Starting before usage requirements are fully understood
* Using a documented AWS service role
* Performing short-term setup that will later be refined

AWS notes that managed policies can provide a starting point, after which Access Analyzer and observed activity can help teams move toward more precise permissions.

---

## 16.2 When to Use Customer Managed Policies

They are usually better when:

* Permissions must be restricted to named resources.
* Only selected actions should be allowed.
* Company-specific conditions are required.
* Permission changes must be controlled by your team.
* A reusable least-privilege policy is needed.
* Auditors require a clear business-specific policy.
* AWS managed policies are too broad.

---

# 17. Understanding AdministratorAccess

`AdministratorAccess` is an AWS managed policy.

Its AWS managed policy ARN is:

```text
arn:aws:iam::aws:policy/AdministratorAccess
```

AWS describes it as providing full access to AWS services and resources. It can be attached to IAM users, groups, and roles.

---

## 17.1 Simplified Policy Document

The core policy is extremely broad:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "*",
      "Resource": "*"
    }
  ]
}
```

This means:

```text
Effect:
Allow

Action:
All AWS actions

Resource:
All AWS resources
```

The policy provides enormous authority.

---

## 17.2 What AdministratorAccess Can Allow

An identity with this policy can generally perform actions such as:

* Create and delete EC2 instances.
* Create and delete S3 buckets.
* Change security groups.
* Create IAM users.
* Create IAM roles.
* Attach policies.
* Create access keys for IAM users.
* Create CloudFront distributions.
* Manage VPC resources.
* Create databases.
* Modify logging.
* Create expensive resources.
* Delegate permissions to other identities.

AWS describes the administrator job function as having full access and the ability to delegate permissions across services and resources.

---

## 17.3 AdministratorAccess Does Not Turn an IAM User Into Root

An IAM administrator with `AdministratorAccess` is still not the root user.

Certain sensitive account-level tasks may still require root credentials.

Examples may include:

* Changing certain root credentials
* Some account recovery operations
* Changing the root email
* Closing a standalone AWS account
* Activating certain account-level billing-access settings

The administrator is extremely powerful but remains an IAM identity.

---

## 17.4 Billing Access May Require Additional Account Configuration

Even when an identity has `AdministratorAccess`, access to the Billing and Cost Management console may require the account-level setting that permits IAM users and roles to access billing information.

AWS notes that IAM access to billing must first be activated before policies can provide access to the billing console.

This setting will be handled deliberately in the account setup workflow.

---

## 17.5 Why This Handbook Uses AdministratorAccess

This handbook uses `AdministratorAccess` for the dedicated training administrator because the learner needs to complete several different AWS projects involving:

* IAM
* S3
* CloudFront
* EC2
* VPC
* Security groups
* CloudWatch
* Other supporting services

Creating a perfect least-privilege policy before the learner understands the required services would make the initial setup unnecessarily complex.

However:

```text
AdministratorAccess is a controlled training convenience.

It is not the correct permission for every person.
```

---

## 17.6 AdministratorAccess Risks

A compromised administrator may be able to:

* Create additional administrators.
* Create access keys.
* Delete resources.
* Weaken security controls.
* Expose private data.
* Launch expensive infrastructure.
* Modify logging.
* Change permissions.
* Cause operational outages.

The administrator must therefore use:

* A unique password
* MFA
* A private sign-in URL
* A trusted computer
* Careful daily practices
* Regular permission review

---

## 17.7 Who Should Not Receive AdministratorAccess

Do not assign it automatically to:

* Developers
* Contractors
* Students
* Billing personnel
* Auditors
* Applications
* EC2 instances
* CI/CD systems
* Every employee
* Temporary support staff

Each identity should receive only the permissions required for its responsibilities.

---

# 18. The Principle of Least Privilege

Least privilege means granting only the minimum permissions required to complete a specific task.

AWS defines least privilege as allowing the smallest set of actions necessary for an identity to fulfill its task.

---

## 18.1 Over-Permissioned Example

A person needs only to upload website files to one S3 bucket.

They receive:

```text
AdministratorAccess
```

This is excessive because it may also allow them to:

* Delete EC2 instances.
* Change IAM users.
* Create databases.
* Modify VPCs.
* Disable logging.
* Access unrelated buckets.

---

## 18.2 Least-Privilege Example

The person receives permission only to:

```text
List one bucket
Upload objects to one bucket
Replace objects in one bucket
Read objects from one bucket
```

The permission does not allow:

```text
Deleting IAM users
Launching EC2 instances
Viewing unrelated buckets
Changing account security
Creating databases
```

---

## 18.3 Least Privilege Dimensions

Least privilege considers:

```text
Which actions?

On which resources?

Under which conditions?

For how long?

From which identity?

From which environment?
```

A well-designed policy can restrict:

* Service
* Action
* Resource
* Region
* Time
* Network
* Tags
* MFA presence

---

## 18.4 Start Broad, Then Reduce Carefully

During early development, teams sometimes begin with broader permissions to discover operational requirements.

This should not become permanent.

A controlled process is:

```text
1. Begin with temporary, reviewed access.

2. Observe which permissions are genuinely used.

3. Generate or write a narrower policy.

4. Test the narrower policy.

5. Remove the broad policy.

6. Continue reviewing access.
```

IAM Access Analyzer and activity data can assist with refining permissions.

---

## 18.5 Least Privilege Is Continuous

Least privilege is not a one-time task.

Permissions must be reviewed when:

* A person changes roles.
* A project ends.
* New services are introduced.
* An application changes.
* An employee leaves.
* A security incident occurs.
* A policy has not been used.
* Temporary access expires.

---

# 19. IAM Best Practices

## Best Practice 1: Do Not Use Root for Daily Work

Use root only for tasks that specifically require root credentials.

Normal AWS project activity should use an appropriate identity.

---

## Best Practice 2: Use Individual Identities

Each person should use an individually attributable identity.

Do not share:

* Usernames
* Passwords
* MFA devices
* Access keys
* Sessions

---

## Best Practice 3: Require MFA

Enable MFA for highly privileged identities.

This includes:

* Root
* Administrators
* Emergency-access identities
* Identities with access to sensitive resources

---

## Best Practice 4: Prefer Temporary Credentials

Where practical, use:

* IAM roles
* IAM Identity Center
* Federation
* Temporary sessions

instead of long-term access keys.

An IAM role does not have the same standard long-term password or access keys associated with it and is designed to be assumed when needed.

---

## Best Practice 5: Avoid Unnecessary Access Keys

Do not create access keys unless a genuine programmatic requirement exists.

Never place access keys in:

* Source code
* Git repositories
* Screenshots
* Chat messages
* Public documentation
* Plain-text project files

---

## Best Practice 6: Assign Permissions Through Groups

For IAM users with similar responsibilities:

```text
Create group
    ↓
Attach policies to group
    ↓
Add users to group
```

This is easier to manage than attaching several policies separately to every user.

---

## Best Practice 7: Apply Least Privilege

Avoid assigning:

```text
Action: "*"
Resource: "*"
```

unless the broad access is genuinely required and deliberately controlled.

---

## Best Practice 8: Review Permissions Regularly

Check:

* Which users exist
* Which groups exist
* Which policies are attached
* Which access keys exist
* When credentials were last used
* Whether permissions remain necessary

---

## Best Practice 9: Remove Unused Identities

When an identity is no longer needed:

1. Disable or remove active credentials.
2. Investigate dependencies.
3. Remove group membership.
4. Remove policies.
5. Delete the identity when safe.

Do not leave abandoned administrators active.

---

## Best Practice 10: Use Clear Names

Examples:

```text
Administrators
Developers
BillingViewers
SecurityAuditors
```

Avoid unclear names:

```text
Group1
Test
NewUsers
Misc
Permissions
```

Clear names make reviews and incident response easier.

---

## Best Practice 11: Protect Administrative Access

Administrative identities should have:

* Unique passwords
* MFA
* Minimal daily exposure
* No credential sharing
* Carefully reviewed access keys
* Secure recovery methods
* Activity monitoring

---

## Best Practice 12: Test Policies Safely

IAM provides policy validation and a Policy Simulator for testing and troubleshooting policies.

A simulator can help evaluate whether a policy may allow or deny selected actions.

Simulation does not replace testing in a controlled environment.

---

## Best Practice 13: Document Permission Decisions

Record:

* Identity name
* Business purpose
* Policy name
* Permission owner
* Date granted
* Review date
* Removal conditions

This prevents permanent access from being granted without accountability.

---

# 20. Common Beginner IAM Mistakes

## Mistake 1: Continuing to Use Root

After creating an administrator, beginners sometimes continue signing in with the root email.

The correct daily workflow is:

```text
Root:
Reserved

IAM administrator:
Used for controlled administration
```

---

## Mistake 2: Giving Everyone AdministratorAccess

Administrator access should not be the default permission.

Most users require a much smaller permission set.

---

## Mistake 3: Creating a Shared Administrator

Do not allow several people to use one administrator password and MFA device.

---

## Mistake 4: Attaching Policies Without Reading Them

Policy names are helpful but not enough.

Review:

* Policy description
* JSON document
* Allowed services
* Allowed actions
* Resource scope
* Policy updates

---

## Mistake 5: Confusing Users and Groups

Remember:

```text
User:
An identity

Group:
A collection of IAM users

Policy:
A permission document
```

---

## Mistake 6: Expecting a Group to Sign In

Groups do not sign in.

Users sign in and inherit group permissions.

---

## Mistake 7: Creating Access Keys Automatically

Console access and programmatic access are separate.

A console administrator does not automatically need access keys.

---

## Mistake 8: Ignoring Explicit Deny

Adding another allow policy does not override an applicable explicit deny.

---

## Mistake 9: Assuming AdministratorAccess Is Root

An IAM administrator is powerful, but it is not the root user.

---

## Mistake 10: Never Reviewing Permissions

A permission that was appropriate six months ago may be unnecessary today.

---

# 26. Knowledge Review

## Question 1

What does IAM stand for?

A. Internet Account Monitoring
B. Identity and Access Management
C. Internal Application Mapping
D. Infrastructure Availability Management

**Answer:** B

---

## Question 2

What are the two primary questions IAM helps answer?

**Answer:**

```text
Who are you?

What are you allowed to do?
```

---

## Question 3

What is authentication?

**Answer:** Authentication is the process of verifying the identity of a person, application, or workload.

---

## Question 4

What is authorization?

**Answer:** Authorization determines which actions an authenticated identity is permitted to perform.

---

## Question 5

Can a user authenticate successfully and still receive AccessDenied?

**Answer:** Yes.

Authentication can succeed while authorization fails because the user lacks the required permission.

---

## Question 6

Is the AWS root user an IAM user?

**Answer:** No.

The root user is the original account-ownership identity.

---

## Question 7

What is an IAM user?

**Answer:** An IAM user is an identity created inside an AWS account that can receive credentials and permissions.

---

## Question 8

Does a newly created IAM user automatically have administrator access?

**Answer:** No.

IAM users begin without permissions unless access is granted.

---

## Question 9

What is an IAM user group?

**Answer:** It is a collection of IAM users used to manage shared permissions.

---

## Question 10

Can an IAM group sign in to AWS?

**Answer:** No.

Users sign in. Groups are permission-management containers.

---

## Question 11

Can a user belong to more than one IAM group?

**Answer:** Yes.

The user can receive permissions from multiple groups, subject to applicable denies and restrictions.

---

## Question 12

Can IAM groups contain other IAM groups?

**Answer:** No.

IAM user groups contain users, not nested groups.

---

## Question 13

What is an IAM policy?

**Answer:** It is a document that defines allowed or denied AWS actions, resources, and optional conditions.

---

## Question 14

Which policy element commonly specifies whether access is allowed or denied?

A. Version
B. Effect
C. ResourceName
D. Username

**Answer:** B

---

## Question 15

What does the `Action` element identify?

**Answer:** It identifies the AWS service operation or operations affected by the policy statement.

---

## Question 16

What does the `Resource` element identify?

**Answer:** It identifies the AWS resource or resources to which the statement applies.

---

## Question 17

What happens when no applicable policy allows a request?

**Answer:** The request is denied by default.

---

## Question 18

What happens when one policy allows an action but another applicable policy explicitly denies it?

**Answer:** The explicit deny overrides the allow.

---

## Question 19

Who creates and maintains AWS managed policies?

**Answer:** AWS.

---

## Question 20

Who creates and maintains customer managed policies?

**Answer:** The customer within their AWS account.

---

## Question 21

Can you directly edit an AWS managed policy?

**Answer:** No.

You can inspect and attach it, but AWS controls its contents and updates.

---

## Question 22

Can one customer managed policy be attached to several users, groups, or roles?

**Answer:** Yes.

---

## Question 23

What is an inline policy?

**Answer:** It is a policy embedded directly in one IAM user, group, or role and maintained in a one-to-one relationship with that identity.

---

## Question 24

What is `AdministratorAccess`?

**Answer:** It is an AWS managed policy that provides full access to AWS services and resources.

---

## Question 25

Does AdministratorAccess make an IAM user identical to the root user?

**Answer:** No.

The administrator is highly privileged but remains an IAM identity, and some account-level tasks may still require root.

---

## Question 26

Should every IAM user receive AdministratorAccess?

**Answer:** No.

Permissions should reflect the user’s responsibilities.

---

## Question 27

What is least privilege?

**Answer:** It is the practice of granting only the minimum permissions required to complete a specific task.

---

## Question 28

Why should access keys not be created automatically?

**Answer:** Long-term access keys introduce security risk and are unnecessary unless a genuine programmatic-access requirement exists.

---

## Question 29

Why are IAM groups useful?

**Answer:** They allow permissions to be assigned consistently to multiple users and simplify onboarding, changes, and access removal.

# Part 1C.4A Summary

AWS Identity and Access Management controls identities and permissions within an AWS account.

In this lesson, you learned:

* IAM controls authentication and authorization.
* Authentication verifies identity.
* Authorization determines permitted actions.
* The root user is not an IAM user.
* IAM users are identities created inside the AWS account.
* IAM groups organize users and shared permissions.
* IAM policies define allowed and denied actions.
* Access is denied by default unless an applicable permission allows it.
* An explicit deny overrides an allow.
* AWS managed policies are created and maintained by AWS.
* Customer managed policies are created and maintained in your account.
* Inline policies belong directly to one identity.
* `AdministratorAccess` provides extremely broad authority.
* An IAM administrator is powerful but is not the root user.
* Least privilege means granting only the permissions required for a task.
* MFA, individual identities, permission reviews, and temporary credentials are important IAM practices.

Remember:

```text
Identity tells AWS who is making the request.

Policy tells AWS what the identity may do.

Authentication verifies the identity.

Authorization evaluates the permission.
```
