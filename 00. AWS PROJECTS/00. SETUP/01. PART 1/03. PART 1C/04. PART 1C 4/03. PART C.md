# Part 1C.4C – Signing In as the IAM Administrator

## Purpose of This Chapter

In Part **1C.4B**, you created your first IAM administrator account. The account now exists, but you have not yet used it.

In this chapter, you will safely transition from the AWS account root user to your new IAM administrator account. You will learn how to locate the IAM sign-in URL, sign in successfully, verify that you are using the correct identity, and confirm that your administrator account has the expected permissions.

After completing this chapter, you should use the IAM administrator account for all remaining projects in this handbook.

# Section 2 — Locate the IAM Sign-In URL

## Option 1 – Use the Saved Sign-In URL

If you saved the IAM sign-in URL during Part 1C.4B, open it in your web browser.

The URL will look similar to:

```text
https://123456789012.signin.aws.amazon.com/console
```

or, if you later configure an account alias:

```text
https://your-company.signin.aws.amazon.com/console
```

Use the URL associated with your AWS account.

---

## Option 2 – Find the Sign-In URL Again

If you did not save the URL:

1. While still signed in as the root user, open the **IAM** console.
2. In the navigation pane, select **Dashboard**.
3. Locate the **AWS account sign-in URL** section.
4. Copy the displayed URL.
5. Store it securely for future use.

---

## Optional — Create an Account Alias

AWS allows you to replace the numeric account ID in the sign-in URL with a friendly account alias.

For example:

Instead of:

```text
123456789012.signin.aws.amazon.com
```

You might use:

```text
veriqta-lab.signin.aws.amazon.com
```

An alias makes the sign-in URL easier to remember but does not change your permissions or account.

If you create an alias, choose a name that is professional, descriptive, and unique within AWS.

# Section 3 — Sign Out of the Root User

Before testing the administrator account, completely sign out of the root account.

1. Select your account menu in the upper-right corner of the AWS Management Console.
2. Choose **Sign out**.
3. Wait until the AWS sign-in page appears.

At this point, the root session has ended.

---

# Section 4 — Sign In as the IAM Administrator

1. Open your IAM sign-in URL.
2. If prompted, enter the AWS account ID or account alias.
3. Enter your IAM username.
4. Enter your administrator password.
5. If MFA is enabled, complete the MFA challenge.
6. Select **Sign in**.

If everything has been configured correctly, the AWS Management Console will open.

---

# Section 5 — Verify Your Identity

Before making any changes in AWS, confirm that you are signed in with the correct account.

## Check the Account Menu

Select the account menu in the upper-right corner of the console.

Confirm:

* You are not signed in as the root user.
* Your IAM username is displayed.
* The correct AWS account is shown.

---

## Open the Security Credentials Page

Navigate to the IAM console and review your user information.

Confirm:

* Username matches the account you created.
* Group membership includes **Administrators**.
* Console access is enabled.

---

# Section 6 — Test Administrator Permissions

Verify that the account has administrative access.

Perform simple read-only checks such as:

* Open Amazon S3.
* Open Amazon EC2.
* Open Amazon VPC.
* Open CloudWatch.
* Open IAM.

Each service should open without an **Access Denied** error.

Do not create unnecessary resources during this verification.

---

# Section 7 — Troubleshooting Common Sign-In Problems

## Incorrect Username

Confirm that the username exactly matches the one created in Part 1C.4B.

---

## Incorrect Password

Re-enter the password carefully.

Remember that passwords are case-sensitive.

---

## Wrong Sign-In URL

Ensure that you are using the IAM sign-in URL for your AWS account, not the generic AWS home page or another AWS account.

---

## Still Signed In as Root

If you still see the root account after signing in, sign out completely and repeat the process using the IAM sign-in URL.

---

## Access Denied

If services display **Access Denied**:

* Verify that the user belongs to the **Administrators** group.
* Verify that the **AdministratorAccess** policy is attached to the group.
* Check that you are signed in with the correct IAM user.
