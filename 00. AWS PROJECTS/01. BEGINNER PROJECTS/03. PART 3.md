## **3.2 Upload the Website Files to S3**

Now you'll move these files from your computer to your S3 bucket in the cloud.

**Step 1:** Go back to the AWS Console (make sure you're signed in as your IAM user).

**Step 2:** In the search bar, type: `S3`
- Click on **"S3"**

**Step 3:** Click on your bucket name: `veriqta-static-website-YOURNAME-2026`

**Step 4:** You're now inside your bucket. It should be empty (showing "There are no objects in this folder").

**Step 5:** Click the orange button **"Upload"**

**Step 6:** On the "Upload" page, you'll see a gray box that says "Drag and drop files and folders you want to upload here"

You have two options:

**Option A — Drag and Drop (Easiest):**
- Open your file manager showing `Project-1-S3-CloudFront/`
- Select all three files: `index.html`, `error.html`, and `style.css`
- Drag them into the gray box on the AWS page
- You'll see three file names appear in a list

**Option B — Click to Select:**
- Click the **"Add files"** button
- Navigate to: `Documents/VERIQTA-AWS-Projects/Project-1-S3-CloudFront/`
- Select `index.html`, `error.html`, and `style.css` (hold Ctrl or Cmd to select multiple)
- Click **"Open"**
- The files appear in the list

**Step 7:** Scroll down to "Destination" and verify it says:
- Bucket: `veriqta-static-website-YOURNAME-2026`
- Folder: `/` (root of the bucket)

**Step 8:** Scroll down to "Permissions" section.
- Under "Predefined ACLs", select **"Grant public-read access"** from the dropdown
- A warning will appear: **"Everyone will be able to read these objects"**
- Check the box: **"I understand the risk of granting public-read access to the specified objects."**

**Why this step?** By default, files in S3 are private. Your website needs to be public so visitors can see it. This setting makes these specific files publicly readable.

**Step 9:** Leave all other settings as default. Scroll to the bottom and click **"Upload"**

**Step 10:** You'll see an "Upload: status" page with a progress bar.
- Wait for all three files to show "Succeeded"
- Click the **"Close"** button

**Step 11:** You should now see all three files listed in your bucket:
- `error.html`
- `index.html`
- `style.css`

**Step 12:** Click on `index.html` to view its details.
- Look at the "Object overview" section.
- You should see "Object URL" which looks like: `https://veriqta-static-website-YOURNAME-2026.s3.amazonaws.com/index.html`
- You should see "ACL" which should say: `Public`

If the ACL says "Private", you missed the permissions step. Go back and re-upload with public-read access.

## **3.3 Configure the Bucket Policy**

The file-level permissions you just set work, but the professional way to manage public access is through a bucket policy. This is a JSON document that defines rules for the entire bucket. We'll add a policy that allows anyone on the internet to read (but not modify) your files.

**Step 1:** While viewing your bucket contents, click the **"Permissions"** tab.

**Step 2:** Scroll down to the section called **"Bucket Policy"**

**Step 3:** Click **"Edit"**

**Step 4:** You'll see a text box labeled "Policy". This is where you paste JSON code.

**Step 5:** Copy and paste the following policy EXACTLY. **Important:** Replace `YOUR-BUCKET-NAME` with your actual bucket name.

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "PublicReadGetObject",
            "Effect": "Allow",
            "Principal": "*",
            "Action": "s3:GetObject",
            "Resource": "arn:aws:s3:::YOUR-BUCKET-NAME/*"
        }
    ]
}
```

**For example,** if your bucket is named `veriqta-static-website-johnsmith-2026`, the `Resource` line should be:
```
"Resource": "arn:aws:s3:::veriqta-static-website-johnsmith-2026/*"
```

**What this policy means:**
- `Version`: The version of the policy language (always use 2012-10-17)
- `Statement`: A list of rules (we have one rule)
- `Sid`: A label for this rule (descriptive name)
- `Effect`: "Allow" means this rule grants permission
- `Principal`: "*" means anyone on the internet
- `Action`: "s3:GetObject" means they can read/download files
- `Resource`: The ARN (Amazon Resource Name) of your bucket and all objects in it (the `/*` means all files)

**Step 6:** After pasting and editing the bucket name, click **"Save changes"**

**Step 7:** If you see a warning about public access, that's expected. Click **"Confirm"** or **"I acknowledge..."** if prompted.

## **3.4 Test the S3 Website**

This is the moment of truth. Your website is live on S3.

**Step 1:** Go back to the **"Properties"** tab of your bucket.

**Step 2:** Scroll down to **"Static website hosting"**

**Step 3:** Click on the **"Bucket website endpoint"** link.
- It looks like: `http://veriqta-static-website-YOURNAME-2026.s3-website-us-east-1.amazonaws.com`

**Step 4:** A new browser tab opens. What you should see:
- The same beautiful website you previewed locally
- Purple gradient background
- White card with "Welcome to My AWS-Powered Website"
- Three feature cards
- The URL in the address bar is your S3 website endpoint

**If you see an error:**
- **403 Forbidden:** Your bucket policy or file permissions are wrong. Go back to Permissions and verify the bucket policy is correct and that files have public-read access.
- **404 Not Found:** Your index.html file isn't named correctly or isn't in the root of the bucket. Check the file name is exactly `index.html` (lowercase, no spaces).
- **No styles (plain HTML):** The `style.css` file wasn't uploaded or the link in HTML is broken. Check that `style.css` is in the same bucket folder as `index.html`.

**Step 5:** Test the error page. In the address bar, manually type a non-existent page:
```
http://veriqta-static-website-YOURNAME-2026.s3-website-us-east-1.amazonaws.com/this-page-does-not-exist
```
- You should see your custom error page with "Oops! Page Not Found"
- Click the "Go Back Home" button — it should take you back to the main page.

**Congratulations.** Your website is live on the internet, served directly from Amazon S3.
