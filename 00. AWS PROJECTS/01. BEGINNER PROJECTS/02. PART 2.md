## **2.1 Project Overview & Architecture**

**What you're building:** A complete static website hosted on Amazon S3 and delivered globally through Amazon CloudFront.

**Why this matters:** S3 is object storage — think of it as a hard drive in the cloud where you store files. But S3 can also act as a web server for static content (HTML, CSS, JavaScript, images). CloudFront is a Content Delivery Network (CDN) — it copies your website to edge locations around the world so visitors get fast load times regardless of where they are.

**The Architecture:**

```
[User's Browser] 
      ↓
[CloudFront Distribution]  ← Global edge locations (fast delivery)
      ↓
[Amazon S3 Bucket]         ← Your website files stored here
      ↓
[HTML/CSS/JS Files]        ← The actual website content
```

**The flow:**
1. A user types your website URL into their browser.
2. CloudFront checks if it has a cached copy of your site at the nearest edge location.
3. If yes, it serves the cached copy instantly.
4. If no, it fetches the files from S3, caches them, and serves them.
5. Future visitors get the cached version — much faster.

**What you'll do:**
- Create an S3 bucket (your storage container)
- Configure it for static website hosting
- Write HTML and CSS files
- Upload them to S3
- Set a bucket policy (security permissions)
- Test the S3 website directly
- Create a CloudFront distribution
- Configure CloudFront settings
- Test the final CDN-powered website
- Troubleshoot common issues
- Clean up resources

## **2.2 Creating the S3 Bucket**

**Step 1:** Sign in to the AWS Console using your IAM user sign-in link (the one you bookmarked in Part 1).

**Step 2:** In the top search bar, type: `S3`
- Click on **"S3"** (Simple Storage Service) when it appears under Services.

**Step 3:** You'll land on the S3 Buckets page. It might be empty or show existing buckets. Click the orange button **"Create bucket"**

**Step 4:** The "Create bucket" page opens. This has several sections. We'll go through each one.

### **Section: General configuration**

**Bucket name:** This must be globally unique across ALL of AWS. No two buckets in the world can have the same name. Use this format:
```
veriqta-static-website-YOURNAME-2026
```
Replace `YOURNAME` with your actual name or initials. For example:
```
veriqta-static-website-johnsmith-2026
```

**Why this format?** It includes your identifier, making it unique, and the year helps with organization.

**AWS Region:** Select a region close to you for lower latency. Look for the dropdown that says "AWS Region".
- If you're in the US: Select **"US East (N. Virginia) us-east-1"** (this is the default and often has the most features)
- If you're in Europe: Select **"Europe (Ireland) eu-west-1"** or **"Europe (Frankfurt) eu-central-1"**
- If you're in Asia: Select **"Asia Pacific (Singapore) ap-southeast-1"** or **"Asia Pacific (Tokyo) ap-northeast-1"**

For this guide, we'll use **US East (N. Virginia) us-east-1**. If you choose a different region, most steps are identical, but some URLs will differ slightly.

---

### **Section: Object Ownership**

This controls who owns the files you upload.

**Step 5:** Under "Object Ownership", you'll see two options:
- ACLs disabled (recommended)
- ACLs enabled

**Select: "ACLs disabled (recommended)"**

Leave the default option checked: **"Bucket owner enforced"**

**Why?** This is the modern, secure way. We'll control access through bucket policies, not individual file permissions.

---

### **Section: Block Public Access settings for this bucket**

This is CRITICAL. By default, AWS blocks ALL public access to buckets. This is a security feature. But since we're building a public website, we need to allow public access.

**Step 6:** Uncheck the box that says:
**"Block all public access"**

When you uncheck it, four sub-checkboxes will appear and become unchecked:
- Block public access to buckets and objects granted through new access control lists (ACLs)
- Block public access to buckets and objects granted through any access control lists (ACLs)
- Block public access to buckets and objects granted through new public bucket or access point policies
- Block public access to buckets and objects granted through any public bucket or access point policies

Make sure ALL FOUR are unchecked.

**Step 7:** A yellow warning box will appear saying: **"Turning off block all public access might result in this bucket and the objects within becoming public"**

This is expected. Check the box that says:
**"I acknowledge that the current settings might result in this bucket and the objects within becoming public."**

---

### **Section: Bucket Versioning**

**Step 8:** Under "Bucket Versioning", leave it as **"Disable"** for now.
- If you enable it, every time you upload a file with the same name, AWS keeps all versions. This is useful for production but adds complexity for learning.

---

### **Section: Tags (Optional)**

**Step 9:** You can skip tags for now, or add one for organization:
- Click **"Add tag"**
- Key: `Project`
- Value: `VERIQTA-Project-1`
- This helps you identify resources later, especially if you have many.

---

### **Section: Default encryption**

**Step 10:** Leave as **"Server-side encryption with Amazon S3 managed keys (SSE-S3)"**
- This is the default and encrypts your files at rest automatically.

---

### **Section: Advanced settings**

**Step 11:** You can leave these as default. Don't change anything here.

---

### **Create the Bucket**

**Step 12:** Scroll to the bottom of the page. Click the orange button **"Create bucket"**

**What you should see:** A green success banner at the top saying: **"Successfully created bucket 'veriqta-static-website-YOURNAME-2026'"**

Your bucket now appears in the S3 Buckets list.

## **2.3 Configure Bucket Settings**

Now we need to tell S3 that this bucket will serve a website. This involves two main configurations: static website hosting and the bucket policy.

### **Enable Static Website Hosting**

**Step 1:** In the S3 Buckets list, click on your bucket name (the one you just created). This opens the bucket's contents page.

**Step 2:** Click the **"Properties"** tab (it's in the row of tabs near the top: Objects, Properties, Permissions, Metrics, Management, Access Points).

**Step 3:** Scroll ALL the way down to the bottom. You'll see a section called **"Static website hosting"**

**Step 4:** Click **"Edit"**

**Step 5:** On the "Edit static website hosting" page:
- Select **"Enable"**
- For "Hosting type", select **"Host a static website"**
- For "Index document", type exactly: `index.html`
- For "Error document", type exactly: `error.html`
- Click **"Save changes"**

**What this does:** When someone visits your website, S3 will automatically look for `index.html` and serve it. If a page isn't found, it will serve `error.html`.

**Step 6:** After saving, scroll back down to "Static website hosting" in the Properties tab.
- You'll now see a **"Bucket website endpoint"** URL.
- It looks like: `http://veriqta-static-website-YOURNAME-2026.s3-website-us-east-1.amazonaws.com`
- **Copy this URL and save it in a text file.** We'll need it later for testing.

**Important note:** This URL uses HTTP, not HTTPS. That's fine for S3 direct access, but CloudFront will provide HTTPS later.
