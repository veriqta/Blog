



# Building Enterprise CI/CD Pipelines

## From Code Commit to Production at Scale

# Part 2 — Building a Production-Ready Enterprise Pipeline

In **Part 1**, we established the foundation of enterprise CI/CD.

We explored source control, build automation, testing, security validation, artifact management, Infrastructure as Code, environment promotion, and quality gates.

Now we move from principles to implementation.

This section focuses on how organizations combine these practices into a complete production-ready delivery system.

The goal is not simply to connect a few tools.

The goal is to design a pipeline that can support:

* Multiple engineering teams
* Frequent releases
* Secure software delivery
* Consistent deployments
* Complex cloud environments
* Kubernetes workloads
* Enterprise governance
* Reliable rollback
* Continuous improvement

A production-grade pipeline must be fast enough for developers, secure enough for the business, and reliable enough for customers.

---

# Designing the Pipeline Architecture

Before selecting tools, organizations must design the delivery architecture.

Many teams make the mistake of choosing Jenkins, GitHub Actions, GitLab CI, or another platform before clearly defining what the pipeline must accomplish.

The tool should support the architecture.

The architecture should not be forced to fit the tool.

A strong enterprise pipeline begins by answering several questions:

* Where is the source code stored?
* What triggers the pipeline?
* Where will builds execute?
* How will secrets be managed?
* Where will artifacts be stored?
* Which environments will exist?
* What quality gates are required?
* How will deployments be approved?
* How will rollbacks work?
* How will pipeline activity be monitored?
* How will audit records be preserved?

These decisions determine the overall pipeline structure.

# Pipeline as Code

Enterprise pipelines should be defined as code.

Pipeline definitions should live inside version control alongside the application.

Examples include:

* Jenkinsfile
* GitHub Actions workflow files
* GitLab CI configuration files
* Azure Pipelines YAML
* Tekton pipeline definitions

Pipeline as Code provides several benefits.

## Version Control

Every pipeline change is recorded.

Teams can see who changed the pipeline, what changed, and why.

## Code Review

Pipeline modifications can pass through pull requests and peer review.

## Reusability

Common pipeline templates can be shared across teams.

## Auditability

Organizations can prove how software moved through the delivery process.

## Recovery

If a pipeline change causes problems, the previous version can be restored.

A pipeline should never exist only inside a graphical configuration screen.

Critical automation must be reproducible.

# Choosing a CI/CD Platform

There is no single platform that is best for every organization.

The correct choice depends on:

* Existing infrastructure
* Team experience
* Cloud platform
* Security requirements
* Integration needs
* Operating model
* Cost
* Scale

Several platforms are commonly used in enterprise environments.

---

# Jenkins

Jenkins is one of the most widely recognized automation servers in software delivery.

It is highly flexible and supports thousands of integrations through plugins.

Jenkins is often selected by enterprises that require extensive customization or already operate complex internal infrastructure.

## Strengths

* Highly customizable
* Large plugin ecosystem
* Supports almost any technology
* Works in cloud and on-premises environments
* Mature enterprise adoption
* Strong support for Pipeline as Code

## Challenges

* Requires infrastructure management
* Plugin maintenance can become complex
* Poorly designed Jenkins environments may be difficult to scale
* Security updates require operational discipline
* Controllers and agents must be managed carefully

A well-designed Jenkins environment typically separates the control layer from build execution.

```text
Jenkins Controller

↓

Build Agents

↓

Docker or Kubernetes Workers

↓

Artifact Repository

↓

Deployment Platform
```

The Jenkins controller should coordinate jobs.

It should not perform heavy builds directly.

Builds should run on dedicated agents or temporary Kubernetes pods.

This reduces resource contention and improves scalability.

---

# GitHub Actions

GitHub Actions provides CI/CD directly inside GitHub repositories.

It is attractive for teams already using GitHub for source control.

Workflows are defined using YAML files stored in the repository.

## Strengths

* Native GitHub integration
* Easy pull request automation
* Large marketplace of reusable actions
* Supports cloud-hosted and self-hosted runners
* Simple developer experience
* Strong support for repository events

## Challenges

* Complex enterprise workflows require governance
* Reusable actions must be reviewed carefully
* Third-party actions may introduce supply chain risks
* Hosted runner costs can increase at scale
* Self-hosted runners require security and infrastructure management

GitHub Actions is especially effective when teams want development and automation to exist in one platform.

---

# GitLab CI/CD

GitLab provides source control, CI/CD, security scanning, package management, and deployment capabilities within one platform.

Pipelines are typically defined in a `.gitlab-ci.yml` file.

## Strengths

* Integrated DevOps platform
* Strong Pipeline as Code capabilities
* Built-in security features
* Container registry integration
* Environment tracking
* Supports SaaS and self-managed deployments

## Challenges

* Platform upgrades require planning in self-managed environments
* Runner architecture must be designed carefully
* Large environments may require significant operational resources
* Teams must establish governance for shared runners and templates

GitLab CI/CD is useful for organizations that prefer a consolidated software delivery platform.

---

# Azure DevOps Pipelines

Azure DevOps provides source control, pipeline automation, artifact management, test management, and work tracking.

It is commonly used by organizations heavily invested in Microsoft technologies.

## Strengths

* Strong integration with Microsoft Azure
* Supports Windows and Linux workloads
* Good enterprise governance capabilities
* YAML and graphical pipeline options
* Integration with Azure Repos and Azure Artifacts
* Suitable for hybrid environments

## Challenges

* Complex configurations can become difficult to manage
* Some organizations use too many platform-specific features
* Migration may become harder if workflows are tightly coupled to Azure

The correct platform is the one that aligns with the organization’s operating model and technical requirements.

# The Core Pipeline Stages

A production-ready enterprise pipeline typically contains several distinct stages.

Each stage should have a clear purpose.

A common structure is:

```text
Checkout

↓

Build

↓

Unit Test

↓

Code Quality Analysis

↓

Security Scan

↓

Package

↓

Publish Artifact

↓

Deploy to Development

↓

Integration Test

↓

Deploy to Staging

↓

Production Approval

↓

Deploy to Production

↓

Validate

↓

Monitor
```

The pipeline should fail early whenever possible.

There is no reason to wait until the final stage to report a problem that could have been detected during the first few minutes.

---

# Stage 1: Source Checkout

The pipeline retrieves the correct version of the source code.

This stage should record:

* Commit ID
* Branch
* Repository
* Triggering user
* Pull request number
* Build number
* Timestamp

These details create traceability between the source change and the final production release.

---

# Stage 2: Dependency Installation

Most applications depend on external packages.

Examples include:

* npm packages
* Python libraries
* Maven dependencies
* NuGet packages
* Operating system packages

Dependencies should be locked to approved versions.

Examples include:

* `package-lock.json`
* `poetry.lock`
* `requirements.txt`
* `pom.xml`
* `packages.lock.json`

Uncontrolled dependency versions can produce inconsistent builds.

The same commit might behave differently depending on when it is built.

Dependency locking improves reproducibility.

---

# Stage 3: Build

The build stage compiles or packages the application.

Examples include:

* Compiling Java code
* Bundling frontend files
* Packaging Python applications
* Building .NET binaries
* Creating Docker images

Build logs should be preserved.

A failed build should stop the pipeline immediately.

---

# Stage 4: Unit Testing

Unit tests provide fast feedback.

They should run early because they are usually cheaper and faster than integration or end-to-end tests.

A strong unit testing stage may include:

* Test execution
* Code coverage
* Test result publishing
* Failure reporting
* Trend tracking

Code coverage should be treated as a useful indicator, not a guarantee of software quality.

A system can have high coverage and still contain poor tests.

---

# Stage 5: Static Code Analysis

Static analysis evaluates the source code without executing it.

It can detect:

* Code smells
* Duplicate code
* Complexity
* Potential bugs
* Style violations
* Maintainability issues

Platforms such as SonarQube are often used to enforce quality standards.

A quality gate may require:

* No critical defects
* No newly introduced vulnerabilities
* Minimum test coverage
* Limited duplicate code
* Acceptable maintainability rating

The most effective approach focuses on preventing new problems from entering the codebase.

---

# Stage 6: Security Validation

Security should be embedded throughout the pipeline.

A complete security stage may include:

* SAST
* Dependency scanning
* Secret detection
* License compliance
* Container scanning
* Infrastructure as Code scanning

Security results should be prioritized by severity.

A pipeline may stop automatically when it detects:

* Critical vulnerabilities
* Exposed credentials
* Disallowed software licenses
* High-risk infrastructure misconfigurations
* Unsigned or untrusted artifacts

Not every low-risk finding must block delivery.

Organizations should define clear security policies based on business impact.

---

# Stage 7: Artifact Creation

Once the application passes validation, the pipeline creates a deployable artifact.

The artifact should include metadata such as:

* Application name
* Version
* Commit ID
* Build number
* Creation date
* Source repository
* Security scan status

An artifact version might look like:

```text
payment-service:2.7.4
```

Or:

```text
payment-service:2.7.4-commit-a31f92
```

The version should make the artifact traceable.

---

# Docker Integration

Containers have become a standard packaging method for enterprise applications.

A Docker image contains:

* Application code
* Runtime
* Dependencies
* Operating system libraries
* Startup instructions

This creates consistency between environments.

The same image can move from development to staging and production.

---

# A Basic Container Build Flow

```text
Source Code

↓

Docker Build

↓

Container Test

↓

Security Scan

↓

Image Tagging

↓

Push to Registry

↓

Deploy to Kubernetes
```

The image should be created once and promoted across environments.

---

# Writing Secure Dockerfiles

A production Dockerfile should follow several important practices.

## Use Small Base Images

Smaller images reduce:

* Attack surface
* Download time
* Storage consumption
* Vulnerability exposure

## Use Trusted Base Images

Images should come from approved registries or verified publishers.

## Avoid Running as Root

Containers should use non-root users whenever possible.

## Remove Unnecessary Tools

Compilers and package managers may not be required in the final runtime image.

## Use Multi-Stage Builds

Multi-stage builds separate compilation from runtime.

Example:

```dockerfile
FROM node:20 AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine

WORKDIR /app
COPY --from=builder /app/dist ./dist
USER node
CMD ["node", "dist/server.js"]
```

The final image contains only what is required to run the application.

---

# Container Image Tagging

Avoid deploying images with only the `latest` tag.

The `latest` tag does not identify a specific build.

Use immutable tags such as:

```text
payment-service:2.7.4
payment-service:build-1842
payment-service:git-a31f92
```

Immutable tags improve:

* Rollback
* Traceability
* Auditability
* Incident investigation

A production deployment should always point to a specific artifact version.

---

# Artifact and Container Registries

Artifacts must be stored in a secure central repository.

Container registries may include:

* Amazon ECR
* Azure Container Registry
* Google Artifact Registry
* JFrog Artifactory
* Sonatype Nexus
* Harbor
* GitHub Container Registry
* GitLab Container Registry

The registry should enforce:

* Authentication
* Role-based access
* Image scanning
* Encryption
* Retention policies
* Immutable tags
* Audit logging
* Replication where required

Production clusters should pull images only from approved registries.

# Deploying to Kubernetes

Kubernetes has become a common target platform for enterprise CI/CD pipelines.

The pipeline does not simply upload an application to a server.

Instead, it submits the desired application state to the Kubernetes API.

This may include:

* Deployments
* Services
* ConfigMaps
* Secrets
* Ingress resources
* Horizontal Pod Autoscalers
* Network policies
* Service accounts

A simplified deployment flow looks like this:

```text
Pipeline

↓

Container Registry

↓

Kubernetes Manifest or Helm Chart

↓

Kubernetes API

↓

Deployment Controller

↓

Pods

↓

Service

↓

Users
```

Kubernetes continuously compares the desired state with the actual state.

If a container fails, Kubernetes may restart it automatically.

If a node fails, workloads may be rescheduled.

---

# Kubernetes Deployment Manifests

A basic deployment manifest may define:

* Application name
* Replica count
* Container image
* Resource requests
* Resource limits
* Health checks
* Environment variables
* Security context

Example:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-service
spec:
  replicas: 3
  selector:
    matchLabels:
      app: payment-service
  template:
    metadata:
      labels:
        app: payment-service
    spec:
      containers:
        - name: payment-service
          image: registry.example.com/payment-service:2.7.4
          ports:
            - containerPort: 8080
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
          livenessProbe:
            httpGet:
              path: /health
              port: 8080
```

The pipeline updates the image version and applies the manifest.

---

# Readiness and Liveness Probes

Health checks are essential during Kubernetes deployment.

## Readiness Probe

Determines whether the application is ready to receive traffic.

If readiness fails, Kubernetes removes the pod from the service endpoints.

## Liveness Probe

Determines whether the application is still functioning.

If liveness fails repeatedly, Kubernetes may restart the container.

## Startup Probe

Helps applications that require additional startup time.

Without proper health checks, Kubernetes may send traffic to an application before it is ready.

---

# Resource Requests and Limits

Production workloads should define CPU and memory requirements.

## Requests

Requests tell Kubernetes how many resources a workload expects.

They influence scheduling.

## Limits

Limits define the maximum resources the container may consume.

Incorrect resource settings can create problems.

Limits that are too low may cause:

* CPU throttling
* Out-of-memory termination
* Poor application performance

Limits that are too high may waste cluster capacity.

Resource settings should be based on performance testing and production monitoring.

---

# Helm Integration

Managing raw Kubernetes YAML becomes difficult as applications and environments grow.

Helm provides packaging and templating for Kubernetes resources.

A Helm chart may contain:

```text
payment-service/
├── Chart.yaml
├── values.yaml
├── values-dev.yaml
├── values-staging.yaml
├── values-prod.yaml
└── templates/
    ├── deployment.yaml
    ├── service.yaml
    ├── ingress.yaml
    └── configmap.yaml
```

The chart defines the application structure.

Values files define environment-specific configuration.

---

# Why Enterprises Use Helm

Helm helps organizations:

* Package Kubernetes applications
* Reuse deployment templates
* Manage environment differences
* Version deployment configurations
* Perform upgrades
* Support rollback
* Standardize application deployments

A pipeline command might look like:

```bash
helm upgrade --install payment-service ./chart \
  --namespace production \
  --values values-prod.yaml \
  --set image.tag=2.7.4
```

The image remains unchanged.

Only the deployment configuration changes.

# Secrets Management

Secrets are among the most sensitive components of a delivery pipeline.

Examples include:

* Database passwords
* API tokens
* Cloud credentials
* Encryption keys
* TLS certificates
* Signing keys

Secrets should never be:

* Hardcoded in source code
* Stored in plain text
* Added directly to pipeline files
* Printed in logs
* Shared through chat messages
* Embedded inside container images

---

# Enterprise Secrets Management Tools

Organizations may use:

* HashiCorp Vault
* AWS Secrets Manager
* Azure Key Vault
* Google Secret Manager
* Kubernetes External Secrets
* CyberArk
* GitHub encrypted secrets
* GitLab CI variables

The pipeline should retrieve secrets only when required.

Access should follow the principle of least privilege.

---

# Short-Lived Credentials

Long-lived credentials create significant risk.

Modern pipelines should use short-lived identities whenever possible.

Examples include:

* Cloud workload identity
* OpenID Connect
* Temporary access tokens
* Kubernetes service accounts
* IAM roles
* Managed identities

Instead of storing a permanent cloud key, the pipeline requests temporary credentials.

These credentials expire automatically.

This reduces the impact of accidental exposure.

# Environment Promotion

After the artifact is created, it moves through environments.

A common flow is:

```text
Development

↓

Integration

↓

Quality Assurance

↓

Staging

↓

Production
```

The artifact should not be rebuilt at each stage.

The same version should be promoted.

Example:

```text
Development:
payment-service:2.7.4

Staging:
payment-service:2.7.4

Production:
payment-service:2.7.4
```

This confirms that production receives the exact artifact already tested.

---

# Configuration Across Environments

While the artifact remains unchanged, configuration may vary.

Examples include:

* Database endpoints
* Domain names
* Replica counts
* Logging levels
* Feature flags
* Resource allocations
* External service URLs

Configuration should be managed separately from the application.

This separation allows the same artifact to run safely in multiple environments.

---

# Manual and Automated Promotion

Promotion can happen in different ways.

## Manual Promotion

An authorized person approves the next deployment stage.

This is common when:

* Regulations require approval
* Production releases follow fixed windows
* Business stakeholders must authorize release
* High-risk changes require review

## Automated Promotion

The pipeline promotes the artifact when all required tests pass.

This is common in mature engineering organizations with strong automated controls.

## Conditional Promotion

The pipeline automates low-risk changes but requires approval for high-risk releases.

This creates a balance between speed and governance.

---

# Parallel Pipeline Execution

Enterprise pipelines should be optimized for speed.

Independent activities can often run in parallel.

Instead of this:

```text
Unit Test

↓

Security Scan

↓

Static Analysis

↓

Dependency Scan
```

The pipeline can execute them simultaneously:

```text
             Unit Test
            ↗
Build ──────→ Security Scan
            ↘
             Static Analysis
            ↘
             Dependency Scan
```

The pipeline continues only after all required checks finish successfully.

Parallelism reduces feedback time.

---

# Pipeline Caching

Build systems often download the same dependencies repeatedly.

Caching can reduce pipeline duration.

Common cache targets include:

* Maven dependencies
* npm packages
* Python packages
* Gradle files
* Docker layers
* Go modules

Caching must be designed carefully.

An outdated or corrupted cache can create inconsistent behavior.

Cache keys should include relevant dependency files.

For example:

```text
cache-key:
operating-system + dependency-lock-file-hash
```

When dependencies change, the cache is refreshed.

---

# Ephemeral Build Environments

Build agents should be temporary whenever possible.

An ephemeral agent is created for one build and destroyed afterward.

Benefits include:

* Clean build environments
* Reduced configuration drift
* Better isolation
* Improved security
* Easier scaling

Kubernetes is often used to create temporary CI workers.

```text
Pipeline Request

↓

Temporary Kubernetes Pod

↓

Build and Test

↓

Publish Results

↓

Delete Pod
```

Each pipeline receives a clean environment.

---

# Reusable Pipeline Templates

Large organizations may operate hundreds of repositories.

Allowing every team to design a completely different pipeline creates inconsistency.

Enterprise platforms often provide reusable templates.

A standard template may include:

* Build stage
* Unit tests
* Security scans
* Artifact publishing
* Deployment logic
* Logging
* Governance controls

Teams supply only application-specific values.

Example:

```yaml

language: java
container_port: 8080
deployment_target: kubernetes

```

Standardization improves:

* Security
* Supportability
* Developer onboarding
* Compliance
* Maintenance
* Reliability

# Enterprise Pipeline Design Patterns

Several design patterns appear frequently in mature CI/CD systems.

---

## Pattern 1: Centralized Pipeline Platform

A central platform team manages the CI/CD infrastructure.

Application teams use shared templates and approved tools.

Benefits:

* Strong governance
* Consistent security
* Easier support
* Standardized practices

Risk:

* The platform team may become a bottleneck if teams cannot self-service.

---

## Pattern 2: Federated Pipeline Ownership

A central team provides standards and core services.

Application teams own their individual pipelines.

Benefits:

* Team autonomy
* Faster innovation
* Flexible implementation

Risk:

* Pipelines may become inconsistent without strong standards.

---

## Pattern 3: Platform Engineering Model

A platform team provides reusable delivery capabilities as an internal product.

Developers interact with self-service workflows.

Examples include:

* Standard pipeline templates
* Approved deployment environments
* Secrets integration
* Artifact repositories
* Observability dashboards
* Developer portals

This model combines governance with developer autonomy.

# Complete Enterprise Pipeline Example

Consider an enterprise payment service deployed to Kubernetes.

The organization uses:

* GitHub for source control
* GitHub Actions for pipeline automation
* SonarQube for code quality
* A dependency scanner
* Amazon ECR for container images
* Helm for Kubernetes deployment
* AWS Secrets Manager for secrets
* Amazon EKS for Kubernetes
* Prometheus and Grafana for monitoring

The workflow begins when a developer creates a pull request.

---

# Step 1: Pull Request Validation

The pipeline automatically performs:

* Code checkout
* Dependency installation
* Unit testing
* Static analysis
* Secret detection
* Dependency scanning
* Dockerfile validation

If a critical check fails, the pull request cannot be merged.

---

# Step 2: Code Review

At least one approved engineer reviews the change.

For critical services, two reviewers may be required.

The pull request can be merged only when:

* Required reviews are complete
* Tests pass
* Security checks pass
* Branch protection rules are satisfied

---

# Step 3: Main Branch Build

After merge, the main pipeline begins.

It performs:

* Clean source checkout
* Application build
* Unit tests
* Integration tests
* Docker image creation
* Container security scan
* Software bill of materials generation
* Image signing
* Image tagging

The resulting image is tagged:

```text
payment-service:2.7.4
```

It is pushed to the approved registry.

---

# Step 4: Development Deployment

The pipeline deploys the image to the development environment using Helm.

Automated smoke tests verify:

* Application startup
* API availability
* Database connectivity
* Authentication
* Health endpoints

If validation fails, the deployment is marked unsuccessful.

---

# Step 5: Integration Environment

The same image moves into the integration environment.

The pipeline runs:

* API tests
* Database tests
* Message queue tests
* Contract tests
* External service simulations

This confirms that the application communicates correctly with other systems.

---

# Step 6: Staging Deployment

The artifact is promoted to staging.

Staging closely resembles production.

The pipeline performs:

* End-to-end testing
* Performance testing
* Security testing
* Deployment validation
* Operational readiness checks

The team reviews the results.

---

# Step 7: Production Approval

Because the payment service is business-critical, production deployment requires approval.

The approval record contains:

* Release version
* Change request
* Test results
* Security results
* Risk level
* Rollback plan
* Approver identity

Once approved, the pipeline continues.

---

# Step 8: Production Deployment

The pipeline deploys the Helm release into production.

Kubernetes gradually replaces old pods with the new version.

Readiness probes ensure new pods receive traffic only when ready.

The deployment pauses if pods fail to become healthy.

---

# Step 9: Production Validation

Immediately after deployment, the pipeline performs:

* Health checks
* Smoke tests
* Synthetic transactions
* Error-rate validation
* Latency checks
* Metrics comparison

The pipeline confirms that the new release is operating correctly.

---

# Step 10: Rollback if Required

If production validation fails, the system initiates rollback.

The previous version remains available:

```text
payment-service:2.7.3
```

The pipeline redeploys the known-good version.

This reduces recovery time.

---

# Sample Pipeline Logic

A simplified workflow may look like:

```yaml
stages:
  - build
  - test
  - security
  - package
  - publish
  - deploy-development
  - integration-test
  - deploy-staging
  - approval
  - deploy-production
  - validate
```

Each stage should publish clear results.

The pipeline must stop whenever a mandatory gate fails.

# Common Pipeline Mistakes

Even well-funded organizations make mistakes when designing CI/CD systems.

---

## Too Many Manual Steps

Manual steps increase:

* Delivery time
* Inconsistency
* Human error
* Operational stress

Manual approval may be necessary.

Manual execution of repeatable technical tasks usually is not.

---

## Using Production as the Final Test Environment

Production should not be the first place where integration or configuration issues are discovered.

Pre-production environments must provide meaningful validation.

---

## Rebuilding Before Production

Rebuilding creates a new artifact.

The production artifact may differ from the one tested in staging.

Promote the same immutable artifact.

---

## Storing Secrets in Pipeline Files

A YAML file is not a secret manager.

Credentials must come from secure external systems.

---

## Using Shared Permanent Build Servers

Long-lived build servers accumulate:

* Old dependencies
* Temporary files
* Configuration drift
* Security risks

Ephemeral agents provide better isolation.

---

## Allowing Untrusted Pipeline Components

Third-party actions, plugins, and scripts can introduce supply chain risk.

All pipeline dependencies should be reviewed and version-pinned.

---

## Ignoring Pipeline Performance

A pipeline that takes hours encourages developers to avoid frequent integration.

Fast feedback is essential.

Optimize slow tests, caching, parallelism, and build infrastructure.

---

# Production-Ready Pipeline Checklist

Before calling a CI/CD pipeline production-ready, verify the following.

## Source Control

* Code is stored in version control.
* Branch protection is enabled.
* Pull requests require review.
* Pipeline definitions are version-controlled.
* Commits are traceable to deployments.

## Build

* Builds are automated.
* Dependencies are locked.
* Builds run in clean environments.
* Artifacts are reproducible.
* Build logs are retained.

## Testing

* Unit tests run automatically.
* Integration tests validate service communication.
* End-to-end tests cover critical workflows.
* Smoke tests validate deployments.
* Test failures block promotion.

## Security

* SAST is enabled.
* Dependencies are scanned.
* Secrets are detected.
* Container images are scanned.
* Infrastructure definitions are scanned.
* Critical findings block delivery.

## Artifact Management

* Artifacts are centrally stored.
* Versions are immutable.
* Artifacts are signed where required.
* The same artifact moves between environments.
* Retention policies are defined.

## Deployment

* Kubernetes manifests or Helm charts are version-controlled.
* Health checks are configured.
* Resource limits are defined.
* Rollback is tested.
* Environment promotion is controlled.

## Secrets

* Secrets are not stored in source control.
* Short-lived credentials are used where possible.
* Access follows least privilege.
* Secret access is audited.
* Logs do not expose sensitive values.

## Observability

* Pipeline metrics are collected.
* Deployment results are monitored.
* Logs are centralized.
* Alerts are actionable.
* Production validation runs automatically.

# Key Takeaways

* A production pipeline begins with architecture, not tools.
* Pipeline as Code makes automation version-controlled and auditable.
* Jenkins, GitHub Actions, GitLab CI, and Azure DevOps can all support enterprise delivery when designed properly.
* Each pipeline stage should answer a clear validation question.
* Containers create consistency across environments.
* Docker images should be secure, immutable, and traceable.
* Kubernetes provides declarative application deployment and recovery.
* Helm helps standardize Kubernetes releases.
* Secrets must come from secure external systems.
* The same artifact should move from development to production.
* Parallel execution and caching improve pipeline speed.
* Ephemeral build agents reduce drift and security risk.
* Reusable templates improve consistency across large organizations.
* Pipelines require logs, metrics, alerts, and operational ownership.
* Deployment is incomplete until production validation succeeds.
