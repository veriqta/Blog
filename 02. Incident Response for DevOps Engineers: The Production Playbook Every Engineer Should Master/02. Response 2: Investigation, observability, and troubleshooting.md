# Incident Response for DevOps Engineers: The Production Playbook Every Engineer Should Master (Continued)

In the previous section, we established the fundamentals of incident response.

We explored severity levels, the incident lifecycle, communication, and how experienced engineers approach the first critical minutes of an outage.

Now comes the most important question.

**How do you actually find the problem?**

This is where incident response transforms from process into engineering.

Senior engineers rarely fix production incidents because they know more commands.

They fix them because they know **where to look first**.

---

# Investigation Begins with Observability

Imagine walking into a hospital.

A patient is unconscious.

The doctor doesn't immediately begin surgery.

They first collect information.

* Blood pressure
* Heart rate
* Oxygen level
* Medical history

Production systems work the same way.

Before changing infrastructure, you need visibility.

This is why observability is one of the most valuable skills a DevOps engineer can develop.

Observability answers three questions:

* **What is happening?**
* **Why is it happening?**
* **Where is it happening?**

Without observability, incident response becomes guesswork.

---

# The Three Pillars of Observability

Modern production systems rely on three primary sources of evidence.

## 1. Metrics

Metrics answer:

**How healthy is the system?**

Examples include:

* CPU utilization
* Memory usage
* Disk I/O
* Network throughput
* Request rate
* Error rate
* Response latency

Metrics help identify *when* something changed.

---

## 2. Logs

Logs answer:

**What actually happened?**

Examples:

```text
Database connection timeout

Authentication failed

Certificate expired

Pod restarted

Disk full

Payment request rejected
```

Logs provide context that metrics cannot.

---

## 3. Traces

Distributed tracing answers:

**Where did the request fail?**

Imagine a customer request passes through:

```text
Load Balancer

↓

API Gateway

↓

Authentication Service

↓

Payment Service

↓

Database
```

A trace reveals exactly where latency or failure occurred.

Without tracing, engineers often spend hours searching across multiple services.

---

# The Golden Signals

Google's Site Reliability Engineering practices introduced four metrics that every production engineer should understand.

These are often called **The Golden Signals**.

## Latency

How long does it take to serve requests?

Increasing latency often appears before complete outages.

---

## Traffic

How much work is the system performing?

Examples include:

* Requests per second
* Active users
* Queue length

Unexpected increases or decreases often indicate problems.

---

## Errors

What percentage of requests are failing?

Examples:

* HTTP 500 errors
* Database failures
* Authentication failures

Error rate is one of the fastest indicators of customer impact.

---

## Saturation

How close are resources to their limits?

Examples:

* CPU at 95%
* Memory exhausted
* Storage nearly full
* Database connections exhausted

High saturation means failure is approaching.

---

# Your First Dashboard

When responding to an incident, avoid opening twenty different dashboards.

Start with one dashboard that answers:

* Are customers affected?
* What changed?
* Which service is unhealthy?
* Which metrics changed first?

Good dashboards reduce investigation time.

Poor dashboards create confusion.

---

# Following the Evidence

One mistake inexperienced engineers make is chasing symptoms instead of causes.

Imagine your dashboard shows:

```text
500 Errors Increasing
```

Restarting application servers may temporarily remove the errors.

But the real problem might be:

* Database unavailable
* Expired TLS certificate
* DNS failure
* Storage exhaustion
* Third-party API outage

Symptoms tell you **where to investigate**.

Evidence tells you **what actually failed**.

---

# A Practical Investigation Workflow

Whenever you receive a production alert, investigate in this order.

## Step 1

Did anything change?

Review:

* Recent deployments
* Infrastructure changes
* Configuration updates
* DNS changes
* Certificate renewals
* Database migrations

Production incidents often begin immediately after change.

---

## Step 2

Check customer impact.

Questions include:

* Can users log in?
* Can payments be processed?
* Are APIs responding?

Business impact determines incident priority.

---

## Step 3

Review application health.

Check:

* Error rates
* Response times
* Active requests
* Failed requests

If applications appear healthy, continue deeper.

---

## Step 4

Review infrastructure.

Look at:

* EC2 instances
* Kubernetes Nodes
* Docker containers
* Auto Scaling Groups
* Load Balancer health

Healthy infrastructure does not necessarily mean healthy applications.

---

## Step 5

Investigate dependencies.

Many production outages originate from dependencies.

Examples include:

* Databases
* Message queues
* Redis
* External APIs
* DNS
* Object storage

Applications frequently fail because something they depend on has failed.

# Kubernetes Incident Investigation

Suppose your monitoring platform reports:

> "Application unavailable."

Where do you begin?

Experienced Kubernetes engineers follow a structured approach.

First, check whether Pods are running.

```bash
kubectl get pods -A
```

If Pods are restarting repeatedly:

```bash
kubectl describe pod <pod-name>
```

Next, inspect application logs.

```bash
kubectl logs <pod-name>
```

If Pods appear healthy, investigate Services.

```bash
kubectl get svc
```

Then Ingress.

```bash
kubectl get ingress
```

Finally, examine cluster events.

```bash
kubectl get events --sort-by=.metadata.creationTimestamp
```

Every command should answer a question.

Never execute commands without understanding why.

---

# Docker Incident Investigation

Containers introduce another layer of troubleshooting.

Start by confirming container status.

```bash
docker ps
```

If the container exited unexpectedly:

```bash
docker logs container_name
```

Then inspect:

* Exit codes
* Restart policies
* Resource limits
* Mounted volumes
* Environment variables

Many "Docker problems" are actually application problems running inside containers.

---

# Linux Still Matters

Cloud platforms and Kubernetes are powerful.

But underneath them is Linux.

Many production incidents eventually require operating system investigation.

Examples include:

Check disk usage.

```bash
df -h
```

Memory usage.

```bash
free -h
```

Running processes.

```bash
top
```

or

```bash
htop
```

Network connections.

```bash
ss -tulpn
```

System logs.

```bash
journalctl -xe
```

Sometimes the entire production incident comes down to one server whose disk quietly reached 100% utilization overnight.

# Investigating AWS Services

Suppose your application runs on AWS.

A production incident occurs.

Where should you look?

CloudWatch

Check:

* CPU
* Memory
* Network
* Alarms
* Logs

Application Load Balancer

Review:

* Healthy targets
* HTTP status codes
* Request count
* Latency

Auto Scaling

Verify:

* Desired capacity
* Instance launches
* Failed scaling events

Amazon RDS

Inspect:

* CPU
* Connections
* Storage
* Replication
* Failover events

AWS Health Dashboard

Never forget to confirm whether AWS itself is experiencing an issue affecting your Region.

Sometimes the problem isn't your architecture.

# Production Incident Example

A financial application suddenly begins returning HTTP 500 errors.

Initial assumption:

"The application deployment failed."

Investigation begins.

CloudWatch shows:

Application servers healthy.

CPU normal.

Memory normal.

Load Balancer healthy.

Pods healthy.

Interesting.

Next, engineers inspect Amazon RDS.

Database connections have reached the configured maximum.

No new connections can be created.

The application isn't failing because of bad code.

It is waiting indefinitely for database connections.

The team identifies a recently released feature that failed to close idle database connections.

The deployment is rolled back.

Connection count returns to normal.

Errors disappear.

Notice what happened.

The first hypothesis was incorrect.

Evidence led the team to the real cause.

---

# Avoid Confirmation Bias

One of the greatest dangers during incident response is confirmation bias.

This happens when engineers become emotionally attached to one explanation.

Example:

"I know it's the deployment."

Hours later, they discover:

The deployment was perfectly healthy.

The actual cause?

A storage volume filled overnight.

Experienced engineers continuously ask:

"What evidence disproves my theory?"

That question often shortens incidents dramatically.

---

# Don't Guess. Verify.

Production engineering is not detective fiction.

Every conclusion should be supported by evidence.

If someone says:

"I think Kubernetes is broken."

Ask:

"What evidence supports that?"

If someone says:

"The database is overloaded."

Ask:

"What metrics confirm that?"

Evidence drives decisions.

Assumptions create outages.
