# Part 2: Designing a Highly Available AWS Network

In Part 1, we explored the mindset behind highly available systems and learned why failures are inevitable in production. We introduced AWS Regions, Availability Zones, and the importance of eliminating single points of failure.

Now it's time to build the foundation of every production AWS architecture.

Before launching EC2 instances, deploying Kubernetes clusters, or configuring databases, experienced cloud engineers design the network.

Why?

Because every application, database, storage service, and monitoring platform communicates over a network.

If your network is poorly designed, no amount of Auto Scaling or load balancing will save your application.

Think of your AWS network as the road system of a city.

You can build the tallest skyscrapers in the world, but if the roads are blocked, nobody can reach them.

The same principle applies to cloud infrastructure.

A resilient application starts with a resilient network.

---

# Why Networking Comes First

Many beginners launch an EC2 instance first and figure out networking later.

It works for small labs.

It doesn't work in production.

Large organizations begin with networking because it affects everything that follows.

Your networking design determines:

* Security
* Scalability
* Connectivity
* Availability
* Compliance
* Disaster recovery
* Operational complexity

Changing a poorly designed network after production deployment is often expensive and risky.

Good engineers build applications.

Great engineers build platforms that applications can safely run on.

---

# Introducing Amazon VPC

The first building block of almost every AWS architecture is the **Amazon Virtual Private Cloud (VPC).**

A VPC is your own logically isolated network inside AWS.

Think of it as renting land before constructing a building.

Without land, there's nowhere to build.

Without a VPC, there's nowhere to deploy your infrastructure.

Inside a VPC, you control:

* IP address ranges
* Subnets
* Routing
* Internet connectivity
* Security boundaries
* Network isolation

Unlike a traditional on-premises network, AWS manages the physical infrastructure. You focus on designing the logical network.

---

# Why Does a VPC Exist?

Imagine AWS without VPCs.

Every customer's servers would share one giant network.

Traffic would be difficult to isolate.

Security would be far more complex.

IP address conflicts would be common.

A VPC solves these problems by giving every customer their own private network.

This isolation improves both security and operational flexibility.

---

# CIDR Planning: A Decision You Can't Ignore

When creating a VPC, one of the first choices you'll make is its IP address range, expressed in CIDR notation.

For example:

```text
10.0.0.0/16
```

This defines the pool of private IP addresses available within the VPC.

A common mistake is choosing a range without considering future growth.

Imagine starting with a small VPC that supports 200 servers.

Two years later, your company acquires another business and needs thousands of additional resources.

If your address space is exhausted, expanding the network becomes difficult and disruptive.

Experienced engineers don't design for today's workload.

They design for tomorrow's business.

---

# Public and Private Subnets

After creating a VPC, the next step is dividing it into subnets.

A subnet is a smaller section of the VPC.

Most production architectures use two main types:

## Public Subnets

Public subnets contain resources that must communicate directly with the internet.

Typical examples include:

* Application Load Balancers
* Bastion Hosts
* NAT Gateways

These resources require public IP addresses or internet connectivity to perform their role.

However, "public" does not mean "unprotected."

Security Groups, AWS WAF, IAM, and other controls still protect these resources.

---

## Private Subnets

Private subnets contain resources that should never be directly exposed to the internet.

Examples include:

* EC2 application servers
* Kubernetes worker nodes
* Amazon RDS databases
* ElastiCache clusters
* Internal APIs

Customers never communicate with these resources directly.

Instead, traffic flows through controlled entry points such as a load balancer.

This design significantly reduces the attack surface.

# A Production Network Layout

A common highly available AWS network looks like this:

```text
                    Internet
                        │
                Internet Gateway
                        │
        ┌──────────────────────────┐
        │          Amazon VPC       │
        │                           │
        │  Public Subnet (AZ-A)     │
        │      Load Balancer        │
        │      NAT Gateway          │
        │                           │
        │  Private Subnet (AZ-A)    │
        │      EC2                  │
        │      RDS                  │
        │                           │
        │  Public Subnet (AZ-B)     │
        │      Load Balancer        │
        │      NAT Gateway          │
        │                           │
        │  Private Subnet (AZ-B)    │
        │      EC2                  │
        │      RDS                  │
        └──────────────────────────┘
```

Notice something important.

The application servers are not directly reachable from the internet.

Only the load balancer accepts public traffic.

This is intentional.

# Why Multiple Availability Zones Matter

Suppose every resource is deployed in a single Availability Zone.

If that zone experiences a networking failure, the application becomes unavailable.

Now consider the same architecture spread across two Availability Zones.

If AZ-A fails, the Application Load Balancer automatically directs traffic to healthy instances in AZ-B.

Customers may never notice the outage.

This is the essence of high availability.

The goal is not to prevent failures.

The goal is to prevent failures from affecting customers.

---

# Internet Gateway

Resources inside a VPC are private by default.

They cannot communicate with the internet unless an Internet Gateway is attached.

An Internet Gateway acts as the bridge between your VPC and the public internet.

Without it:

* Websites cannot receive customer traffic.
* Public APIs become inaccessible.
* Load balancers cannot serve external users.

However, attaching an Internet Gateway alone does not make every resource public.

Routing rules and subnet configuration determine which resources can use it.

---

# NAT Gateway

Application servers often need outbound internet access.

For example, they may need to:

* Download software updates
* Retrieve operating system packages
* Access external APIs
* Pull Docker images
* Download security patches

However, exposing these servers directly to the internet increases risk.

A NAT Gateway solves this problem.

It allows resources in private subnets to initiate outbound internet connections while preventing unsolicited inbound traffic.

This approach improves security without sacrificing functionality.

---

# Route Tables

A Route Table tells AWS where network traffic should go.

Think of it as the GPS for your network.

When a packet arrives, AWS checks the Route Table to determine its destination.

For example:

* Traffic destined for the internet may be sent to an Internet Gateway.
* Traffic from private subnets may be routed through a NAT Gateway.
* Traffic within the VPC remains internal.

Incorrect routing is one of the most common causes of connectivity problems.

---

# Security Groups vs Network ACLs

These two services often confuse new AWS engineers.

Both filter network traffic, but they operate differently.

## Security Groups

Security Groups act as virtual firewalls attached directly to resources such as EC2 instances.

Characteristics:

* Stateful
* Instance-level protection
* Allow rules only
* Automatically permit return traffic

For most workloads, Security Groups provide the primary layer of network security.

---

## Network ACLs

Network ACLs protect entire subnets.

Characteristics:

* Stateless
* Subnet-level protection
* Allow and deny rules
* Return traffic must be explicitly permitted

Many organizations use Network ACLs as an additional defense layer, particularly in highly regulated environments.

# Production Incident

Imagine a deployment completes successfully.

The EC2 instances are healthy.

The Application Load Balancer reports healthy targets.

Yet customers receive timeout errors.

Where do experienced engineers begin?

Not by restarting servers.

Instead, they ask structured questions:

* Is DNS resolving correctly?
* Is the Load Balancer reachable?
* Are Security Groups allowing traffic?
* Are Route Tables configured correctly?
* Is the Internet Gateway attached?
* Has a Network ACL blocked inbound traffic?
* Is the problem isolated to one Availability Zone?

By narrowing the problem systematically, they identify the root cause instead of guessing.

---

# Common Networking Mistakes

Even experienced teams occasionally make networking mistakes.

Some of the most common include:

* Placing databases in public subnets.
* Deploying all workloads in a single Availability Zone.
* Using overly permissive Security Groups.
* Forgetting outbound rules for private workloads.
* Poor CIDR planning that limits future growth.
* Treating networking as an afterthought.
* Exposing SSH or RDP directly to the internet.
* Ignoring network monitoring and flow logs.

Most production outages aren't caused by AWS.

They're caused by architecture decisions.

---

# Production Best Practices

When designing a highly available AWS network:

* Use at least two Availability Zones for production workloads.
* Keep databases and application servers in private subnets.
* Expose only the Application Load Balancer to the internet.
* Plan CIDR ranges for future growth.
* Apply the principle of least privilege to Security Groups.
* Enable VPC Flow Logs for troubleshooting.
* Document your network architecture.
* Test failover scenarios regularly.
* Avoid hardcoding IP addresses where possible.
* Review routing changes through change management processes.

---

# Production Readiness Checklist

Before deploying production workloads, verify that:

* VPC CIDR ranges are documented.
* Public and private subnets exist in multiple Availability Zones.
* Internet Gateway is configured correctly.
* NAT Gateways are deployed for private workloads.
* Route Tables have been validated.
* Security Groups follow least privilege.
* Network ACLs are reviewed and tested.
* VPC Flow Logs are enabled.
* Network diagrams are current.
* Failover between Availability Zones has been tested.
