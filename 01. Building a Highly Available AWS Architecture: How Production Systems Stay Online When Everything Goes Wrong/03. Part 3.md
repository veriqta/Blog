# Part 3: Load Balancers, Auto Scaling, and Compute Architecture

In Part 2, we built the network that supports a highly available AWS environment.

We created a Virtual Private Cloud (VPC), divided it into public and private subnets, discussed Internet Gateways and NAT Gateways, and learned why networking decisions determine the resilience of everything built on top of them.

Now it's time to deploy the application itself.

Most applications begin with a single server.

That approach works surprisingly well in development.

It often works in staging.

It may even work in production for a while.

Until one day it doesn't.

The server becomes overloaded.

A hardware component fails.

The operating system crashes.

A deployment goes wrong.

Or someone accidentally terminates the instance.

At that moment, customers discover a painful truth.

The application depended on one machine.

Production engineering exists to eliminate that dependency.

---

# Why One Server Is Never Enough

Imagine you're running a popular online learning platform.

Traffic averages around 800 users every minute.

Everything appears healthy.

Then your CEO publishes a viral post on LinkedIn.

Traffic suddenly increases to 30,000 concurrent users.

Your single EC2 instance begins consuming:

* 100% CPU
* Nearly all available memory
* Thousands of incoming network connections

Soon, response times increase dramatically.

Customers begin refreshing pages.

Those additional requests increase the load even further.

Eventually the application becomes unresponsive.

The code isn't necessarily broken.

The infrastructure simply wasn't designed for growth.

Scalability is not something you add after success.

It is something you design before success arrives.

---

# The Role of a Load Balancer

Imagine walking into a supermarket with twenty checkout counters.

Instead of allowing every customer to queue at the first cashier, an employee directs shoppers to whichever checkout is available.

This keeps waiting times low and prevents one cashier from becoming overwhelmed.

An Application Load Balancer performs a similar role.

Instead of sending every request to one server, it distributes traffic across multiple healthy servers.

This improves:

* Availability
* Performance
* Scalability
* Fault tolerance
* User experience

Without a load balancer, one server becomes a bottleneck.

With one, traffic can be distributed intelligently across many servers.

---

# What Is an Application Load Balancer?

The **Application Load Balancer (ALB)** is a Layer 7 load balancer.

That means it understands HTTP and HTTPS traffic.

Instead of forwarding packets blindly, it can make routing decisions based on application-level information.

Examples include:

* URL paths
* Hostnames
* HTTP headers
* Query strings

This flexibility makes ALBs ideal for modern web applications and microservices.

For example:

```
api.company.com
        ↓
Application Load Balancer
        ↓
-------------------------------
/users     → User Service

/orders    → Order Service

/payments  → Payment Service

/admin     → Admin Portal
```

One load balancer can intelligently route traffic to many different applications.

---

# Why Load Balancers Improve High Availability

Consider an architecture with three EC2 instances.

```
Users

↓

Application Load Balancer

↓

EC2-1

EC2-2

EC2-3
```

If EC2-2 crashes, the load balancer automatically stops sending traffic to it.

Customers continue using EC2-1 and EC2-3.

Most users never notice the failure.

Notice what happened.

We didn't prevent the server from failing.

We prevented the server failure from becoming a customer outage.

That is the essence of resilient architecture.

---

# Health Checks: The Silent Guardian

How does the load balancer know a server has failed?

It doesn't wait for customers to complain.

Instead, it continuously performs health checks.

Every few seconds, it sends requests to a configured endpoint, such as:

```
/health

or

/ready
```

If the application responds successfully, the server remains in service.

If health checks fail repeatedly, the server is removed from rotation.

Only healthy instances receive customer traffic.

This automatic detection dramatically reduces downtime.

---

# Designing Reliable Health Checks

One common mistake is creating health checks that only verify whether the web server is running.

Imagine the application returns:

```
HTTP 200 OK
```

But the database connection has already failed.

Customers still receive errors.

The load balancer continues sending traffic because the health endpoint appears healthy.

A production-ready health check should verify dependencies that are critical to serving requests.

Examples include:

* Database connectivity
* Cache availability
* Disk space
* Queue connectivity
* Required external services

Good health checks reflect the application's actual ability to serve customers.

# Auto Scaling: Preparing for Growth

Even multiple servers eventually become insufficient.

Traffic fluctuates.

Morning traffic differs from evening traffic.

Weekdays differ from weekends.

Holiday sales differ from ordinary days.

Running fifty servers all day is expensive.

Running one server during peak demand is risky.

Auto Scaling solves this problem.

Instead of running a fixed number of instances, AWS automatically adjusts capacity based on demand.

When traffic increases, new servers launch automatically.

When demand decreases, unnecessary servers are terminated.

This keeps applications responsive while controlling costs.

---

# Auto Scaling Groups

An Auto Scaling Group (ASG) manages a collection of EC2 instances.

It ensures the desired number of healthy instances is always available.

Suppose the desired capacity is:

```
4 Instances
```

If one instance crashes unexpectedly, the Auto Scaling Group immediately launches a replacement.

Customers continue using the remaining instances while the replacement starts.

Engineers don't need to wake up at 2 AM simply because one virtual machine failed.

Automation handles the recovery.

---

# Launch Templates

Every new EC2 instance needs instructions.

Which Amazon Machine Image should it use?

Which instance type?

Which Security Groups?

Which IAM role?

Which startup script?

Rather than configuring these repeatedly, AWS stores the configuration inside a **Launch Template**.

Whenever Auto Scaling launches a new server, it uses this template.

This guarantees consistency across every instance.

Consistency is one of the most underrated aspects of reliable infrastructure.

---

# Scaling Policies

How does Auto Scaling decide when to launch more servers?

Using scaling policies.

Common examples include:

* CPU utilization above 70%
* Memory utilization above 80%
* Request count exceeding a threshold
* Queue length increasing
* Custom CloudWatch metrics

Choosing appropriate metrics is critical.

Scaling too early increases costs.

Scaling too late hurts customer experience.

Experienced engineers choose metrics that reflect business impact rather than infrastructure utilization alone.

---

# Multi-AZ Compute Architecture

Production workloads should never place all EC2 instances inside a single Availability Zone.

Instead, distribute them across multiple zones.

```
Users

↓

Application Load Balancer

↓

AZ-A
EC2-1
EC2-2

AZ-B
EC2-3
EC2-4
```

If AZ-A experiences a failure, traffic automatically shifts to AZ-B.

The application remains available while AWS recovers the affected infrastructure.

# Production Incident

It's 11:42 AM.

A software deployment completed twenty minutes ago.

Suddenly response times increase dramatically.

CloudWatch alarms begin firing.

Customers report intermittent HTTP 503 errors.

The Application Load Balancer shows several unhealthy targets.

An inexperienced engineer immediately restarts every EC2 instance.

A senior engineer investigates first.

Questions include:

* Did the issue begin immediately after deployment?
* Which Availability Zone is affected?
* Which instances are unhealthy?
* What changed?
* Are health checks failing?
* Is CPU saturated?
* Are memory limits exhausted?
* Did application logs report exceptions?
* Has database latency increased?

By following evidence instead of assumptions, the team identifies the root cause: a configuration error introduced during deployment that caused health checks to fail.

The unhealthy instances are removed automatically.

Traffic continues flowing to healthy servers.

A corrected deployment restores full capacity without taking the application offline.

---

# Common Mistakes

Organizations frequently make mistakes such as:

* Running only one EC2 instance.
* Disabling health checks.
* Using weak health endpoints.
* Scaling only on CPU metrics.
* Deploying every instance into one Availability Zone.
* Ignoring CloudWatch alarms.
* Forgetting to test Auto Scaling.
* Using manual recovery procedures instead of automation.

Most availability problems begin long before an outage.

They begin with architecture decisions.

---

# Production Best Practices

Experienced AWS engineers typically follow these practices:

* Deploy across multiple Availability Zones.
* Place all customer traffic behind an Application Load Balancer.
* Use Auto Scaling Groups for production workloads.
* Create meaningful health check endpoints.
* Store instance configuration in Launch Templates.
* Continuously monitor scaling behavior.
* Test instance failure regularly.
* Automate recovery wherever possible.
* Keep application servers stateless whenever practical.
* Review scaling policies as traffic patterns evolve.

---

# Production Readiness Checklist

Before moving to production, verify that:

* Application Load Balancer is deployed.
* Health checks validate real application functionality.
* Auto Scaling Groups span multiple Availability Zones.
* Launch Templates are version controlled.
* Scaling policies are tested under load.
* CloudWatch alarms are configured.
* EC2 instances are deployed in private subnets.
* Security Groups follow least privilege.
* Instance replacement has been tested.
* Deployment rollback procedures are documented.
