# What Does High Availability Actually Mean?

High Availability is the ability of a system to remain operational despite failures in its infrastructure or software components.

A highly available application minimizes interruptions by eliminating single points of failure.

Consider a simple application.

```
User

↓

Web Server

↓

Database
```

At first glance, this seems perfectly acceptable.

Until the server crashes.

Suddenly every customer loses access.

Nothing is wrong with the application.

Nothing is wrong with the code.

One server failed.

The business stopped.

This architecture contains a **Single Point of Failure (SPOF).**

Senior engineers spend much of their time identifying and removing these hidden weaknesses.

---

# Understanding Single Points of Failure

A Single Point of Failure is any component whose failure causes the entire system to stop functioning.

Examples include:

* One EC2 instance
* One database server
* One Internet connection
* One DNS server
* One Load Balancer
* One Availability Zone

Many organizations unknowingly build applications with multiple single points of failure.

Everything appears stable during testing.

Months later, one component fails.

Production collapses.

Customers don't care whether the issue came from AWS.

They only know the application stopped working.

---

# A Simple Exercise

Imagine you're reviewing an architecture diagram.

For every component, ask yourself:

**"If this disappears right now, what happens?"**

If the answer is,

*"Everything stops."*

You've found a single point of failure.

This mindset is one of the biggest differences between junior and senior engineers.

Junior engineers focus on making systems work.

Senior engineers focus on making systems continue working.

# High Availability vs Fault Tolerance vs Disaster Recovery

These terms are often used interchangeably, but they describe different goals.

Understanding the distinction is essential for designing production systems.

## High Availability

High Availability aims to keep applications running despite the failure of individual components.

Example:

An EC2 instance crashes.

The Application Load Balancer routes traffic to healthy instances.

Customers continue using the application.

The service remains available.

---

## Fault Tolerance

Fault Tolerance goes a step further.

The system continues operating with **no noticeable interruption** even when components fail.

Fault-tolerant systems are significantly more expensive because redundancy exists almost everywhere.

Examples include:

* Aircraft control systems
* Medical equipment
* Stock exchange platforms
* Spacecraft systems

Most business applications do not require complete fault tolerance.

They require high availability.

---

## Disaster Recovery

Disaster Recovery focuses on restoring service after a catastrophic event.

Examples include:

* Entire AWS Region failure
* Ransomware attack
* Data corruption
* Massive infrastructure outage

Disaster recovery answers questions such as:

* How quickly can we recover?
* How much data can we afford to lose?
* Where are backups stored?
* Can another Region take over?

Notice the difference.

High Availability tries to avoid downtime.

Disaster Recovery accepts downtime but minimizes its impact.

# Understanding AWS Global Infrastructure

Everything in AWS begins with its global infrastructure.

Without understanding it, designing a resilient architecture becomes guesswork.

AWS organizes infrastructure into several layers.

```
Global Infrastructure

↓

Regions

↓

Availability Zones

↓

Data Centers
```

Each layer exists for a specific purpose.

---

## AWS Regions

A Region is a separate geographic area containing multiple Availability Zones.

Examples include:

* us-east-1
* us-west-2
* eu-west-1
* eu-central-1
* ap-southeast-2

Regions are isolated from one another.

This isolation improves resilience.

If one Region experiences a significant outage, another Region can continue serving customers.

However, Regions are not intended for every application.

Running workloads across multiple Regions increases:

* Cost
* Operational complexity
* Network latency
* Data replication challenges

Senior engineers understand that more infrastructure isn't automatically better.

Architecture should match business requirements.

---

## Availability Zones

Within every Region are multiple Availability Zones, commonly called AZs.

Each Availability Zone consists of one or more physically separate data centers.

AWS designs these zones to minimize the chance that a single event affects all of them simultaneously.

Although Availability Zones are connected by high-speed, low-latency networking, they remain isolated in critical ways.

This isolation is intentional.

If power, cooling, or networking fails in one zone, the others can continue operating.

For production systems, Multi-AZ deployment is often the first major step toward high availability.
